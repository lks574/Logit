# Bare Workflow

**한 줄 정의**: `ios/`·`android/` 네이티브 프로젝트를 git에 커밋하고 **개발자가 직접 소유·수정**하는 RN 개발 방식. [[Managed Workflow]]([[CNG]])의 반대편으로, 최대 자유도와 최대 유지보수 비용을 함께 가진다.

**iOS/AOS로 치면**: 그냥 평소의 네이티브 개발이다. Xcode 프로젝트와 Gradle 스크립트가 저장소에 있고, 원하는 대로 편집한다. 다만 그 안에 RN 런타임과 JS [[Bundle]]이 들어있을 뿐이다.

## 설명

Bare workflow에서는 네이티브 프로젝트가 소스 오브 트루스다. Xcode에서 build phase를 추가하고, entitlement를 켜고, Gradle 설정을 고치고, AppDelegate에 코드를 넣는 — 네이티브 개발자에게 완전히 익숙한 작업을 그대로 한다. RN 커뮤니티 CLI로 만든 프로젝트(`ios/`·`android/` 커밋)가 전형적인 bare이며, Expo 도구를 쓰면서도 네이티브 디렉토리를 커밋하면 역시 bare로 분류된다. Expo의 대부분 라이브러리와 [[EAS]] Build는 bare 프로젝트에서도 동작한다.

**자유도** — bare를 선택하는 이유:

- 네이티브 프로젝트의 **모든 것**을 제약 없이 수정할 수 있다. 커스텀 빌드 설정, 특수한 서드파티 SDK 통합 절차, App Extension/위젯, 비표준 프로젝트 구조 등.
- [[Config Plugin]]으로 표현하기 어렵거나 플러그인이 제공되지 않는 네이티브 변경을 그냥 손으로 하면 된다.
- 기존 네이티브 앱에 RN을 넣는 [[Brownfield]]는 정의상 bare일 수밖에 없다 — 네이티브 프로젝트가 원래 주인이기 때문이다.

**업그레이드 비용** — bare의 대가:

- RN 신버전은 네이티브 템플릿 변경(빌드 설정, Podfile, Gradle, AppDelegate 시그니처 등)을 동반한다. bare에서는 이 변경분을 **내가 커스터마이즈한 네이티브 프로젝트에 수동 병합**해야 한다. 수정이 쌓일수록 병합이 어려워지고, 이것이 RN 업그레이드 악명의 주 원인이다.
- 두 플랫폼의 빌드 시스템 지식이 상시 필요하다. 팀에 네이티브 역량이 없으면 유지보수가 막힌다.
- 네이티브 상태가 선언적으로 기술되어 있지 않으므로, "이 프로젝트에 어떤 수동 수정이 왜 있는지"가 문서화·커밋 로그에 의존한다.

**현재 시점의 판단 기준** — [[Managed Workflow]] 노트에 정리했듯 managed의 제약이 크게 줄어서, Expo 공식 권장은 "가능하면 [[CNG]]로 시작하고, 정말 필요할 때만 bare로"다. `npx expo prebuild`로 생성된 네이티브 프로젝트를 커밋하는 순간부터 bare가 되는 것이므로 전환 자체는 쉽다. 반대로 bare → managed 복귀는 그동안의 수동 수정을 전부 [[Config Plugin]]으로 옮겨 적어야 해서 비싸다. 즉 **bare는 나가기 쉬운 문이 아니라 들어가기 쉬운 문**이다.

네이티브 시니어의 관점에서: bare는 편안하게 느껴질 것이다 — 익숙한 도구를 그대로 쓰니까. 하지만 그 편안함의 청구서가 업그레이드 시즌마다 날아온다는 점, 그리고 팀의 JS 개발자들은 네이티브 디렉토리를 블랙박스로 여긴다는 점을 감안해 선택해야 한다.

요약 비교: managed(CNG)는 "선언에서 생성, 업그레이드 쉬움, 임의 수정 불가", bare는 "직접 소유, 무엇이든 수정 가능, 업그레이드는 수동 병합". 판단 질문은 하나다 — "Config Plugin으로 표현할 수 없는 네이티브 수정이 정말 있는가?"
