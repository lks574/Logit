# Bridge

**한 줄 정의**: 구(舊) 아키텍처에서 JS와 네이티브가 JSON 직렬화된 메시지를 비동기 배치 큐로 주고받던 통신 계층. 현재는 [[JSI]] 기반 [[New Architecture]]로 대체되었다.

**iOS/AOS로 치면**: 두 프로세스가 XPC나 AIDL/Binder로 통신하는 그림을 떠올리면 된다. 같은 프로세스 안이지만, 서로의 객체를 직접 만질 수 없고 모든 것을 메시지로 직렬화해 던지는 구조라는 점에서 IPC와 닮았다.

## 설명

RN 초기 설계(2015~)의 핵심이 Bridge였다. JS 스레드와 네이티브(UI) 스레드는 완전히 분리된 세계였고, 서로 통신하려면 (1) 호출 정보를 JSON 문자열로 직렬화하고 (2) 비동기 메시지 큐에 넣고 (3) 배치로 모아서 상대편에 넘기고 (4) 역직렬화해서 실행하는 과정을 거쳤다.

"버튼을 탭했다" → 네이티브가 JS로 이벤트 전송 → JS가 처리 후 "이 뷰를 이렇게 바꿔라" → 다시 네이티브로 전송, 이 왕복이 전부 직렬화+비동기였다.

이 설계의 의도 자체는 합리적이었다. JS와 네이티브를 완전히 분리(decouple)하면 서로의 스레드를 블로킹하지 않고, 구현도 단순해진다. 하지만 규모가 커지면서 세 가지 병목이 드러났다.

- **직렬화 비용.** 큰 데이터(긴 리스트, 이미지 메타데이터 등)를 넘길 때마다 JSON 인코딩/디코딩이 발생한다.
- **비동기 강제.** 동기적으로 답을 받을 방법이 없어서, 스크롤 중 새 셀의 레이아웃을 JS에 물어보면 답이 오기 전에 화면이 지나가 버린다 — 빠른 스크롤에서 빈 화면이 번쩍이는 고질적 현상의 원인.
- **혼잡(congestion).** 모든 통신이 하나의 큐를 지나므로, 이벤트 폭주 시 밀린다.

네이티브 개발자가 이 시대의 흔적을 만나는 곳:

- 오래된 블로그 글의 "passes over the bridge" 같은 표현
- `RCTBridgeModule`(iOS)이나 `ReactContextBaseJavaModule`(Android)을 구현하는 구식 네이티브 모듈 코드
- "bridge를 건너는 횟수를 줄여라"라는 옛 성능 팁들

레거시 라이브러리를 읽거나 마이그레이션할 때 이 맥락을 알아야 한다.

RN 0.76부터 [[New Architecture]]가 기본이 되면서 Bridge는 [[JSI]] 직접 호출로 대체되었고, Bridge 코드가 아예 로드되지 않는 실행 모드를 [[Bridgeless]]라고 부른다. 구식 모듈들도 당분간은 interop layer를 통해 신 아키텍처 위에서 동작하지만, 방향은 명확히 퇴역이다.
