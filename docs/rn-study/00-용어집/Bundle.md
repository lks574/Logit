# Bundle

> [!abstract] 한 줄 정의
> 앱의 모든 JS/TS 소스와 npm 의존성을 [[Metro]]가 하나의 파일로 묶은 산출물. 네이티브 앱은 시작 시 이 파일을 로드해 JS 로직을 실행한다.

> [!info] iOS/AOS로 치면
> 컴파일된 바이너리에 대응하는 "JS 쪽 실행 산출물". 네이티브 앱 입장에서는 앱에 내장된 리소스 파일 하나다 — 에셋 카탈로그나 raw 리소스처럼 번들에 실려 가되, 내용물이 실행 가능한 코드라는 점이 다르다.

## 📖 설명

RN 앱은 두 종류의 코드로 구성된다: Xcode/Gradle이 컴파일한 네이티브 바이너리, 그리고 JS 번들.

네이티브 쪽은 껍데기(엔진, 렌더러, 네이티브 모듈)이고, 화면 구성과 비즈니스 로직 대부분은 번들 안의 JS에 들어 있다. 이 분리가 RN의 여러 특성 — 빠른 개발 사이클, [[OTA Update]] — 의 전제다.

**dev와 release에서 번들의 위치가 다르다**는 것이 핵심 멘탈 모델이다.

- **개발(dev)**: 번들이 앱에 없다. 앱이 시작하면 [[Metro]] dev 서버에 HTTP로 접속해 번들을 내려받아 실행한다. 그래서 JS 수정이 네이티브 재빌드 없이 즉시 반영되고([[Fast Refresh]]), Metro가 죽어 있으면 앱이 뜨지 않는다.
- **릴리즈(release)**: 빌드 과정에서 Metro가 번들을 생성해 앱 패키지에 내장한다. iOS는 `main.jsbundle`, Android는 `index.android.bundle`이라는 이름이 관례다. 사용자 기기에서는 네트워크 없이 이 내장 번들을 로드한다.

기본 엔진인 [[Hermes]]를 쓰는 경우, 릴리즈 번들은 JS 텍스트가 아니라 **미리 컴파일된 Hermes 바이트코드(.hbc)** 다. 덕분에 런타임에 파싱/컴파일 비용 없이 바로 실행된다.

파일 이름은 그대로 `main.jsbundle`이어도 내용물은 바이트코드일 수 있다는 점을 알아두면, 번들 파일을 열어봤을 때 텍스트가 아니어도 당황하지 않는다.

번들 크기는 곧 앱 시작 시간과 다운로드 크기에 영향을 준다. import한 라이브러리는 (트리 셰이킹의 한계 내에서) 대부분 번들에 포함되므로, 네이티브에서 프레임워크 크기를 관리하듯 JS 의존성 크기도 관리 대상이다.

프로덕션 번들은 minify되어 스택 트레이스가 뭉개지므로, 크래시 리포팅에는 source map(난독화 해제 매핑 — iOS의 dSYM, Android의 mapping.txt에 대응)을 함께 업로드한다.

[[OTA Update]]는 "네이티브 바이너리는 그대로 두고 이 번들 파일만 교체한다"는 메커니즘이다.

번들이 무엇인지 이해하면 OTA의 가능 범위와 한계(네이티브 코드는 못 바꾼다)가 자연스럽게 따라온다.
