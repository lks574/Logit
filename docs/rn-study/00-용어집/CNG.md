# CNG

**한 줄 정의**: Continuous Native Generation. `ios/`·`android/` 네이티브 프로젝트를 손으로 관리하는 소스가 아니라 **설정(app.json + [[Config Plugin]])에서 언제든 다시 생성하는 산출물**로 취급하는 Expo의 철학.

**iOS/AOS로 치면**: "xcodeproj를 커밋하지 말고 xcodegen/tuist로 생성하자"의 전면 확장판. 혹은 Infrastructure as Code — 서버를 손으로 만지는 대신 선언 파일에서 매번 재생성하는 Terraform식 사고를 네이티브 프로젝트에 적용한 것.

## 설명

전통적 RN 프로젝트에서 `ios/`와 `android/`는 소스 코드다. 팀원들이 직접 수정하고, git에 커밋하고, 충돌을 머지한다.

문제는 이 디렉토리들이 시간이 지나며 **"어쩌다 이렇게 됐는지 아무도 모르는 상태"** 가 된다는 것 — 3년 전 어떤 라이브러리 설치 안내가 시키는 대로 고친 Gradle 설정, 지금은 안 쓰는 라이브러리의 잔재, RN 업그레이드 때 템플릿과 어긋난 부분들. RN 버전 업그레이드가 고통스러운 주된 이유가 바로 이 네이티브 프로젝트의 드리프트다.

CNG는 관점을 뒤집는다. 네이티브 프로젝트에 대한 모든 요구사항 — 번들 ID, 권한, 엔타이틀먼트, 서드파티 SDK 설정 — 을 선언(app.json + Config Plugin)으로 표현하고, 실제 `ios/`/`android/`는 그 선언에서 [[Prebuild]]로 **매번 새로 생성**한다. 생성물은 git에 커밋하지 않는다.

"Continuous"라는 이름은 이 생성이 일회성 scaffold가 아니라 반복 가능한(idempotent) 과정임을 강조한다 — CI가 빌드마다 새로 생성해도 같은 결과가 나온다.

이 철학이 사주는 것들:

- **업그레이드 단순화** — Expo SDK를 올리면 새 템플릿으로 재생성하면 끝. 템플릿 변경분을 수동 머지할 필요가 없다.
- **드리프트 제거** — 네이티브 상태의 유일한 원천이 선언 파일이므로, "내 로컬에서만 되는" 네이티브 설정이 존재할 수 없다.
- **리뷰 가능성** — 네이티브 변경이 pbxproj diff가 아니라 app.json diff로 나타난다.

대가도 있다. 네이티브 프로젝트를 직접 수정할 수 없으므로, 선언으로 표현 안 되는 커스텀은 [[Config Plugin]]을 직접 작성해야 한다(진입 장벽). 그리고 팀이 규율을 지켜야 한다 — 급하다고 생성된 `ios/`를 손대는 순간 다음 재생성 때 소실된다.

이 트레이드오프가 맞지 않는 프로젝트(대규모 네이티브 커스텀, [[Brownfield]] 통합)는 네이티브 디렉토리를 커밋하고 직접 관리하는 [[Bare Workflow]]를 택한다.

Expo SDK 57 기준, CNG + [[Prebuild]] + [[EAS]] 빌드 조합이 Expo가 권장하는 표준 워크플로우다.
