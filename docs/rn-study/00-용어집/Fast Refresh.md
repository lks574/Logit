# Fast Refresh

**한 줄 정의**: 코드를 저장하면 앱을 재시작하지 않고 변경된 모듈만 교체해 화면에 반영하는 RN의 개발 기능. 가능한 경우 컴포넌트의 [[State]]를 유지한 채 갱신한다.

**iOS/AOS로 치면**: SwiftUI Preview나 Compose Live Edit에 대응하지만 한 단계 위다 — 프리뷰 캔버스가 아니라 **시뮬레이터/실기기에서 실제로 돌아가는 앱**이 그 상태 그대로 갱신된다. Xcode의 "빌드 → 앱 재시작 → 그 화면까지 다시 이동" 사이클이 통째로 사라진 경험.

## 설명

네이티브 개발의 고질적 마찰은 수정→확인 사이클이다. 코드 한 줄 고치고 컴파일, 앱 재시작, 로그인, 딥한 화면까지 다시 이동…

RN에서는 파일을 저장하면 1~2초 안에 지금 보고 있는 화면이 갱신된다. 딥한 네비게이션 스택, 입력하던 폼, 스크롤 위치까지 유지된 채로. 이것이 많은 개발자가 RN 개발 경험을 선호하는 가장 큰 실용적 이유다.

동작 원리 개요: [[Metro]] dev 서버가 파일 변경을 감지하면 해당 모듈만 다시 변환해 앱으로 푸시한다(HMR, Hot Module Replacement). 앱 쪽 React 런타임은 새 모듈로 교체한 뒤 영향받는 컴포넌트를 [[Re-render]]한다.

이때 React는 컴포넌트의 정체성을 추적해서, **같은 컴포넌트로 인식되면 [[State]]와 [[Hook]] 상태를 보존**한다. JS가 인터프리터 위에서 도는 동적 언어라서 가능한 일 — 컴파일된 바이너리를 교체해야 하는 네이티브와의 근본적 차이다.

상태가 보존되는 조건과 리셋되는 조건을 알아두면 좋다.

- **보존**: 함수 컴포넌트 내부만 수정한 경우.
- **리셋(전체 리로드 폴백)**: 파일이 컴포넌트 외의 것(일반 함수, 상수)을 함께 export하는 경우, 컴포넌트 이름을 바꾼 경우, Hook의 개수/순서가 달라진 경우 — 안전하게 상태를 버린다.
- **무력**: 네이티브 코드나 설정 변경. 새 네이티브 모듈 추가, [[Config Plugin]] 변경 등은 네이티브 재빌드가 필요하다.

SwiftUI Preview와의 차이를 짚으면: Preview는 캔버스 안의 격리된 렌더링이고, 실제 앱 상태·네트워크·네비게이션과는 분리되어 있다. Fast Refresh는 진짜 앱, 진짜 데이터, 진짜 기기 위에서 동작한다.

Compose Live Edit이 방향성은 비슷하지만 적용 범위 제약이 더 크다.

문법 에러를 저장하면 앱에 에러 화면(RedBox)이 뜨고, 고쳐서 다시 저장하면 회복된다. 앱을 완전히 다시 시작하고 싶으면 dev 메뉴(시뮬레이터에서 iOS는 Cmd+D, Android는 Cmd+M)에서 Reload를 누르면 된다.
