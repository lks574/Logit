# Hermes

**한 줄 정의**: React Native 앱을 위해 Meta가 만든 경량 JavaScript 엔진으로, JS를 빌드 타임에 바이트코드로 미리 컴파일(AOT)해서 앱 시작 속도와 메모리 사용량을 개선한다.

**iOS/AOS로 치면**: iOS에서 Swift 코드가 빌드 타임에 기계어로 컴파일되어 배포되는 것과 발상이 같다. Android로 치면 ART가 설치 시점에 dex를 AOT 컴파일하는 것과 유사하다. "런타임에 해석/최적화하던 일을 미리 해둔다"는 아이디어.

## 설명

RN 앱의 JS 코드는 어딘가에서 실행되어야 한다. 브라우저가 없으니 앱 안에 JS 엔진을 내장하는데, 과거에는 iOS의 JavaScriptCore(JSC, Safari 엔진)를 썼다. Hermes는 이를 대체하는 RN 전용 엔진이며, 현재 RN과 Expo의 기본 엔진이다.

핵심 차별점은 **JIT(Just-In-Time) 컴파일이 없다**는 것.

- 범용 엔진(V8, JSC): 앱 실행 중에 핫한 코드를 기계어로 컴파일(JIT)해 장기 실행 성능을 높인다. 대신 시작할 때마다 파싱·컴파일·워밍업 비용을 치른다.
- Hermes: 빌드 타임에 JS 소스를 파싱·컴파일해서 **Hermes 바이트코드(.hbc)** 로 만들어 앱에 내장하고, 런타임에는 이 바이트코드를 인터프리터로 실행만 한다(AOT).

왜 이런 선택을 했을까. 모바일 앱에서 중요한 것은 장시간 돌린 후의 피크 성능보다 **TTI(Time To Interactive), 즉 시작 속도**다. Hermes는 이미 컴파일된 바이트코드를 메모리에 매핑해서 바로 실행하므로 시작 시 파싱/컴파일 비용이 0에 가깝고, 메모리 점유도 작다.

iOS 개발자에게 특히 와닿는 배경: **iOS는 서드파티 앱의 JIT를 사실상 금지한다**(실행 가능한 메모리 페이지를 앱이 동적으로 쓸 수 없음, W^X 정책). 그래서 iOS에서는 어떤 JS 엔진을 쓰든 JIT가 불가능하고 인터프리터 모드로 돌아간다. "어차피 JIT를 못 쓴다면, 처음부터 인터프리터 실행에 최적화된 엔진을 만들자"가 Hermes의 설계 철학과 잘 맞아떨어진다.

Hermes는 C++로 작성되어 [[JSI]]와 자연스럽게 결합하며, [[New Architecture]]의 전제 조건에 가깝다. 디버깅은 Chrome DevTools 프로토콜(CDP)을 통해 지원된다. 릴리즈 빌드에서 [[Bundle]]은 JS 텍스트가 아니라 .hbc 바이트코드 형태로 들어간다는 점도 기억해 두면 좋다.

실무에서 만나는 지점:

- 크래시 리포트에 Hermes 스택 프레임이 찍힌다.
- 일부 최신 JS 문법·Intl API의 지원 여부가 엔진에 따라 다를 수 있다(구체 범위는 공식 문서 확인).
- 프로파일링/디버깅 도구 선택 시 "엔진이 Hermes인지"가 변수가 된다.

Expo SDK 57 기준 Hermes가 기본값이며, 특별한 이유가 없으면 바꾸지 않는다.
