# JSI

**한 줄 정의**: JavaScript Interface. JS 엔진과 C++ 사이를 직렬화 없이 직접 연결하는 얇은 C++ API 계층으로, JS에서 네이티브 함수를 **동기 호출**할 수 있게 한다.

**iOS/AOS로 치면**: Swift↔Objective-C 인터롭이나 Kotlin↔JNI에 가깝다. 특히 JNI 비유가 정확하다 — "다른 언어의 객체를 프록시로 잡고 직접 메서드를 호출하는 인터페이스". 다만 JNI보다 훨씬 가볍고 추상화되어 있다.

## 설명

구 아키텍처의 [[Bridge]]는 JS와 네이티브가 서로를 전혀 모른 채 JSON 메시지를 비동기 큐로 주고받는 구조였다. 모든 데이터가 직렬화/역직렬화를 거치고, 호출은 항상 비동기라 "지금 당장 값이 필요한" 상황(레이아웃 측정, 동기 저장소 읽기 등)을 처리할 수 없었다.

JSI는 이 근본 문제를 푼다.

JSI는 특정 JS 엔진에 종속되지 않는 **C++ 추상 인터페이스**다. `jsi::Runtime`, `jsi::Value`, `jsi::Object` 같은 타입으로 JS 세계를 C++에서 다룰 수 있게 하고, [[Hermes]]든 JSC든 이 인터페이스만 구현하면 RN이 엔진을 갈아끼울 수 있다. 엔진 교체 가능성 자체가 JSI 덕분이다.

핵심 개념이 **HostObject**다. C++ 객체가 HostObject 인터페이스를 구현하면 JS 쪽에서는 평범한 JS 객체처럼 보이고, 프로퍼티 접근이나 메서드 호출이 곧바로 C++ 코드 실행으로 이어진다. 반대로 C++에서 JS 함수를 붙잡아 직접 호출할 수도 있다. 즉 두 세계가 **서로의 메모리에 있는 객체 참조를 직접 주고받는다**. JSON으로 복사·직렬화하던 Bridge와의 결정적 차이다.

이로 인해 가능해진 것들:

- **동기 호출** — JS가 네이티브 함수를 부르고 리턴값을 그 자리에서 받는다.
- **[[Turbo Module]]** — 필요한 시점에 lazy하게 모듈을 JS에 노출.
- **[[Fabric]]** — 렌더러가 JS와 C++ 사이에서 동기적으로 레이아웃을 측정.
- **[[Reanimated]]의 [[Worklet]]** 처럼 JS 코드를 UI 스레드에서 실행하는 라이브러리.

주의할 점: "동기 호출이 가능하다"는 것이지 "모든 호출을 동기로 해야 한다"는 뜻이 아니다. JS 스레드를 블로킹하면 프레임 드랍으로 직결되므로, 무거운 작업은 여전히 비동기(Promise)로 설계한다.

동기 호출은 꼭 필요한 곳 — 즉시 읽어야 하는 설정값, 레이아웃 측정 등 — 에만 쓰는 도구다.

네이티브 개발자 입장에서 JSI를 직접 만질 일은 드물다. 보통은 [[Turbo Module]]·[[Expo Modules API]] 같은 상위 레이어를 쓰고, JSI는 그 밑바닥에서 돌아가는 배관이라고 이해하면 충분하다.
