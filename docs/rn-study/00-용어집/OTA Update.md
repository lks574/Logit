# OTA Update

**한 줄 정의**: Over-The-Air Update. 스토어 심사·재설치 없이 **JS [[Bundle]]과 에셋만** 서버에서 내려받아 교체하는 업데이트 방식. Expo 생태계에서는 [[EAS]] Update가 이를 담당한다.

**iOS/AOS로 치면**: 네이티브에는 존재하지 않는 능력이라 정확한 대응물이 없다. 가장 가까운 감각은 "서버 주도 UI/Remote Config를 극한까지 밀어붙인 것" — 다만 설정값이 아니라 **앱 로직 코드 전체**가 교체된다. 웹 개발자가 배포 버튼을 누르면 모든 사용자가 새 버전을 보는 그 경험을 앱에 이식한 것.

## 설명

RN 앱은 네이티브 바이너리(껍데기)와 JS 번들(로직·화면)로 나뉜다. 스토어 심사가 필요한 것은 바이너리 쪽이고, JS 번들은 앱이 시작할 때 로드하는 리소스일 뿐이다. OTA는 이 구조를 이용한다: 앱이 시작 시(또는 지정된 시점에) 업데이트 서버를 확인하고, 새 번들이 있으면 다운로드해 두었다가 다음 실행부터 그것을 로드한다.

버그 수정이 "심사 3일 대기 + 사용자 업데이트 유도" 없이 수 시간 내에 전체 사용자에게 도달할 수 있다.

**한계가 능력만큼 중요하다.** OTA로 바꿀 수 있는 것은 JS와 에셋뿐이다. 다음은 불가능하며 반드시 스토어 배포가 필요하다:

- 네이티브 코드 변경(네이티브 모듈 추가/수정, 네이티브 코드가 포함된 패키지 추가·버전 변경)
- [[Config Plugin]]·app.json의 네이티브 반영 사항(권한, 엔타이틀먼트, 아이콘 등)
- Expo SDK / RN 버전 업그레이드(네이티브 계층이 바뀌므로)

여기서 **런타임 호환성** 개념이 나온다. JS 번들은 자신이 기대하는 네이티브 API가 바이너리에 있어야 동작하므로, 업데이트는 호환되는 네이티브 빌드(runtime version이 일치하는 빌드)에만 배포되어야 한다.

[[EAS]] Update는 runtimeVersion과 채널(channel) 개념으로 이를 관리한다 — "이 업데이트는 이 런타임을 가진 빌드들에게만 간다". 이 매칭이 어긋나면 업데이트 직후 크래시가 나므로, OTA 운영의 핵심 규율이다.

**스토어 정책과의 관계**도 자주 나오는 질문이다. Apple 개발자 계약은 인터프리터로 실행되는 스크립트의 다운로드를 조건부로 허용한다 — 대략 "앱의 본래 목적을 바꾸지 않고, 스토어 버전과 심사 취지에 어긋나는 기능을 몰래 추가하지 않는 범위".

버그 수정과 점진적 개선은 업계에서 널리 통용되는 사용법이지만, 심사 후 앱 성격을 바꾸는 식의 남용은 계정 제재 사유다. Google Play도 유사한 취지의 정책을 가진다. 정확한 조항은 각 스토어 정책 원문 확인.

운영 팁 수준의 상식: 잘못된 번들이 나가면 원격으로 롤백할 수 있다는 것이 OTA의 또 다른 가치이고, 반대로 "OTA만 믿고 QA를 느슨하게" 하다가 전 사용자에게 크래시를 즉시 배포하는 사고도 가능하다. 강력한 만큼 배포 파이프라인 규율(스테이징 채널 → 프로덕션 채널)이 필요하다.
