# Re-render

**한 줄 정의**: [[State]]가 바뀌었을 때 React가 컴포넌트 **함수 전체를 다시 실행**해 새 UI 기술(엘리먼트 트리)을 얻는 과정. 함수 재실행이 곧바로 네이티브 뷰를 다시 그리는 것은 아니다.

**iOS/AOS로 치면**: Compose의 recomposition과 거의 동일한 개념. SwiftUI로 치면 상태 변경 시 `body`가 재평가되는 것. `setNeedsDisplay`/`invalidate()` 같은 네이티브 draw 무효화와는 층위가 다르다.

## 설명

React 컴포넌트는 "state → UI 기술"을 리턴하는 함수다. state가 바뀌면 React는 그 함수를 다시 호출한다. 이때 함수 본문의 지역 변수 선언, 계산, [[JSX]] 리턴까지 **전부 다시 실행**된다. 이것이 리렌더다.

네이티브 개발자가 가장 먼저 교정해야 할 오해: **리렌더 ≠ 화면 다시 그리기.** 리렌더의 산출물은 픽셀이 아니라 가벼운 JS 객체 트리(엘리먼트 트리)다. React는 이전 렌더의 트리와 새 트리를 비교([[Reconciliation]])해서 **실제로 달라진 부분만** 네이티브 측에 반영한다. RN에서는 이 최소 변경분이 [[Fabric]] 렌더러를 통해 [[Shadow Tree]] 갱신 → 네이티브 뷰 속성 변경으로 이어진다. 즉 컴포넌트 함수가 100번 재실행돼도 결과 트리가 같으면 네이티브 뷰는 아무것도 안 바뀐다. 그래서 "리렌더가 잦다"는 것 자체는 죄가 아니고, JS 스레드에서 함수 재실행 비용이 문제될 때만 최적화 대상이다.

**리렌더 전파 규칙** — 언제 리렌더되는가:

1. 자신의 state가 바뀌었을 때 (`setState` 호출).
2. **부모가 리렌더되면 자식도 리렌더된다.** [[Props]]가 실제로 바뀌었는지와 무관하다. React는 기본적으로 props 비교를 하지 않고 그냥 자식 함수를 다시 호출한다 (막으려면 `React.memo` → [[Memoization]]).
3. 구독 중인 Context 값이 바뀌었을 때.

반대로, 자식의 state 변경은 부모를 리렌더시키지 않는다. 전파는 아래 방향으로만 흐른다. 이 점은 Compose가 읽는 상태 단위로 스코프를 잘게 잘라 건너뛰는(skippable) 것과 다르다 — React는 컴파일러 마법 없이 "부모가 다시 실행되면 그 리턴값에 포함된 자식 호출도 다시 실행된다"는 단순한 함수 실행 모델을 따른다.

리렌더 모델에서 나오는 실전 함의:

- 렌더 함수 본문은 **순수**해야 한다. 부수효과(네트워크, 구독, 로그 등)는 `useEffect`로 뺀다. 언제든, 몇 번이든 재실행될 수 있기 때문이다.
- 렌더마다 객체/함수 리터럴이 **새로 생성**된다. `onPress={() => ...}`는 매 렌더 새 함수다. 대부분 문제없지만, `React.memo`된 자식에 넘길 때는 참조가 바뀌어 memo가 무력화된다 → [[Memoization]].
- 무거운 계산이 렌더 경로에 있으면 매 리렌더마다 반복된다 → `useMemo` 검토.
- 리렌더 폭이 넓다면 state를 더 낮은 컴포넌트로 내려서(colocate) 영향 범위를 줄이는 것이 memo보다 먼저다.

디버깅 시에는 React DevTools의 highlight updates나 `console.log`를 컴포넌트 함수 최상단에 찍어 리렌더 빈도를 확인할 수 있다.
