# Reconciliation

**한 줄 정의**: [[Re-render]]로 얻은 새 엘리먼트 트리를 이전 트리와 비교(diffing)해서, 실제 뷰에 가할 **최소 변경 집합**을 계산하는 React의 핵심 알고리즘.

**iOS/AOS로 치면**: SwiftUI가 `body` 재평가 후 뷰 identity를 기준으로 실제 렌더 트리에 차이만 반영하는 것, Compose의 recomposition 후 슬롯 테이블 비교, 혹은 `UICollectionViewDiffableDataSource`/`DiffUtil`이 하는 diff 계산과 같은 발상이다.

## 설명

React 컴포넌트는 리렌더 때마다 UI 전체 기술을 새로 리턴한다. 이걸 그대로 네이티브 뷰 재생성으로 옮기면 성능이 파괴되므로, React는 이전/새 트리를 비교해 "이 Text의 문자열만 바뀜", "이 View 하나 삭제됨" 같은 최소 패치를 뽑아낸다. 이 비교 과정이 reconciliation이다. RN에서는 계산된 패치가 [[Fabric]] 렌더러를 거쳐 [[Shadow Tree]]와 실제 네이티브 뷰에 반영된다.

일반적인 트리 diff는 O(n³)라서, React는 두 가지 휴리스틱으로 O(n)에 처리한다:

1. **타입이 다르면 서브트리를 통째로 버린다.** 같은 자리의 엘리먼트가 `<View>`에서 `<ScrollView>`로 바뀌면 비교하지 않고 기존 서브트리를 언마운트(내부 [[State]] 소멸)하고 새로 만든다. 타입이 같으면 인스턴스를 유지하고 바뀐 [[Props]]만 갱신한다.
2. **형제 목록은 `key`로 신원을 매칭한다.** key가 없으면 인덱스 순서로 비교한다.

**key가 중요한 이유** — 리스트 맨 앞에 항목을 하나 삽입했다고 하자. key가 없으면 React는 인덱스로 비교하므로 "0번이 바뀌고, 1번이 바뀌고, … 마지막에 하나 추가"로 오판해 전체를 갱신한다. 항목마다 안정적인 `key`(보통 데이터의 id)가 있으면 "새 key 하나 삽입, 나머지는 이동"으로 정확히 인식한다. SwiftUI `ForEach(items, id: \.id)`의 `id`와 정확히 같은 역할이다.

여기서 나오는 실전 규칙:

- **key로 배열 인덱스를 쓰지 말 것** (순서가 바뀌거나 삽입/삭제가 있는 리스트에서). 항목의 신원이 어긋나 다른 항목의 state(입력값, 스크롤 위치 등)를 물려받는 버그가 난다.
- key는 형제 사이에서만 유일하면 된다. 전역 유일할 필요 없다.
- 같은 데이터라도 key를 바꾸면 컴포넌트를 강제로 재마운트(리셋)할 수 있다 — state 초기화 트릭으로 종종 쓴다.
- 컴포넌트 함수 **안에서** 다른 컴포넌트를 정의하면 매 렌더마다 "새 타입"이 되어 휴리스틱 1에 걸려 매번 재마운트된다. 컴포넌트는 항상 최상위에 정의한다.

참고로 diff를 어떻게 스케줄링하느냐는 별도 주제다. React는 Fiber라는 내부 아키텍처로 reconciliation 작업을 잘게 쪼개 중단·재개할 수 있게 해놓았고, 이 덕에 급한 업데이트(입력 반응)가 덜 급한 업데이트(화면 전환)를 앞지를 수 있다. 용어로만 알아두면 된다 — "reconciler = diff 알고리즘, Fiber = 그 실행 엔진".

정리하면: [[Re-render]]가 "새 설계도를 다시 그리는 것"이라면, reconciliation은 "새 설계도와 기존 건물을 대조해 공사 지시서를 뽑는 것"이다. 이 구조 덕에 개발자는 상태→UI 전체를 선언하기만 하고, 증분 업데이트는 프레임워크가 책임진다.
