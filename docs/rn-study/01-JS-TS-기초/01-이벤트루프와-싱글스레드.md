# 이벤트루프와 싱글스레드

> JS는 스레드가 1개뿐인데 어떻게 비동기를 처리하며, React Native에서 JS 스레드를 블로킹하면 무슨 일이 벌어지는가?

## iOS/AOS 대응 개념

| JS 개념 | iOS (Swift) | Android (Kotlin) | 비고 |
|---|---|---|---|
| JS 스레드 (단일) | Main thread + GCD main queue | Main thread + `Looper.getMainLooper()` | 단, JS 스레드는 UI를 직접 그리지 않음 |
| 이벤트 루프 | `RunLoop` / `CFRunLoop` | `Looper` + `MessageQueue` | 큐에서 작업을 꺼내 실행하는 무한 루프 |
| 매크로태스크 큐 | `DispatchQueue.main.async {}` | `Handler.post {}` | setTimeout, I/O 콜백 등 |
| 마이크로태스크 큐 | 직접 대응 없음 (개념상 우선순위 높은 큐) | 직접 대응 없음 | Promise 콜백, `queueMicrotask` |
| `setTimeout(fn, 0)` | `DispatchQueue.main.async {}` | `Handler.post {}` | "다음 턴으로 미루기" |
| JS 스레드 블로킹 | 메인 스레드 블로킹 (ANR 아님, 하지만 유사한 죄) | 메인 스레드 블로킹 → ANR | RN에선 터치 응답·JS 애니메이션이 멈춤 |

## 왜 이렇게 설계됐나

JS는 브라우저에서 DOM을 조작하기 위해 태어났다. 여러 스레드가 동시에 DOM을 건드리면 락(lock)이 필요해지고, 락은 데드락과 레이스 컨디션을 낳는다. 그래서 JS는 **"스레드는 하나, 대신 절대 기다리지 않는다(non-blocking)"** 전략을 택했다.

- 동시성 문제(락, 뮤텍스, `@Synchronized`, actor isolation)가 언어 차원에서 존재하지 않는다. 어떤 코드도 실행 도중에 다른 JS 코드에게 선점당하지 않는다 (run-to-completion).
- 대신 오래 걸리는 작업(네트워크, 타이머, 파일 I/O)은 전부 **콜백 기반 비동기**로 위임하고, 완료되면 큐에 콜백을 넣어 나중에 실행한다.

Swift의 actor나 Kotlin의 single-threaded dispatcher가 주는 "이 안에서는 레이스가 없다"는 보장을, JS는 언어 전체가 기본으로 갖는 셈이다. 그 대가가 "블로킹하면 전부 멈춤"이다.

## 동작 원리

### 구성 요소

1. **콜스택 (Call Stack)** — 지금 실행 중인 함수 프레임. 네이티브의 스택과 동일한 개념. 하나뿐이다.
2. **태스크 큐 (매크로태스크 큐)** — `setTimeout`/`setInterval` 콜백, 네이티브 모듈에서 넘어온 이벤트(터치, 네트워크 응답) 등이 대기.
3. **마이크로태스크 큐** — `Promise.then/catch/finally` 콜백, `await` 이후 재개 지점, `queueMicrotask`가 대기. **매크로태스크보다 항상 먼저** 처리된다.

### 루프의 규칙

```
loop {
  매크로태스크 1개를 꺼내 콜스택에서 실행 (끝까지)
  마이크로태스크 큐를 전부 비운다 (실행 중 추가된 것 포함)
  (브라우저라면 여기서 렌더링 — RN에선 네이티브 UI 스레드가 별도로 존재)
}
```

핵심 규칙 두 가지:

- **run-to-completion**: 콜스택이 빌 때까지 아무것도 끼어들지 못한다. 실행 중인 함수가 3초 걸리면 3초 동안 터치 이벤트 콜백도, 타이머도 실행되지 않는다.
- **마이크로태스크 우선**: 하나의 매크로태스크가 끝날 때마다 마이크로태스크 큐를 *전부* 비운다. 그래서 `Promise.resolve().then(...)`은 `setTimeout(fn, 0)`보다 항상 먼저 실행된다.

### `setTimeout(fn, 0)`의 진짜 의미

"0ms 후 실행"이 아니라 **"현재 콜스택이 다 비고, 마이크로태스크도 다 처리된 다음 턴에 실행해달라"**는 뜻이다. `DispatchQueue.main.async {}`, `Handler.post {}`와 정확히 같은 관용구다. 무거운 작업을 잘게 쪼개 사이사이에 이벤트 처리 기회를 주는 용도로 쓴다. 실제 지연은 0ms가 보장되지 않는다(큐 상태에 따라 밀림).

### React Native 맥락

RN 앱에는 최소 3종류의 스레드가 있다:

- **UI(메인) 스레드** — 네이티브 뷰 렌더링, 터치 수신. iOS/AOS의 그 메인 스레드.
- **JS 스레드** — 앱 로직 전체([[Hermes]] 엔진 위에서 이벤트 루프가 돈다). 비즈니스 로직, [[State]] 갱신, [[Re-render]] 계산이 전부 여기서 실행된다.
- 그 외 백그라운드 스레드들 (레이아웃, I/O 등).

**JS 스레드에서 무거운 동기 연산 = 네이티브에서 메인 스레드 블로킹과 같은 죄다.** UI 스레드가 살아 있으니 스크롤 같은 네이티브 주도 동작은 일부 살아남지만, 버튼 탭 핸들러·[[State]] 갱신·JS 주도 애니메이션·네비게이션이 전부 멈춘다. [[New Architecture]]에서는 [[JSI]]를 통해 JS↔네이티브가 동기 호출도 가능해졌지만, "JS 스레드는 하나"라는 사실은 변하지 않는다. 진짜 병렬이 필요하면 네이티브 모듈로 내리거나 `react-native-worklets` 같은 별도 러너를 쓴다.

## 코드 예시

실행 순서 퀴즈. 출력 순서를 예측해보고 아래 답과 비교할 것.

```typescript
console.log('1: sync start');

setTimeout(() => {
  console.log('2: setTimeout 0');
}, 0);

Promise.resolve().then(() => {
  console.log('3: promise then');
  // 마이크로태스크 안에서 또 마이크로태스크를 추가하면?
  Promise.resolve().then(() => console.log('4: nested then'));
});

(async () => {
  console.log('5: async fn sync part'); // await 전까지는 동기 실행
  await Promise.resolve();
  console.log('6: after await');        // 여기부터 마이크로태스크
})();

console.log('7: sync end');
```

**정답: 1 → 5 → 7 → 3 → 6 → 4 → 2**

풀이:

1. 동기 코드가 콜스택에서 끝까지 실행: `1`, `5`(async 함수도 첫 await까지는 동기), `7`.
2. 콜스택이 비면 마이크로태스크 큐를 전부 비움: `3` → (`6`은 `3`보다 먼저 큐에 들어갔지만 `then` 등록 시점 기준으로 `3`이 먼저) → `6` → 실행 중 추가된 `4`까지 소진.
3. 마지막으로 매크로태스크: `2`.

블로킹의 파괴력을 보여주는 예시:

```typescript
function heavySync(ms: number) {
  const end = Date.now() + ms;
  while (Date.now() < end) {} // busy wait — 절대 이렇게 하지 말 것
}

// 이 5초 동안: 모든 터치 핸들러, setTimeout, Promise 콜백,
// setState로 인한 리렌더가 전부 정지한다.
heavySync(5000);
```

퀴즈 2 — await와 타이머의 조합:

```typescript
async function f() {
  console.log('A');
  await new Promise<void>((resolve) => setTimeout(resolve, 0));
  console.log('B'); // resolve가 매크로태스크(setTimeout)에서 호출되므로
                    // B는 "다음 매크로태스크 턴 이후"에 실행된다
}

f();
Promise.resolve().then(() => console.log('C'));
console.log('D');
// 정답: A → D → C → B
// A(동기) → D(동기) → C(마이크로태스크) → [매크로태스크: setTimeout이 resolve 호출] → B
```

교훈: `await`가 마이크로태스크냐 매크로태스크냐는 **기다리는 Promise가 언제/어디서 settle되느냐**에 달렸다. `await Promise.resolve()`는 바로 다음 마이크로태스크 턴, `await sleep(0)`은 매크로태스크 한 턴 뒤다.

쪼개기 관용구 (네이티브의 "chunk 단위로 main queue에 되돌리기"와 동일):

```typescript
async function processAll<T>(items: T[], work: (item: T) => void) {
  for (let i = 0; i < items.length; i++) {
    work(items[i]);
    if (i % 100 === 0) {
      // 이벤트 루프에 턴을 양보 — 이 사이에 터치/렌더가 처리됨
      await new Promise((resolve) => setTimeout(resolve, 0));
    }
  }
}
```

## 함정 (Pitfalls)

- **"async니까 다른 스레드에서 돌겠지"라는 착각.** `async` 함수의 본문도 전부 JS 스레드에서 실행된다. `Task.detached`나 `Dispatchers.Default` 같은 것이 없다. `async`는 스레드를 바꾸지 않고 *실행을 잘게 쪼개 인터리빙*할 뿐이다. CPU-bound 작업은 async로 감싸도 여전히 JS 스레드를 통째로 먹는다.
- **JSON.parse도 동기다.** 수 MB짜리 응답을 `JSON.parse`하면 그 시간만큼 JS 스레드가 멈춘다. 네이티브에서 메인 스레드에서 대용량 디코딩을 하는 것과 같다.
- **`setTimeout(fn, 0)`은 즉시가 아니다.** 큐가 밀려 있으면 수백 ms 뒤에 실행될 수 있다. 정밀 타이밍 용도로 쓰면 안 된다.
- **마이크로태스크 폭주.** 마이크로태스크 안에서 계속 마이크로태스크를 추가하면 매크로태스크(터치 이벤트 포함)가 영원히 실행되지 못한다. `while(true)`와 같은 효과.
- **레이스 컨디션이 "없다"고 방심하면 안 된다.** 데이터 레이스(메모리 동시 접근)는 없지만, `await` 지점마다 다른 코드가 끼어들 수 있으므로 논리적 레이스(check-then-act 사이에 상태가 바뀜)는 여전히 존재한다. Swift actor의 reentrancy 문제와 정확히 같은 부류다.
- **개발 모드 체감 성능.** 디버그 [[Bundle]]은 최적화가 꺼져 있어 JS 스레드가 훨씬 쉽게 밀린다. 성능 판단은 release 빌드로 할 것.
