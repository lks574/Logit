# 클로저와 스코프

> JS 클로저는 Swift 클로저와 무엇이 같고 무엇이 다르며, React Native에서 [[Stale Closure]]라는 함정이 왜 생기는가?

## iOS/AOS 대응 개념

| JS 개념 | iOS (Swift) | Android (Kotlin) | 차이 |
|---|---|---|---|
| 클로저(함수 + 캡처 환경) | closure `{ [captures] in ... }` | lambda / 익명 함수 | 개념 동일 |
| 렉시컬 스코프 | 렉시컬 스코프 | 렉시컬 스코프 | 동일 |
| 변수 캡처 | 기본 참조 캡처, 캡처 리스트로 값 캡처 가능 | 참조 캡처 (지역 변수도 outer 값 변경 관찰) | **JS엔 캡처 리스트가 없다. 항상 참조 캡처** |
| `[weak self]` | 있음 (retain cycle 방지) | 불필요 (GC) | JS도 GC라 불필요 — 대신 다른 문제([[Stale Closure]])가 있음 |
| `let` / `const` | `var` / `let` | `var` / `val` | **이름 매핑이 꼬여 있음: JS `const` ≈ Swift `let`** |
| `var` (JS 레거시) | 대응 없음 | 대응 없음 | 함수 스코프 + 호이스팅. 쓰지 말 것 |
| 블록 스코프 | `{}` 블록 스코프 | `{}` 블록 스코프 | JS는 `let`/`const`만 블록 스코프 |

## 왜 이렇게 설계됐나

JS에서 함수는 일급 객체이고, 비동기 프로그래밍이 전부 콜백/클로저로 이루어진다([[01-이벤트루프와-싱글스레드]]). 콜백이 자신이 정의된 시점의 변수에 접근할 수 없다면 이 모델은 성립하지 않는다. 그래서 JS의 모든 함수는 **정의된 위치의 스코프 체인을 자동으로, 전부, 참조로 캡처**한다.

Swift가 캡처 리스트를 도입한 이유(ARC의 retain cycle, 값/참조 캡처 선택)가 JS엔 없다:

- **GC(가비지 컬렉션)** 이라 순환 참조가 있어도 도달 불가능해지면 수거된다. `[weak self]` 상당이 필요 없다.
- 값 캡처를 선택할 문법이 없다. 값 캡처가 필요하면 새 변수에 복사해서 그 변수를 캡처하게 만든다.

대신 React의 함수형 컴포넌트 모델과 결합하면서 **[[Stale Closure]]**(오래된 렌더의 값을 물고 있는 클로저)라는 JS+React 고유의 함정이 생겼다. retain cycle이 "메모리를 못 놓는 문제"라면, stale closure는 "과거를 못 놓는 문제"다.

## 동작 원리

### 렉시컬 스코프

스코프는 **코드가 씌어진 위치**에서 정적으로 결정된다(호출 위치가 아님). 함수는 자신을 감싼 바깥 스코프들의 체인을 갖고, 변수 조회는 안→밖으로 올라간다. Swift/Kotlin과 동일하므로 직관대로 동작한다.

### let / const / var

- `const` — 재할당 불가. Swift `let`, Kotlin `val`. **단, 객체 내부는 변경 가능하다** (참조가 고정일 뿐). Swift에서 `let x: SomeClass`의 프로퍼티는 바꿀 수 있는 것과 같다. JS 객체는 전부 참조 타입이므로 struct 같은 값 타입 보호는 없다.
- `let` — 재할당 가능. Swift `var`, Kotlin `var`.
- `var` — ES5 레거시. 블록 스코프가 아니라 **함수 스코프**이고, 선언이 함수 최상단으로 끌어올려진다(호이스팅). 현대 코드에서는 사용 금지. lint로 차단하는 게 표준.

기본은 `const`, 재할당이 정말 필요할 때만 `let` — Swift에서 `let` 우선인 것과 같은 관행.

### 클로저 캡처: "참조 캡처"의 정확한 의미

JS 클로저는 **변수의 값을 복사하지 않고, 변수 자체(변수가 사는 환경 레코드)를 캡처**한다. 즉:

- 캡처된 변수를 나중에 바깥에서 **재할당**하면, 클로저는 새 값을 본다. (원시값이라도! "원시값은 값 복사"라는 통념은 *할당*에 대한 얘기고, *캡처*는 변수 바인딩 자체를 공유한다.)
- 같은 변수를 캡처한 여러 클로저는 그 변수를 **공유**한다.
- 캡처는 스냅샷이 아니다. 스냅샷이 필요하면 명시적으로 지역 변수에 복사해야 한다.

```typescript
let count = 0;
const read = () => count;
count = 42;
read(); // 42 — 캡처 시점의 0이 아니다. 변수를 공유하기 때문
```

Swift로 치면 캡처 리스트 없이 캡처한 `var`와 동일한 거동이다 (`{ [count] in }` 같은 값 캡처 선택지가 JS엔 없음).

`for` 루프에서 `let`으로 선언한 변수는 **반복(iteration)마다 새 바인딩**이 만들어진다. 그래서 루프 안에서 만든 클로저들이 각자 그 회차의 값을 본다. (`var`였다면 전부 마지막 값을 공유 — 고전적인 면접 문제.)

### React/RN에서 왜 중요한가

함수형 컴포넌트는 **렌더마다 함수 본문 전체가 다시 실행**된다. 그 렌더에서 만들어진 이벤트 핸들러·effect 콜백은 **그 렌더 시점의 props/[[State]] 변수**를 캡처한 클로저다. [[State]]가 바뀌어 [[Re-render]]가 일어나면 *새* 클로저들이 만들어지지만, 어딘가(타이머, 구독, 네이티브 콜백)에 등록해둔 *옛* 클로저는 여전히 옛 값을 물고 있다 — 이것이 [[Stale Closure]]다. SwiftUI의 `@State`는 프로퍼티 래퍼를 통해 항상 최신 저장소를 읽으므로 이 문제가 없고, 그래서 네이티브 개발자가 특히 잘 걸린다.

## 코드 예시

```typescript
// 1) 클로저 기본 — 카운터 팩토리
function makeCounter() {
  let count = 0;                 // 이 변수는 makeCounter 종료 후에도 살아 있다 (GC가 클로저 때문에 유지)
  return {
    increment: () => ++count,
    current: () => count,        // 두 클로저가 같은 count를 공유
  };
}
const c = makeCounter();
c.increment(); // 1
c.increment(); // 2
c.current();   // 2

// 2) 루프 캡처 — let은 회차마다 새 바인딩
const fns: Array<() => number> = [];
for (let i = 0; i < 3; i++) {
  fns.push(() => i);
}
fns.map((f) => f()); // [0, 1, 2]  (var였다면 [3, 3, 3])

// 3) 값 스냅샷이 필요할 때 — 명시적 복사 (캡처 리스트의 수동 구현)
function schedule(label: string) {
  const captured = label;        // 이후 label이 바뀌어도 captured는 고정
  setTimeout(() => console.log(captured), 1000);
}
```

비동기와 클로저의 결합 — check-then-act 레이스:

```typescript
let submitting = false;

async function submit(payload: object) {
  if (submitting) return;        // check
  submitting = true;             // act — 여기까지는 안전 (동기 구간은 원자적)
  try {
    await api.post('/log', payload); // 이 await 동안 다른 submit 호출이 끼어들 수 있지만
                                     // submitting 플래그가 참조 캡처라 최신값을 보고 걸러진다
  } finally {
    submitting = false;
  }
}
declare const api: { post(url: string, body: object): Promise<void> };
```

이 패턴이 동작하는 이유가 바로 "참조 캡처"다. 클로저가 `submitting`의 스냅샷을 떠갔다면 중복 방지 플래그는 성립하지 않는다.

RN에서의 [[Stale Closure]] 재현과 해법:

```typescript
import { useEffect, useRef, useState } from 'react';
import { Text } from 'react-native';

export function Timer() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    const id = setInterval(() => {
      // BAD: 이 클로저는 "첫 렌더의 count(=0)"를 캡처했다.
      // setCount(count + 1) 은 매번 0 + 1 → 영원히 1에서 멈춘다.
      // setCount(count + 1);

      // GOOD: 함수형 업데이트 — 최신 state를 인자로 받으므로 캡처에 의존하지 않음
      setCount((prev) => prev + 1);
    }, 1000);
    return () => clearInterval(id); // cleanup (deinit/onCleared 상당)
  }, []); // 빈 deps = 첫 렌더의 클로저를 계속 사용한다는 뜻

  return <Text>{count}</Text>;
}

// 항상 최신 값을 읽어야 하는 장수 콜백에는 ref 패턴
export function useLatest<T>(value: T) {
  const ref = useRef(value);
  ref.current = value;           // 렌더마다 갱신 — ref는 렌더를 넘어 살아남는 상자
  return ref;                    // 콜백에서 ref.current로 읽으면 항상 최신
}
```

## 함정 (Pitfalls)

- **[[Stale Closure]] 3대 발생지: 이벤트 핸들러, `setInterval`/`setTimeout`, `useEffect`.** 공통 패턴은 "클로저를 한 번 등록해두고 오래 쓰는 것". 등록 시점의 렌더에 얼어붙은 값을 계속 읽는다. 해법 우선순위: (1) [[State]] 갱신이면 함수형 업데이트 `setX(prev => ...)`, (2) effect의 deps 배열에 읽는 값을 정직하게 나열, (3) 정말 장수 콜백이면 `useRef`로 최신값 통로 확보.
- **`[weak self]` 반사신경은 접어둘 것.** JS는 GC라 클로저가 무언가를 캡처해도 retain cycle 걱정은 없다. 다만 **논리적 누수**는 있다: 구독/타이머를 해제하지 않으면 클로저와 그것이 캡처한 모든 것이 살아남는다. `deinit`에서 해제하던 습관을 effect cleanup(`return () => ...`)으로 옮기면 된다.
- **`const`가 불변을 의미하지 않는다.** `const arr = []`에 `arr.push(1)`은 합법. 재할당만 막는다. 값 수준 불변이 필요하면 `Object.freeze`(얕음), `readonly` 타입([[05-TypeScript-핵심]]), 혹은 불변 업데이트 관행(spread로 새 객체)을 쓴다. React는 참조 비교로 변경을 감지하므로 **불변 업데이트가 사실상 의무**다.
- **호이스팅과 TDZ.** `let`/`const`도 선언 전 접근 시 에러(temporal dead zone)가 나는데, 에러 메시지가 "not defined"가 아니라 "Cannot access before initialization"이라 처음엔 당황스럽다. 선언을 위로 올리면 그만.
- **`this`는 별개의 지뢰밭.** JS의 `this`는 렉시컬 캡처가 아니라 호출 방식으로 결정된다(화살표 함수만 렉시컬 `this`). RN 함수형 컴포넌트 + 화살표 함수 세계에선 거의 만날 일이 없지만, 클래스 기반 라이브러리 콜백에 메서드를 그대로 넘기면 `this`가 풀린다. 화살표 함수로 감싸는 게 안전하다.
- **deps 배열을 속이지 말 것.** ESLint `react-hooks/exhaustive-deps` 경고를 끄고 deps를 비우는 것은 "이 클로저는 영원히 첫 렌더의 세계에 산다"고 선언하는 것이다. 경고가 뜨면 deps를 채우거나 구조를 바꿔야지, 경고를 꺼선 안 된다.
