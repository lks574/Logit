# 모듈 시스템

> [!abstract] 한 줄 요약
> `import`/`export`는 Swift `import`·Gradle 모듈과 무엇이 다르고, [[Metro]]는 이것을 어떻게 하나의 [[Bundle]]로 만드는가?

## 🔁 iOS/AOS 대응 개념

| JS 개념 | iOS (Swift) | Android (Kotlin) | 차이 |
|---|---|---|---|
| 모듈 = **파일 1개** | 모듈 = 프레임워크/타깃 (파일 여러 개) | 모듈 = Gradle module | **JS는 파일 단위**. 훨씬 잘게 쪼개짐 |
| `export` | `public`/`open` | `public` | export 안 하면 파일 밖에서 접근 불가 (`internal`보다 좁은 file-private에 가까움) |
| `import { x } from './a'` | `import Framework` (전체) | `import com.a.x` (심볼) | Kotlin의 심볼 import에 가까움 + **상대 경로 사용** |
| default export | 직접 대응 없음 | 직접 대응 없음 | "이 파일의 대표 하나" 관행 |
| npm 패키지 | SPM package / CocoaPod | Maven artifact | [[06-패키지-매니저와-node_modules]] 참조 |
| [[Metro]] + [[Bundle]] | 컴파일 + 링크 → 바이너리 | R8/D8 → dex | 번들러가 모든 모듈을 파일 1개로 합침 |
| path alias (`@/...`) | 대응 없음 (모듈명이 곧 네임스페이스) | 대응 없음 | tsconfig + 번들러 설정으로 구현 |
| barrel (index.ts) | 프레임워크의 umbrella header 비슷 | 대응 없음 | 재수출 파일 관행 |

## 🧭 왜 이렇게 설계됐나

원래 JS(브라우저)엔 모듈이 없었다 — 모든 스크립트가 전역 스코프를 공유했다. 이를 해결하려고 Node.js가 **CommonJS**(`require`/`module.exports`, 동기 로딩)를 만들었고, 이후 언어 표준으로 **ESM**(ECMAScript Modules, `import`/`export`)이 들어왔다(ES2015).

둘의 결정적 설계 차이:

- **CommonJS**: `require`는 그냥 함수 호출. 런타임에, 조건문 안에서도, 동적 문자열로도 호출 가능. 유연하지만 정적 분석이 어렵다.
- **ESM**: `import`/`export`는 문(statement)이며 파일 최상단에 정적으로 선언된다. 실행 전에 의존성 그래프가 확정되므로 정적 분석, tree shaking(미사용 export 제거), 순환 참조의 부분적 해소가 가능하다.

RN 코드는 ESM 문법으로 작성하지만, [[Metro]]가 빌드 시 이를 CommonJS 스타일의 모듈 시스템으로 변환해 [[Bundle]]에 담는다. 즉 **문법은 ESM, 런타임 거동은 CommonJS에 가깝다**. 네이티브의 "컴파일 타임 모듈 해석"과 달리, 어느 쪽이든 모듈 해석은 빌드 도구의 몫이다.

## ⚙️ 동작 원리

### ESM 문법: named vs default

```typescript
// math.ts
export const PI = 3.14159;                    // named export (여러 개 가능)
export function add(a: number, b: number) { return a + b; }
export default class Calculator { /* ... */ } // default export (파일당 최대 1개)
```

```typescript
// consumer.ts
import Calculator, { PI, add } from './math';  // default는 이름 자유, named는 이름 고정
import { add as plus } from './math';          // 이름 바꾸기
import * as math from './math';                // 네임스페이스로 전부
export { PI } from './math';                   // 재수출 (barrel의 재료)
```

- **named export**: 이름이 계약이다. 오타나 없는 심볼은 TS가 잡는다. IDE 자동 import·리팩터링이 잘 동작한다.
- **default export**: import하는 쪽이 이름을 마음대로 짓는다. 파일마다 다른 이름으로 불리는 사고가 가능. 팀 컨벤션으로 named를 선호하는 곳이 많다 (React 컴포넌트만 default로 하는 절충안도 흔함 — Expo Router처럼 default export를 *요구*하는 프레임워크도 있으니 프레임워크 규약 우선).

### CommonJS와의 관계

```javascript
// CommonJS (node_modules의 오래된 패키지, metro.config.js 등에서 여전히 만남)
const { add } = require('./math');
module.exports = { add };
```

TS/[[Metro]] 환경에서는 ESM으로 작성해도 CJS 패키지를 `import`로 불러올 수 있게 상호 운용을 처리해준다(`esModuleInterop`). 직접 `require`를 쓸 일은 설정 파일과 일부 동적 로딩 외엔 거의 없다.

### Metro에서의 동작 개요

1. 진입점(예: `index.js`)부터 `import`를 따라 **의존성 그래프**를 구축한다.
2. 각 파일을 변환(Babel: TS 제거, JSX 변환)하고 모듈 ID를 부여한다.
3. 전부 합쳐 **하나의 [[Bundle]]** 파일로 만든다. 각 모듈은 함수로 감싸져 최초 `require` 시점에 1회 실행되고 결과가 캐시된다 — **모듈은 싱글턴**이다. 모듈 최상위에 쓴 코드는 앱 시작(정확히는 최초 import) 시 실행된다.
4. 개발 중엔 파일 변경 시 해당 모듈만 교체(Fast Refresh).

정적 그래프가 전제라서 **`import`의 경로는 문자열 리터럴이어야 한다**. `import(someVariable)` 같은 동적 경로는 Metro가 그래프에 넣을 수 없다.

### 순환 의존성 (circular dependency)

A가 B를 import하고 B가 A를 import하면, 둘 중 하나는 **아직 초기화가 끝나지 않은 모듈**을 받게 된다. 크래시가 아니라 `undefined`가 조용히 흘러들어오는 경우가 많아 악질이다. 증상: "분명 export했는데 import하면 undefined", 특히 모듈 최상위에서 그 값을 바로 사용할 때. Swift/Kotlin에서는 모듈 간 순환이 빌드 에러로 원천 차단되지만 JS는 **경고 없이 통과**한다(Metro 설정이나 ESLint `import/no-cycle`로 탐지 가능).

### barrel 파일 (index.ts)

디렉토리의 공개 API를 한 파일로 모아 재수출하는 관행:

```typescript
// components/index.ts
export { Button } from './Button';
export { Card } from './Card';
```

- 장점: import 경로가 짧아지고(`from '@/components'`), 디렉토리의 공개 표면이 명시된다 — 프레임워크의 public 인터페이스와 비슷한 효과.
- 단점: barrel 하나를 import하면 **그 barrel이 재수출하는 모든 모듈이 로드**된다(앱 시작 비용 증가). barrel끼리 얽히면 순환 의존성의 온상이 된다. 큰 프로젝트에선 "외부 공개용 barrel만 허용, 내부끼리는 직접 경로" 규칙이 흔하다.

### path alias

`../../..` 지옥을 피하기 위한 별칭. **두 군데를 맞춰야 한다**: TS(타입 검사용)와 번들러(실제 해석용).

```jsonc
// tsconfig.json
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": { "@/*": ["src/*"] }
  }
}
```

Expo 프로젝트는 `experiments.tsconfigPaths`(SDK 50+ 기본 활성)로 [[Metro]]가 tsconfig paths를 그대로 읽는다. 순수 RN이면 `babel-plugin-module-resolver`로 Babel 쪽에도 같은 매핑을 선언해야 한다. **tsconfig만 고치면 타입 검사는 통과하는데 번들이 깨지는** 흔한 사고의 원인.

## 💻 코드 예시

순환 의존성 최소 재현:

```typescript
// a.ts
import { bValue } from './b';
export const aValue = 'A';
console.log('a.ts 로드, bValue =', bValue);

// b.ts
import { aValue } from './a';
export const bValue = 'B';
console.log('b.ts 로드, aValue =', aValue); // undefined! a.ts가 아직 초기화 중
```

`a.ts`가 먼저 로드되면: a → b를 import → b가 a를 import하지만 a는 초기화 미완 → b는 `aValue = undefined`를 관찰. 해법: 공유 대상을 제3의 모듈로 추출하거나, 사용 시점을 함수 안으로 늦추거나(지연 접근), 의존 방향을 한쪽으로 정리.

지연 접근으로 순환 끊기:

```typescript
// b.ts — 최상위에서 aValue를 쓰지 않고, 실제 사용 시점(함수 호출)으로 늦춘다.
// 함수가 호출될 즈음엔 a.ts 초기화가 끝나 있으므로 안전하다.
import { aValue } from './a';
export function describeA() {
  return `a is ${aValue}`; // 최상위가 아닌 함수 본문에서 접근
}
```

alias + barrel 실사용:

```typescript
// src/features/workout/index.ts (barrel — 이 feature의 공개 API)
export { WorkoutCard } from './components/WorkoutCard';
export { useWorkout } from './hooks/useWorkout';
export type { Workout } from './types';

// src/screens/HomeScreen.tsx
import { WorkoutCard, useWorkout, type Workout } from '@/features/workout';
```

## ⚠️ 함정 (Pitfalls)

- **모듈 최상위 코드는 import 즉시 실행된다.** 최상위에서 무거운 초기화(SDK init, 큰 테이블 생성)를 하면 그 모듈을 import하는 순간(대개 앱 시작) 비용을 치른다. 초기화는 함수로 감싸 필요 시점에 호출.
- **모듈은 싱글턴 — 상태를 담으면 전역 상태다.** 모듈 최상위의 `let cache = {}`는 앱 전체가 공유한다. 의도적으로 쓰면 유용하지만(간이 싱글턴), 테스트 간 오염의 주범이기도 하다.
- **default export 이름 불일치.** `import Btn from './Button'`도 `import Button from './Button'`도 전부 합법이라 리네임 리팩터링에서 누락된다. named export 위주가 안전.
- **순환 의존성은 에러가 아니다.** "가끔 undefined" 류의 미스터리 버그를 만나면 순환부터 의심. `npx madge --circular src/` 같은 도구로 주기적 점검.
- **대소문자.** macOS 파일시스템은 대소문자 비구분이 기본이라 `import './button'`(실제 파일 `Button.tsx`)이 로컬에선 되고 CI(Linux)에서 깨질 수 있다.
- **type-only import.** 타입만 쓸 거면 `import type { Workout } from ...`로 표기하는 습관을 들이면 런타임 의존이 생기지 않아 순환 문제를 피하는 데도 도움이 된다.
- **네이티브 모듈과 혼동 금지.** 여기서 말한 모듈은 전부 JS 파일 얘기다. 네이티브 코드가 딸린 패키지는 [[Autolinking]]과 pod install이 별도로 필요하다 — [[06-패키지-매니저와-node_modules]] 참조.
