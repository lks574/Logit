# 패키지 매니저와 node_modules

> [!abstract] 한 줄 요약
> package.json·lockfile·node_modules는 CocoaPods/SPM/Gradle의 무엇에 해당하며, RN에서 "npm install만으로 끝나지 않는" 패키지는 무엇인가?

## 🔁 iOS/AOS 대응 개념

| JS 개념 | iOS | Android | 비고 |
|---|---|---|---|
| `package.json` | `Podfile` / `Package.swift` | `build.gradle(.kts)` | 의존성 선언 + 스크립트 + 메타데이터 |
| `dependencies` | pod / SPM dependency | `implementation` | 앱 실행에 필요 |
| `devDependencies` | (빌드 도구는 별도 관리) | (Gradle plugin, `testImplementation`에 가까움) | 빌드·테스트·lint 도구. [[Bundle]]에 안 들어감 |
| lockfile | `Podfile.lock` / `Package.resolved` | Gradle lockfile / versions catalog 고정 | 정확한 해석 결과 고정 |
| `node_modules/` | `Pods/` | `~/.gradle` 캐시 | **단, 프로젝트 로컬 + 어마어마한 파일 수** |
| npm registry | CocoaPods trunk / GitHub | Maven Central | 중앙 레지스트리 |
| semver `^` `~` | pod의 `~>` | `+` 버전 범위 | 범위 지정 문법 |
| `npx` | `mint run` 비슷 | `./gradlew`의 wrapper 정신과 유사 | 로컬/임시 설치된 CLI 실행 |
| `scripts` | Xcode scheme / fastlane lane | Gradle task | `npm run <이름>` |

## 🧭 왜 이렇게 설계됐나

npm 생태계의 특징은 **초소형 패키지의 초대형 그래프**다. 네이티브에서 의존성 10~30개가 보통이라면, RN 앱의 `node_modules`엔 전이 의존성 포함 수백~천 개 패키지가 들어온다. 이 규모 때문에:

- **버전 범위(semver) + lockfile 분리**: `package.json`엔 "허용 범위"만 적고, 실제로 해석된 정확한 버전 트리는 lockfile에 기록한다. `Podfile`/`Podfile.lock` 관계와 동일한 철학인데 그래프가 훨씬 크다.
- **프로젝트 로컬 설치**: 전역 공유 캐시 대신 프로젝트마다 `node_modules`를 둔다(디스크는 낭비되지만 프로젝트 간 간섭이 없다). pnpm은 전역 저장소 + 하드링크로 이 낭비를 해결한 구현.
- npm은 원래 같은 패키지의 **여러 버전 동시 설치**를 허용한다. A가 lodash@3, B가 lodash@4를 요구하면 둘 다 들어간다. 네이티브에선 불가능한(단일 버전으로 수렴해야 하는) 일이다. 단, RN의 네이티브 모듈은 결국 하나의 바이너리에 링크되므로 이 자유가 통하지 않는다 — 아래 RN 특수사항 참조.

## ⚙️ 동작 원리

### package.json

```jsonc
{
  "name": "logit",
  "version": "1.0.0",
  "main": "index.js",
  "scripts": {
    "start": "expo start",        // npm run start (start/test 등 일부는 npm start로 축약 가능)
    "typecheck": "tsc --noEmit",
    "lint": "eslint ."
  },
  "dependencies": {               // 앱 런타임에 필요 → JS [[Bundle]]에 포함될 수 있음
    "react": "19.1.0",
    "react-native": "0.81.4",
    "expo": "~54.0.0"
  },
  "devDependencies": {            // 개발 도구 → 번들에 안 들어감
    "typescript": "~5.9.0",
    "@types/react": "~19.1.0",
    "eslint": "^9.0.0"
  }
}
```

- `dependencies` vs `devDependencies`: 앱 코드가 `import`하는 것은 dependencies, 빌드/검사 도구(TS, ESLint, Babel 플러그인, `@types/*`)는 devDependencies. 앱 프로젝트에선 섞여도 당장 티가 안 나지만(어차피 다 설치됨), 번들 위생과 의도 표현을 위해 구분한다.
- `scripts`: 셸 명령 별칭. `npm run <name>`으로 실행하며, 이때 `node_modules/.bin`이 PATH에 추가되어 로컬 설치된 CLI(tsc, eslint)가 그대로 불린다.

### semver와 `^` `~`

버전은 `MAJOR.MINOR.PATCH`. semver 규약상 MAJOR는 breaking change, MINOR는 기능 추가(호환), PATCH는 버그 픽스.

- `^1.2.3` — MAJOR 고정: `>=1.2.3 <2.0.0`. npm install 시 기본으로 붙는 접두사.
- `~1.2.3` — MINOR까지 고정: `>=1.2.3 <1.3.0`.
- `1.2.3` — 정확히 그 버전.
- 주의: `^0.2.3`은 `<0.3.0`으로 해석된다 (0.x에선 MINOR가 breaking 취급). RN 자체가 0.x라 이 규칙이 실제로 중요하다.

"semver는 규약이지 보장이 아니다" — patch 업데이트가 실제로 뭔가를 깨는 일은 흔하다. 그래서 lockfile이 필수다.

### lockfile

`package-lock.json`(npm) / `yarn.lock` / `pnpm-lock.yaml`. 범위 선언을 실제 버전 트리로 해석한 **결과 전체**(정확한 버전 + 무결성 해시 + 전이 의존성 포함)를 기록한다.

- **반드시 커밋한다.** `Podfile.lock`을 커밋하는 것과 같은 이유 — 팀원·CI가 동일한 트리를 재현.
- CI에서는 `npm install` 대신 `npm ci`(lockfile 그대로 클린 설치, 불일치 시 실패)를 쓴다.
- lockfile 충돌은 손으로 머지하지 말고 `npm install`을 다시 돌려 재생성.

### npm / yarn / pnpm

| | npm | yarn (berry) | pnpm |
|---|---|---|---|
| 위치 | Node 동봉 (기본) | 별도 설치 | 별도 설치 |
| node_modules 구조 | 호이스팅(평탄화) | 설정에 따라 다양 | 전역 저장소 + 심링크 (엄격) |
| 강점 | 표준, 무난 | 워크스페이스 성숙 | 디스크·속도, 유령 의존성 차단 |

기능은 대부분 수렴했다. **한 프로젝트에는 하나만** 쓴다(lockfile이 제각각이라 섞으면 트리가 달라짐). 프로젝트에 어떤 lockfile이 있는지 보고 그 도구를 따르면 된다. pnpm의 엄격한 심링크 구조는 일부 RN 도구와 궁합 문제가 있었던 역사가 있으니 RN에서 채택 시 공식 문서/템플릿 확인.

### node_modules 구조와 호이스팅

npm/yarn classic은 의존성을 최대한 **최상위로 평탄화(호이스팅)** 한다. A→C, B→C면 C를 `node_modules/C` 한 곳에 두고, 버전 충돌 시에만 `node_modules/A/node_modules/C`처럼 중첩한다.

부작용이 **유령 의존성(phantom dependency)**: 내 `package.json`에 선언하지 않은 패키지도 (다른 패키지가 끌어온 덕에) 최상위에 존재해서 `import`가 *되어버린다*. 그 패키지가 어느 날 트리에서 사라지면 내 코드가 깨진다. pnpm은 구조적으로 이를 차단한다.

`node_modules`는 재생성 가능한 산출물이다: `.gitignore` 대상이며, 꼬였을 때의 최후 수단이 `rm -rf node_modules && npm install`(+ iOS면 pod 재설치)이다.

### npx

`npx <cli>` — 로컬 `node_modules/.bin`의 CLI를 실행하고, 없으면 임시로 받아 실행. 전역 설치 없이 일회성 도구를 돌리는 표준 방법. RN에서 흔한 예: `npx expo start`, `npx pod-install`, `npx react-native@latest init`.

### RN 특수사항: JS-only vs 네이티브 코드 포함 패키지

npm 패키지는 두 부류로 나뉜다:

1. **JS-only** (lodash, zustand, date-fns...) — 설치하면 끝. [[Metro]]가 [[Bundle]]에 넣어준다.
2. **네이티브 코드 포함** (react-native-reanimated, react-native-mmkv 등 — 대개 이름이 `react-native-*`이거나 `expo-*`) — JS 위에 Swift/Kotlin/C++ 코드가 딸려 있다. 설치 후:
   - [[Autolinking]]이 `node_modules`를 스캔해 네이티브 프로젝트에 자동 연결하고 (수동으로 Podfile/settings.gradle을 편집하던 과거 방식의 자동화),
   - iOS는 `pod install`(또는 Expo면 `npx expo prebuild` 흐름)로 CocoaPods 의존성을 설치하고,
   - **네이티브를 다시 빌드**해야 한다. JS만 리로드해서는 절대 반영되지 않는다.

[[Expo Go]]는 미리 빌드된 네이티브 바이너리라서 **Expo Go에 포함되지 않은 네이티브 모듈은 추가할 수 없다** — 그래서 커스텀 네이티브 의존성이 필요하면 [[Dev Client]](직접 빌드한 개발용 앱)로 넘어간다. 어떤 패키지가 네이티브 코드를 포함하는지는 README나 `npx expo install` 호환성 안내로 확인하고, Expo 프로젝트에서는 `npm install` 대신 **`npx expo install`**을 쓰면 현재 SDK와 호환되는 버전을 골라준다.

## 💻 코드 예시

일상 명령 치트시트:

```bash
# 설치
npm install                       # package.json + lockfile 기준 전체 설치 (pod install 상당)
npm ci                            # CI용: lockfile 엄격 재현
npx expo install expo-haptics     # Expo: SDK 호환 버전으로 설치

# 추가/제거
npm install zustand               # dependencies에 추가 (^ 접두사로 기록됨)
npm install -D @types/lodash      # devDependencies에 추가
npm uninstall zustand

# 조사
npm ls react                      # 이 패키지가 트리 어디에, 몇 버전으로 있나 (중복 진단)
npm outdated                      # 범위 내/밖 업데이트 현황
npm why react-dom                 # 누가 이걸 끌어왔나 (pnpm/yarn 도 유사 명령 제공)

# 네이티브 모듈 설치 후 (bare RN iOS)
npx pod-install                   # cd ios && pod install 래퍼

# 꼬였을 때 (마지막 수단)
rm -rf node_modules && npm install
```

semver 범위 감각 확인:

```jsonc
{
  "dependencies": {
    "libA": "^2.3.1",   // 2.3.1 이상 3.0.0 미만 — 2.9.x 까지 허용
    "libB": "~2.3.1",   // 2.3.1 이상 2.4.0 미만
    "libC": "2.3.1",    // 정확히 2.3.1
    "libD": "^0.4.2"    // 0.4.2 이상 0.5.0 미만 (0.x 특례 — RN 버전 지정에서 중요)
  }
}
```

## ⚠️ 함정 (Pitfalls)

- **"설치했는데 크래시": 네이티브 리빌드 누락.** 네이티브 코드 포함 패키지를 넣고 JS 리로드만 하면 "Native module cannot be found" 류 에러. 반드시 pod install + 앱 재빌드. 반대로 제거할 때도 리빌드 필요.
- **[[Expo Go]]에서 안 되는 걸 계속 붙잡는 것.** Expo Go에 없는 네이티브 모듈은 어떤 설정으로도 Expo Go에서 못 돌린다. [[Dev Client]] 빌드로 전환이 정답.
- **lockfile 미커밋 / 무시.** "내 머신에선 되는데"의 최대 공급원. lockfile은 커밋하고, CI는 `npm ci`.
- **패키지 매니저 혼용.** yarn으로 설치하던 프로젝트에 npm install을 치면 lockfile이 2개가 되고 트리가 갈라진다. 저장소에 있는 lockfile 종류를 먼저 확인.
- **react/react-native 중복 인스턴스.** 의존성이 각자 react를 끌어와 두 벌이 설치되면 "Invalid hook call" 같은 난해한 에러가 난다. `npm ls react`로 단일 버전 수렴 확인. 라이브러리들이 react를 `peerDependencies`(호스트 앱이 제공한다는 선언)로 두는 이유가 이것이다.
- **유령 의존성.** import가 된다고 선언된 의존성이 아니다. 직접 import하는 패키지는 반드시 자신의 package.json에 선언.
- **버전을 손으로 올릴 때 Expo SDK 호환 무시.** RN/Expo 생태계는 버전 매트릭스가 빡빡하다(RN 버전 ↔ Expo SDK ↔ 각 라이브러리). 임의 업그레이드보다 `npx expo install --fix`, SDK 업그레이드 가이드(공식 문서 확인)를 따르는 게 안전.
- **postinstall 스크립트 = 임의 코드 실행.** 패키지 설치는 스크립트 실행을 동반할 수 있다. 공급망 공격의 통로이므로, 잘 모르는 패키지를 습관적으로 install하는 문화는 위험하다. 다운로드 수·유지보수 상태 확인 습관.
