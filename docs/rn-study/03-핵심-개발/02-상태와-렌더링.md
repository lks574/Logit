# 상태와 렌더링

> [!abstract] 한 줄 요약
> [[State]]가 바뀌면 컴포넌트 **함수 전체가 다시 실행**된다 — 이 한 문장이 React 멘탈모델의 전부이고, [[Stale Closure]]부터 useEffect 오용까지 모든 버그가 여기서 갈라진다.

## 🔁 iOS/AOS 대응 개념

| RN ([[Hook]]) | iOS (SwiftUI) | Android (Compose) | 역할 |
|---|---|---|---|
| `useState` | `@State` | `remember { mutableStateOf() }` | 렌더를 유발하는 로컬 상태 |
| `useEffect` | `.onAppear`/`.onChange`/`.task` | `LaunchedEffect`/`DisposableEffect` | 외부 시스템과의 동기화(부수효과) |
| `useRef` | 프로퍼티에 값 보관 (뷰 갱신 없이) | `remember { mutableStateOf() }` 없이 `remember { }` 로 보관한 값 | 렌더와 무관한 가변 값 / 뷰 참조 |
| `useContext` | `@Environment`/`@EnvironmentObject` | `CompositionLocal` | 트리 하향 암묵적 주입 |
| [[Re-render]] | `body` 재평가 | Recomposition | 상태 변경 → UI 함수 재실행 |
| [[Reconciliation]] | SwiftUI의 뷰 diff (identity 기반) | Compose의 positional memoization | 이전/이후 선언 비교 후 최소 반영 |

## 🧭 왜 이렇게 설계됐나

명령형 UI(UIKit)의 고질병은 "상태와 화면이 어긋나는" 것이다 — `isLoading = false`로 바꿨는데 스피너 숨기는 걸 깜빡하는 류의 버그. React의 답은 **화면을 상태의 순수 함수로 만들기**: 상태가 바뀌면 그냥 함수를 다시 돌려서 "새 화면 선언"을 얻고, 프레임워크가 이전 선언과 diff([[Reconciliation]])해서 네이티브 뷰에 최소 변경만 적용한다. SwiftUI가 `body`를 다시 평가하고 Compose가 recompose하는 것과 동일한 설계이며, "함수 재실행이 싸고 실제 뷰 변경은 프레임워크가 최소화한다"는 전제도 같다.

[[Hook]]이 함수 재실행 사이에서 상태를 유지하는 방법은 **호출 순서**다. React는 컴포넌트별로 훅 호출을 순서대로 슬롯에 저장하므로, 훅을 조건문/반복문 안에서 호출하면 안 된다는 "Rules of Hooks"가 생긴다. Compose의 positional memoization과 유사한 메커니즘이고, SwiftUI가 property wrapper 저장소를 뷰 identity에 붙이는 것과 목적이 같다.

## ⚙️ 동작 원리

### 핵심 멘탈모델: 렌더 = 함수 재실행

```tsx
function Counter() {
  const [count, setCount] = useState(0);
  const label = `현재: ${count}`;        // 매 렌더마다 새로 계산
  const onPress = () => setCount(count + 1); // 매 렌더마다 새 함수 객체
  return <Pressable onPress={onPress}><Text>{label}</Text></Pressable>;
}
```

`setCount` 호출 → React가 이 컴포넌트를 다시 렌더 예약 → `Counter()` 함수가 **처음부터 다시 실행**된다. `label`, `onPress` 모두 새로 만들어진다. `useState`는 "이 슬롯에 저장된 최신 값"을 돌려주므로 `count`만 이어진다. UIKit의 "객체가 살아있고 메서드로 일부만 고친다"는 그림을 버려야 한다. **각 렌더는 그 시점 상태의 스냅샷**이며, 렌더 안에서 만든 클로저는 그 스냅샷을 캡처한다(→ [[Stale Closure]]).

두 가지 따름정리:
- `count`는 렌더 안에서 상수다. `setCount(1); console.log(count)`를 해도 `count`는 여전히 이전 값 — 다음 렌더에서야 새 값이다. `@State`가 즉시 갱신되는 SwiftUI와 다른 지점.
- 이전 값 기반 갱신은 함수형 업데이트로: `setCount(prev => prev + 1)`. 연속 호출·비동기 콜백에서도 안전하다.

### Re-render 전파 규칙

[[Re-render]]는 **상태를 가진 컴포넌트에서 시작해 그 자식 트리 전체로** 내려간다. 기본 규칙: **부모가 리렌더되면 자식도 리렌더된다 — [[Props]]가 안 바뀌었어도.** props 비교로 스킵하려면 `React.memo`([[Memoization]])가 필요하며, 이는 opt-in이다. Compose가 파라미터 안정성(stable)으로 자동 스킵하는 것과 다르고, SwiftUI가 `body` 의존성을 추적해 필요한 뷰만 다시 평가하는 것과도 다르다. React는 기본이 "다시 실행", 최적화가 수동이다.

단, 리렌더 ≠ 네이티브 뷰 갱신. 함수 재실행 결과를 [[Reconciliation]]이 diff해서, 실제로 바뀐 부분만 [[Fabric]]을 통해 네이티브 뷰에 커밋한다. 그래서 "리렌더가 좀 많아도" 대개 문제없고, 문제가 되는 건 무거운 계산이나 거대한 리스트가 함께 재실행될 때다.

### useEffect — lifecycle 콜백이 아니라 동기화 도구

네이티브 개발자가 가장 크게 다치는 곳. `useEffect(fn, deps)`의 정확한 의미는:

> **렌더가 끝난 뒤, `deps` 배열의 값 중 하나라도 이전 렌더와 다르면(`Object.is` 비교) `fn`을 실행한다. 실행 전에 직전 effect의 cleanup을 먼저 부른다.**

- `deps = []` → 비교 대상이 없어 마운트 후 1회 + 언마운트 시 cleanup. *결과적으로* `viewDidLoad`/`onAppear`처럼 보일 뿐이다.
- `deps = [userId]` → `userId`가 바뀐 렌더 후마다 재실행. `onChange(of: userId)` / `LaunchedEffect(userId)`에 대응.
- `deps` 생략 → 매 렌더 후 실행(거의 항상 실수).

현대 React의 관점: **useEffect는 "이 컴포넌트의 상태를 외부 시스템(네트워크, 구독, 타이머, 네이티브 모듈)과 동기화하는 도구"**다. "마운트 시점에 뭘 한다"가 아니라 "이 값들이 이 상태인 동안 이 동기화가 유지된다"로 읽어야 한다. cleanup은 `deinit`가 아니라 "직전 동기화 해제"다 — 언마운트 때만 아니라 **deps가 바뀔 때마다** 불린다. `DisposableEffect`의 `onDispose`와 같은 의미론이다.

`viewDidLoad`처럼 쓰다 생기는 전형적 버그: `[]`로 fetch를 걸고 `userId`를 클로저에서 쓰면, 유저가 바뀌어도 첫 유저 데이터만 보인다. deps에 넣었어야 할 값을 빼먹은 것 — 그래서 lint 규칙(`exhaustive-deps`)이 "effect 안에서 쓰는 반응형 값은 전부 deps에"라고 강제한다.

또 하나: React 18+의 개발 모드(StrictMode)는 effect를 **일부러 mount→cleanup→mount 두 번 실행**해서 cleanup 누락을 드러낸다. "두 번 불려요"는 버그가 아니라 진단 기능이다.

### useRef vs useState

| | `useState` | `useRef` |
|---|---|---|
| 값 변경 시 리렌더 | O | X |
| 값 읽기 | 렌더 스냅샷 | `.current`에서 항상 최신 |
| 용도 | 화면에 보이는 것 | 타이머 ID, 이전 값, `TextInput` 포커스 제어 등 |

기준은 하나: **화면에 보여야 하면 state, 아니면 ref.** `useRef`는 "리렌더를 유발하지 않는 인스턴스 변수 자리"다 — UIKit에서 뷰컨트롤러 프로퍼티에 슬쩍 담아두던 그 값들의 새 집. 렌더 중에 `.current`를 읽어 화면을 만들면 안 된다(갱신돼도 다시 안 그려지므로).

### useContext

트리 아래로 props를 계단식으로 넘기는 "prop drilling"을 피하는 주입 메커니즘. `Provider`가 감싼 서브트리 어디서든 `useContext(ThemeContext)`로 꺼낸다. `@Environment` / `CompositionLocal`과 동일한 개념. 주의: **Provider의 value가 바뀌면 그 context를 구독하는 모든 컴포넌트가 리렌더**되므로, 자주 바뀌는 값(스크롤 위치 등)을 context에 넣지 않는다.

## 💻 코드 예시

RN 0.76+ / TypeScript. 네 훅 + cleanup + [[Stale Closure]] 회피를 한 화면에.

```tsx
import { createContext, useContext, useEffect, useRef, useState } from 'react';
import { View, Text, Pressable, StyleSheet } from 'react-native';

const ThemeContext = createContext<'light' | 'dark'>('light');

function useCountdown(seconds: number) {
  const [remaining, setRemaining] = useState(seconds);

  useEffect(() => {
    setRemaining(seconds); // seconds가 바뀌면 리셋
    const id = setInterval(() => {
      // 함수형 업데이트 → interval 클로저가 낡아도(stale) 항상 최신값 기준
      setRemaining(prev => (prev > 0 ? prev - 1 : 0));
    }, 1000);
    return () => clearInterval(id); // cleanup: seconds 변경 시 + 언마운트 시
  }, [seconds]);

  return remaining;
}

function TimerScreen() {
  const theme = useContext(ThemeContext);
  const [duration, setDuration] = useState(10);
  const remaining = useCountdown(duration);

  // 리렌더와 무관한 값: 버튼 탭 횟수를 로그용으로만 추적
  const tapCount = useRef(0);

  return (
    <View style={[styles.box, theme === 'dark' && styles.dark]}>
      <Text style={styles.big}>{remaining}s</Text>
      <Pressable
        onPress={() => {
          tapCount.current += 1;            // 리렌더 없음
          setDuration(d => d + 10);         // 리렌더 유발
          console.log('taps:', tapCount.current);
        }}
      >
        <Text>+10초 (탭 {tapCount.current}회는 다음 렌더에나 보임)</Text>
      </Pressable>
    </View>
  );
}

export default function App() {
  return (
    <ThemeContext.Provider value="light">
      <TimerScreen />
    </ThemeContext.Provider>
  );
}

const styles = StyleSheet.create({
  box: { flex: 1, alignItems: 'center', justifyContent: 'center', gap: 12 },
  dark: { backgroundColor: '#222' },
  big: { fontSize: 48, fontVariant: ['tabular-nums'] },
});
```

### Stale Closure 실전 예시

```tsx
// ❌ 버그: []로 만든 interval 클로저는 count=0 스냅샷을 영원히 캡처
const [count, setCount] = useState(0);
useEffect(() => {
  const id = setInterval(() => setCount(count + 1), 1000); // 항상 0+1
  return () => clearInterval(id);
}, []); // count가 deps에 없다 → 1에서 멈춘다

// ✅ 수정 1: 함수형 업데이트 — 클로저가 값을 캡처할 필요 자체를 없앤다
useEffect(() => {
  const id = setInterval(() => setCount(prev => prev + 1), 1000);
  return () => clearInterval(id);
}, []);

// ✅ 수정 2: deps에 count — 매초 interval을 재설치 (동작하지만 비효율)
```

Swift 클로저의 캡처와 같은 현상이다. 다른 점은 빈도 — React는 **모든 렌더가 새 클로저를 만들기 때문에** "어느 렌더의 클로저가 살아남았나"가 항상 문제가 된다.

## ⚠️ 함정 (Pitfalls)

- **`setState`는 동기적으로 값을 안 바꾼다.** 같은 렌더 안에서 set 직후 읽으면 옛 값. "다음 렌더의 입력을 예약"하는 것이다. 참고로 React 18+는 이벤트 핸들러 안의 여러 `set` 호출을 **자동 배칭**해 한 번의 리렌더로 합친다 — `setA(); setB();`가 두 번 그리지 않는다.
- **`useEffect` 콜백 자체를 `async`로 만들 수 없다.** effect는 cleanup 함수(또는 undefined)를 반환해야 하는데 async 함수는 Promise를 반환하기 때문. 내부에 async 함수를 선언하고 즉시 호출하는 패턴을 쓴다:
  ```tsx
  useEffect(() => {
    let cancelled = false;
    (async () => {
      const data = await fetchUser(userId);
      if (!cancelled) setUser(data);   // 언마운트/유저 변경 후 setState 방지
    })();
    return () => { cancelled = true; };
  }, [userId]);
  ```
- **state를 직접 변이(mutate)하면 리렌더가 안 된다.** `todos.push(x); setTodos(todos)`는 같은 참조라 변경 감지 실패. 항상 새 객체/배열: `setTodos([...todos, x])`. 값 타입(struct) 세계에서 온 Swift 개발자가 참조 타입 배열에서 자주 걸린다.
- **useEffect를 viewDidLoad로 취급**: "마운트 때 한 번"이라는 생각으로 `[]`를 쓰고 안에서 props/state를 참조 → 값이 바뀌어도 effect는 옛 세상에 산다. deps는 "실행 시점 지정"이 아니라 **"이 effect가 의존하는 반응형 값 목록"**이다.
- **cleanup이 언마운트 때만 불린다고 착각**: deps 변경 시마다 "이전 effect의 cleanup → 새 effect" 순서로 돈다. 구독 해제를 안 하면 deps가 바뀔 때마다 구독이 쌓인다.
- **effect 안에서 setState + 그 state가 deps에** → 무한 루프. 객체/배열을 deps에 넣을 때도 매 렌더 새 참조면 같은 문제.
- **파생 값을 state로 두지 말 것.** `fullName`을 `useState`로 만들고 effect로 동기화하는 패턴은 렌더 두 번 + 어긋남의 근원. 렌더 중 계산(`const fullName = first + last`)이면 충분하다. SwiftUI에서 computed property로 풀 것을 `@State` 두 개로 만드는 실수와 동일.
- **훅을 조건문 안에서 호출** → 호출 순서 슬롯이 어긋나 런타임 에러. 조기 return보다 훅이 먼저 와야 한다.
- **`useRef.current`를 렌더링에 사용** → 값이 바뀌어도 화면이 안 바뀐다. 화면에 보일 값은 state로.
