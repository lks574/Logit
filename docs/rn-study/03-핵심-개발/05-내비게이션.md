# 내비게이션

> RN에는 내장 내비게이션이 **없다** — 사실상 표준인 [[React Navigation]]과 그 위의 파일 기반 래퍼 [[Expo Router]] 중에서 고르며, 둘 다 native-stack을 쓰면 진짜 `UINavigationController`/`Fragment` 전환이 돈다.

## iOS/AOS 대응 개념

| RN | iOS | Android |
|---|---|---|
| (native-)stack navigator | `UINavigationController` / SwiftUI `NavigationStack` | Jetpack Navigation / Compose `NavHost` back stack |
| tab navigator | `UITabBarController` / `TabView` | `BottomNavigationView` / `NavigationBar` |
| `presentation: 'modal'` | `present(_:animated:)` sheet | `DialogFragment` / Compose dialog destination |
| 라우트 파라미터 | segue sender / `NavigationLink(value:)` | Safe Args / `navArguments` |
| `useFocusEffect` | `viewWillAppear`/`viewWillDisappear` | `onResume`류 / `LifecycleObserver` |
| [[Expo Router]] `app/` 디렉토리 | — (대응물 없음, 웹 프레임워크 유래) | — |
| 딥링크 | Universal Links / URL Scheme | App Links / intent-filter |

## 왜 이렇게 설계됐나

React 철학에서 내비게이션도 "그냥 상태"다 — 어떤 화면 스택이 쌓여 있는지도 [[State]]고, UI는 그 함수다. 그래서 RN 코어는 내비게이션을 포함하지 않았고 커뮤니티에 맡겼다. 살아남은 승자가 **React Navigation**: 내비게이션 상태를 JS에서 선언적으로 관리하되, `native-stack` 등장 이후로는 **실제 화면 전환은 네이티브에 위임**한다. iOS에서는 진짜 `UINavigationController`가 push/pop하고(스와이프 백, 네이티브 헤더, 전환 애니메이션 전부 OS 것), Android에서는 `Fragment` 기반으로 돈다. "JS로 흉내 낸 전환"(구형 JS stack)의 이질감 문제를 이렇게 해소했다.

**Expo Router**는 그 위에 웹 프레임워크(Next.js류)의 발명을 얹은 것이다: **파일 시스템이 곧 라우트 정의**. `app/settings.tsx`를 만들면 `/settings` 라우트가 생기고, 모든 화면이 URL을 가지므로 **딥링크가 설정 없이 자동**으로 열린다. 내부 구현은 React Navigation이므로 대립 관계가 아니라 "선언 방식이 다른 상위 레이어"다. Expo 신규 프로젝트의 기본 템플릿이기도 하다.

## 동작 원리

### React Navigation — 컴포넌트로 선언

내비게이터를 JSX로 조립한다. 참고로 **이 레포(Logit)는 이 방식**이다 — `src/navigation/RootNavigator.tsx`에서 native-stack + bottom-tabs를 조합하고, 모달 화면들을 `Stack.Group screenOptions={{ presentation: 'modal' }}`로 묶는다.

```tsx
// Logit의 실제 구조 요약 (src/navigation/RootNavigator.tsx)
const Tab = createBottomTabNavigator<TabParamList>();
const Stack = createNativeStackNavigator<RootStackParamList>();

function MainTabs() {
  return (
    <Tab.Navigator>
      <Tab.Screen name="Home" component={HomeScreen} />
      <Tab.Screen name="Calendar" component={CalendarScreen} />
      {/* ... */}
    </Tab.Navigator>
  );
}

export function RootNavigator() {
  return (
    <NavigationContainer>
      <Stack.Navigator>
        <Stack.Screen name="MainTabs" component={MainTabs} />
        <Stack.Group screenOptions={{ presentation: 'modal' }}>
          <Stack.Screen name="RecordForm" component={RecordFormScreen} />
        </Stack.Group>
      </Stack.Navigator>
    </NavigationContainer>
  );
}
```

- **navigator 종류**: `native-stack`(네이티브 push/pop — 기본 선택지), `stack`(JS 구현 — 전환 커스텀이 아주 특수할 때만), `bottom-tabs`, `drawer`. 모달은 별도 navigator가 아니라 stack 화면의 `presentation: 'modal'` 옵션.
- **이동**: `navigation.navigate('Detail', { id: '42' })` / `navigation.goBack()`. 파라미터는 대상 화면에서 `route.params.id`로 수신.
- **타입 안전**: `RootStackParamList` 타입(화면명 → 파라미터 형태 매핑)을 navigator 제네릭에 넘기면 잘못된 화면명·파라미터가 컴파일 에러가 된다. Safe Args에 해당하는 것을 TypeScript 타입으로 얻는 셈. Logit도 `src/navigation/types.ts`에 이 타입을 둔다.

### Expo Router — 파일로 선언

`app/` 디렉토리의 파일 구조가 곧 라우트다. (Expo SDK 57 기준, 세부는 https://docs.expo.dev/versions/v57.0.0/ 의 Router 문서 확인.)

```
app/
  _layout.tsx        ← 루트 레이아웃 (Stack 선언; NavigationContainer는 내장이라 불필요)
  index.tsx          ← "/"        (홈)
  settings.tsx       ← "/settings"
  log/
    [id].tsx         ← "/log/42"  (동적 세그먼트 → useLocalSearchParams로 수신)
  (tabs)/            ← URL에 안 나타나는 그룹 (탭 묶음 관용구)
    _layout.tsx      ← Tabs 선언
    index.tsx
    stats.tsx
```

- `_layout.tsx`가 React Navigation의 navigator 선언에 해당한다: `<Stack>` / `<Tabs>`를 반환하면 그 폴더의 파일들이 화면으로 등록된다.
- 이동은 URL로: `router.push('/log/42')` 또는 `<Link href="/log/42">`. 웹 개발과 동일한 감각.
- **딥링크 자동**: 모든 화면에 URL이 있으므로 scheme/Universal Link가 들어오면 해당 경로가 그냥 열린다. React Navigation에서 수동으로 작성하던 `linking` 설정이 사라진다.
- 타입 안전 라우트: typed routes 기능으로 `href` 문자열도 컴파일 타임 검증 가능(Expo 문서에서 활성화 방법 확인).

### 어느 쪽을 쓰나

| | React Navigation 직접 | Expo Router |
|---|---|---|
| 라우트 정의 | JSX로 명시적 | 파일 구조로 암묵적 |
| 딥링크 | `linking` 수동 구성 | 자동 |
| 웹 지원 | 별도 작업 | URL 기반이라 자연스러움 |
| 자유도 | 높음 (구조를 코드로 완전 제어) | 컨벤션 우선 |
| 적합 | 기존 코드베이스, 특수 구조 | 신규 Expo 프로젝트 (현재 기본 권장) |

신규 Expo 프로젝트라면 Expo Router가 기본값이다. Logit처럼 React Navigation을 직접 쓰는 코드베이스도 여전히 완전히 유효하다 — 아래 개념(native-stack, focus lifecycle, 파라미터)은 어느 쪽이든 동일하게 적용된다.

### 화면 lifecycle — useFocusEffect

네이티브의 상식과 달리, **스택 뒤로 가려진 화면은 언마운트되지 않는다**(상태 보존을 위해 마운트 유지). 즉 `useEffect(fn, [])`는 `viewDidLoad`에 가깝고, `viewWillAppear`류가 필요하면 focus 이벤트를 써야 한다.

```tsx
import { useFocusEffect } from '@react-navigation/native'; // Expo Router도 동일 훅 제공
import { useCallback } from 'react';

useFocusEffect(
  useCallback(() => {
    refetch();                    // 화면이 포커스될 때마다 (viewWillAppear)
    return () => stopPolling();   // 포커스를 잃을 때 (viewWillDisappear)
  }, [refetch]),
);
```

`useCallback`으로 감싸는 것이 규약이다(매 렌더 재실행 방지). 단발 확인에는 `useIsFocused()` 훅도 있다. 탭 전환에서도 동일하게 동작한다 — 탭의 다른 화면들 역시 마운트된 채 유지된다.

## 코드 예시

Expo SDK 57 / Expo Router 기준. 탭 + 스택 + 동적 라우트 + 모달 최소 구성.

```tsx
// app/_layout.tsx — 루트: 스택 + 모달
import { Stack } from 'expo-router';

export default function RootLayout() {
  return (
    <Stack>
      <Stack.Screen name="(tabs)" options={{ headerShown: false }} />
      <Stack.Screen name="add" options={{ presentation: 'modal', title: '기록 추가' }} />
    </Stack>
  );
}
```

```tsx
// app/(tabs)/_layout.tsx — 탭 정의
import { Tabs } from 'expo-router';

export default function TabLayout() {
  return (
    <Tabs>
      <Tabs.Screen name="index" options={{ title: '홈' }} />
      <Tabs.Screen name="stats" options={{ title: '통계' }} />
    </Tabs>
  );
}
```

```tsx
// app/(tabs)/index.tsx — 홈: 이동 3종 (Link, router.push, 모달)
import { View, Text, Pressable, StyleSheet } from 'react-native';
import { Link, useRouter } from 'expo-router';

export default function HomeScreen() {
  const router = useRouter();
  return (
    <View style={styles.center}>
      <Link href="/log/42"><Text>42번 기록 보기 (Link)</Text></Link>
      <Pressable onPress={() => router.push({ pathname: '/log/[id]', params: { id: '7' } })}>
        <Text>7번 기록 보기 (router.push)</Text>
      </Pressable>
      <Pressable onPress={() => router.push('/add')}>
        <Text>기록 추가 (모달)</Text>
      </Pressable>
    </View>
  );
}

const styles = StyleSheet.create({
  center: { flex: 1, alignItems: 'center', justifyContent: 'center', gap: 16 },
});
```

```tsx
// app/log/[id].tsx — 파라미터 수신 + focus lifecycle
import { useCallback } from 'react';
import { View, Text } from 'react-native';
import { useLocalSearchParams, useFocusEffect, Stack } from 'expo-router';

export default function LogDetailScreen() {
  const { id } = useLocalSearchParams<{ id: string }>(); // URL이라 항상 string

  useFocusEffect(
    useCallback(() => {
      console.log(`log ${id} focused`);        // viewWillAppear
      return () => console.log('blurred');     // viewWillDisappear
    }, [id]),
  );

  return (
    <View style={{ flex: 1, padding: 16 }}>
      <Stack.Screen options={{ title: `기록 #${id}` }} />
      <Text>Log id: {id}</Text>
    </View>
  );
}
```

이 구조에서 `myapp://log/42`(scheme은 `app.json`의 `scheme` 필드) 딥링크가 별도 설정 없이 상세 화면을 연다. Universal Links/App Links까지 붙이려면 도메인 검증 파일이 필요 — Expo 문서의 deep linking 가이드 확인.

## 함정 (Pitfalls)

- **"뒤로 가려진 화면이 죽는다"는 가정**: 스택 아래 화면은 마운트 유지 → `useEffect([], )`는 다시 안 불린다. "돌아올 때마다 갱신"은 `useFocusEffect`로. `viewWillAppear`에 하던 일을 `useEffect`에 넣는 것이 네이티브 출신의 1순위 버그.
- **`useFocusEffect`에 `useCallback` 누락**: 매 렌더마다 effect가 재실행되어 사실상 매 렌더 콜백이 된다. 반드시 감싼다.
- **파라미터로 객체/함수 넘기기**: 파라미터는 "URL에 실을 수 있는 작은 식별자"여야 한다. 큰 객체를 넘기면 딥링크·상태 복원이 깨지고, Expo Router에서는 직렬화 문제. id만 넘기고 대상 화면이 스토어/서버에서 조회하는 패턴으로.
- **Expo Router 파라미터는 string**: URL 유래라 `id`가 숫자여도 `string`(또는 `string[]`)으로 온다. `Number(id)` 변환을 잊으면 `===` 비교가 조용히 실패한다.
- **`navigate` vs `push`**: React Navigation의 `navigate`는 스택에 같은 화면이 있으면 재사용(파라미터만 갱신)하고, `push`는 무조건 쌓는다. `pushViewController`만 알던 감각으로 `navigate`를 쓰면 "왜 새 화면이 안 쌓이지"가 된다.
- **native-stack 헤더를 두고 커스텀 헤더 뷰를 또 그리기**: native-stack의 헤더는 진짜 네이티브 내비게이션 바다. 완전 커스텀 UI가 필요하면 `headerShown: false`로 끄고 직접 그린다(Logit이 이 방식). 둘을 겹치면 이중 헤더.
- **안드로이드 하드웨어 뒤로가기**: iOS 출신이 잊기 쉬운 존재. 기본 동작(스택 pop)은 자동이지만, 폼 이탈 확인 등 개입하려면 `usePreventRemove` 또는 `beforeRemove` 이벤트 — 버전별 API는 React Navigation 문서 확인.
- **모달 안에서 또 push하려면 구조가 필요**: 모달 위에 스택을 쌓으려면 모달 자체가 자기 스택(navigator)을 가져야 한다. `UINavigationController`를 present하는 것과 같은 구조를 라우트 트리로 표현하는 것.
- **웹 감각으로 `router.replace` 남용**: 히스토리를 지우므로 뒤로가기가 기대와 달라진다. 온보딩 완료→홈 진입처럼 "돌아가면 안 되는" 전환에만.
