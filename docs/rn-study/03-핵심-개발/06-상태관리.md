# 06. 상태관리 — 계층적 접근

> [!abstract] 한 줄 요약
> "어떤 상태관리 라이브러리를 쓰나요?"는 잘못된 질문이다. 상태의 **종류**(로컬/전역/서버)와 **변경 빈도**에 따라 가장 가벼운 도구부터 단계적으로 올라가는 것이 현대 React/RN의 표준이다.

## 🔁 iOS/AOS 대응 개념

| RN | iOS (SwiftUI) | Android (Compose) |
|---|---|---|
| `useState` (로컬 [[State]]) | `@State` | `remember { mutableStateOf() }` |
| props drilling | 이니셜라이저로 값 전달 | Composable 파라미터 전달 |
| Context | `@EnvironmentObject` / `.environment()` | `CompositionLocal` |
| Zustand 스토어 | `ObservableObject` / `@Observable` 싱글턴 | `ViewModel` + `StateFlow` |
| Zustand 선택적 구독 (selector) | `@Observable`의 프로퍼티 단위 추적 | `StateFlow.map().distinctUntilChanged()` |
| TanStack Query (서버 상태) | 직접 구현하거나 별도 캐싱 레이어 | Repository + Room/캐시 + Flow |
| Redux | (직접 대응 없음 — TCA가 유사한 철학) | (직접 대응 없음 — MVI 계열이 유사) |

핵심 차이: 네이티브에서는 `ViewModel`/`ObservableObject` 하나가 "화면 상태 + 서버 데이터 + 캐싱"을 다 떠안는 경우가 많다. React 생태계는 이걸 **클라이언트 상태와 서버 상태로 쪼개는 것**을 표준으로 정착시켰다.

## 🧭 왜 이렇게 설계됐나

React의 [[State]] 모델은 "상태가 바뀌면 해당 컴포넌트부터 아래로 [[Re-render]]"라는 단순한 규칙 하나로 돌아간다. 이 규칙이 단순한 대신, **상태를 어디에 둘 것인가**가 성능과 복잡도를 결정한다.

- 상태를 너무 위에 두면 → 넓은 서브트리가 불필요하게 리렌더된다.
- 상태를 너무 아래에 두면 → 형제 컴포넌트와 공유할 수 없다.

그래서 커뮤니티가 오랜 시행착오 끝에 도달한 결론이 계층적 접근이다: **가능한 가장 낮은 곳에, 가능한 가장 가벼운 도구로.**

또 하나의 역사적 전환점은 "서버 상태" 개념의 분리다. 2015~2019년경에는 Redux 스토어에 API 응답을 넣고 로딩/에러 플래그를 손으로 관리하는 것이 표준이었다. 하지만 서버 데이터는 클라이언트 상태와 성격이 다르다 — **내가 소유하지 않고, 언제든 낡아질(stale) 수 있으며, 여러 화면이 같은 데이터를 요청한다.** 이 문제(캐싱, 재검증, 중복 제거)를 전담하는 TanStack Query가 등장하면서 Redux의 존재 이유 상당 부분이 사라졌고, 남은 클라이언트 전역 상태는 Zustand 같은 초경량 라이브러리로 충분해졌다.

## ⚙️ 동작 원리 — 5단계 사다리

### 1단계: 로컬 `useState` — 대부분은 여기서 끝난다

폼 입력값, 토글, 모달 열림 여부처럼 한 컴포넌트(또는 한 화면)만 쓰는 상태는 그냥 `useState`. SwiftUI `@State`와 같은 위치다. "일단 전역 스토어부터 만들자"는 네이티브 개발자가 흔히 가져오는 습관인데, React에서는 안티패턴이다.

### 2단계: props drilling — 몇 단계는 그냥 둬도 된다

부모의 상태를 자식 2~3단계 아래로 props로 내려보내는 것은 문제가 아니다. 명시적이고, 타입이 따라오고, 데이터 흐름이 눈에 보인다. Compose에서 파라미터로 값을 내려보내는 것과 똑같다. drilling이 5단계를 넘어가고 중간 컴포넌트들이 그 값을 쓰지도 않으면서 통과만 시키기 시작할 때, 그때 다음 단계를 고려한다.

### 3단계: Context — 저빈도 전역 (테마, 인증, 로케일)

```
<ThemeProvider>        ← 값 제공
  <App>
    ...깊은 어딘가...
      <Button>         ← useContext(Theme)로 바로 꺼냄
```

Context는 "provider 아래 어디서든 값을 꺼낼 수 있는 통로"다. `@EnvironmentObject` / `CompositionLocal`과 같은 발상.

**비용을 정확히 알아야 한다**: Context 값이 바뀌면 **그 Context를 구독하는 모든 컴포넌트가 리렌더**된다. 값의 일부만 바뀌어도, 그 일부를 안 쓰는 소비자도 전부. 선택적 구독이 없다. 그래서 Context는 **거의 안 바뀌는 값**(테마, 로그인 세션, 언어 설정)에 적합하고, 초당 수십 번 바뀌는 값에는 부적합하다.

참고로 이 레포(Logit)는 오프라인 우선의 소규모 앱이라 Context + AsyncStorage 조합(`src/store/StoreContext.tsx`)으로 전체 스토어를 구성했다 — 데이터 변경 빈도가 낮으면 이 정도로도 실전에서 충분하다는 실례.

### 4단계: Zustand — 고빈도 전역 클라이언트 상태

Context의 "소비자 전체 리렌더" 문제를 푸는 것이 Zustand다. 스토어는 React 바깥에 사는 단순한 객체이고, 컴포넌트는 **selector로 필요한 조각만 구독**한다. 그 조각이 바뀔 때만 리렌더된다. `StateFlow`를 `map` + `distinctUntilChanged`로 잘라 collect하는 것과 같은 효과.

- Provider가 필요 없다 (전역 싱글턴).
- 보일러플레이트가 거의 없다 (액션 = 그냥 함수).
- React 바깥(이벤트 핸들러, 네이티브 모듈 콜백)에서도 `useStore.getState()`로 읽고 쓸 수 있다.

### 5단계: TanStack Query — 서버 상태는 별개의 문제

서버에서 온 데이터는 상태관리가 아니라 **캐시 관리** 문제다:

- **캐싱**: 같은 key의 데이터는 한 번만 저장, 화면 간 공유.
- **재검증(revalidation)**: stale 판정 시간(`staleTime`)이 지나면 포커스/재마운트 시 백그라운드에서 다시 fetch.
- **중복 제거(deduplication)**: 열 개 컴포넌트가 동시에 같은 query를 써도 네트워크 요청은 1번.
- 로딩/에러/재시도/페이지네이션이 전부 내장.

이걸 Zustand나 Redux로 직접 짜면 결국 TanStack Query를 재발명하게 된다. **"서버 상태는 TanStack Query, 클라이언트 상태는 Zustand(또는 그 이하)"** 가 현재의 사실상 표준 조합이다.

### 결정 요약

```
이 상태를 다른 화면/컴포넌트가 쓰는가?
 ├─ No → useState (1단계)
 └─ Yes → 서버에서 온 데이터인가?
     ├─ Yes → TanStack Query (5단계)
     └─ No → 얼마나 자주 바뀌는가?
         ├─ 거의 안 바뀜(테마·인증) → Context (3단계)
         └─ 자주 바뀜 → Zustand (4단계)
   ※ 공유 범위가 부모-자식 2~3단계뿐이면 그냥 props (2단계)
```

영속화가 필요하면: Zustand는 `persist` 미들웨어 + AsyncStorage 어댑터로 스토어를 디스크에 저장할 수 있다(설정 상세는 공식 문서 확인). `UserDefaults`/`DataStore`에 수동 직렬화하던 코드가 미들웨어 한 줄로 줄어든다.

### Redux는 어디에?

Redux(현재는 Redux Toolkit)는 틀렸다기보다 **맥락이 좁아졌다**. 단일 스토어 + 액션 로그 + 시간여행 디버깅 + 엄격한 규약이 필요한 대규모 팀/앱에서는 여전히 유효하다. 하지만 신규 프로젝트의 기본값은 아니다. TCA(iOS)를 써본 적 있다면 그 트레이드오프(강한 규약 ↔ 보일러플레이트)를 그대로 떠올리면 된다.

## 💻 코드 예시

기준: RN 0.76+ / Expo SDK 57 / zustand 4+ / TypeScript.

### Zustand — 선택적 구독

```typescript
// store/useWorkoutStore.ts
import { create } from 'zustand';

type WorkoutState = {
  reps: number;
  restSec: number;
  addRep: () => void;
  resetRest: () => void;
};

export const useWorkoutStore = create<WorkoutState>()((set) => ({
  reps: 0,
  restSec: 60,
  addRep: () => set((s) => ({ reps: s.reps + 1 })),
  resetRest: () => set({ restSec: 60 }),
}));
```

```typescript
// components/RepCounter.tsx
import React from 'react';
import { Button, Text, View } from 'react-native';
import { useWorkoutStore } from '../store/useWorkoutStore';

export function RepCounter() {
  // selector: reps만 구독 → restSec가 바뀌어도 이 컴포넌트는 리렌더되지 않음
  const reps = useWorkoutStore((s) => s.reps);
  const addRep = useWorkoutStore((s) => s.addRep);

  return (
    <View>
      <Text>{reps} reps</Text>
      <Button title="+1" onPress={addRep} />
    </View>
  );
}
```

### TanStack Query — 서버 상태

```typescript
// hooks/useProfile.ts
import { useQuery } from '@tanstack/react-query';

type Profile = { id: string; name: string };

async function fetchProfile(): Promise<Profile> {
  const res = await fetch('https://api.example.com/me');
  if (!res.ok) throw new Error('failed');
  return res.json();
}

export function useProfile() {
  return useQuery({
    queryKey: ['profile'],       // 이 key로 캐싱·중복제거
    queryFn: fetchProfile,
    staleTime: 60_000,           // 1분간은 fresh → refetch 안 함
  });
}
```

```typescript
// components/ProfileCard.tsx — 어느 화면에서 몇 번을 호출해도 요청은 공유된다
import React from 'react';
import { ActivityIndicator, Text } from 'react-native';
import { useProfile } from '../hooks/useProfile';

export function ProfileCard() {
  const { data, isPending, error } = useProfile();
  if (isPending) return <ActivityIndicator />;
  if (error) return <Text>불러오기 실패</Text>;
  return <Text>{data.name}</Text>;
}
```

(앱 루트에 `QueryClientProvider` 세팅 필요 — 공식 문서 확인.)

## ⚠️ 함정 (Pitfalls)

1. **"일단 전역 스토어부터"** — ViewModel 습관. 로컬 `useState`로 시작하고, 공유가 필요해질 때 끌어올려라. 반대 방향(전역 → 로컬)의 리팩터링이 훨씬 고통스럽다.
2. **Context에 고빈도 값 넣기** — 예: 스크롤 위치, 타이머 틱을 Context로 뿌리면 소비자 전원이 초당 수십 번 [[Re-render]]. Context는 선택적 구독이 없다는 사실을 잊지 말 것.
3. **Context value 객체를 매 렌더마다 새로 생성** — `<Ctx.Provider value={{ user, logout }}>`처럼 인라인 객체를 넘기면 provider가 리렌더될 때마다 참조가 바뀌어 소비자 전원이 리렌더된다. `useMemo`로 감쌀 것 ([[Memoization]]).
4. **Zustand에서 selector 없이 스토어 통째로 구독** — `const store = useWorkoutStore()`는 스토어의 아무 필드나 바뀌어도 리렌더된다. Zustand를 쓰는 이유가 사라진다. 항상 `useWorkoutStore((s) => s.필드)`.
5. **Zustand selector가 매번 새 객체를 반환** — `useStore((s) => ({ a: s.a, b: s.b }))`는 매번 새 참조 → 매번 리렌더. 필드별로 나눠 구독하거나 `useShallow`(zustand/react/shallow) 사용.
6. **서버 데이터를 Zustand/Redux에 복사해 넣기** — "fetch 해서 스토어에 저장"은 캐시 무효화 지옥의 입구다. 서버 상태는 TanStack Query에 맡기고, 스토어에는 순수 클라이언트 상태만.
7. **이벤트 핸들러 안의 낡은 상태** — 클로저가 캡처한 옛 값을 읽는 문제. [[Stale Closure]] 노트 참고. Zustand는 `useStore.getState()`로 항상 최신 값을 읽는 탈출구가 있다.
