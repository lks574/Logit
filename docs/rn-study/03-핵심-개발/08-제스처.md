# 08. 제스처 — react-native-gesture-handler

> [!abstract] 한 줄 요약
> RN 내장 터치 시스템은 이벤트가 JS 스레드를 거치므로 제스처 인식이 끊길 수 있다. react-native-gesture-handler(RNGH)는 인식을 네이티브에 맡기고, [[Reanimated]]와 결합해 추적까지 UI 스레드에서 끝내는 표준 조합을 이룬다.

## 🔁 iOS/AOS 대응 개념

| RN (RNGH) | iOS | Android |
|---|---|---|
| `Gesture.Tap()` | `UITapGestureRecognizer` | `GestureDetector.onSingleTapUp` / Compose `detectTapGestures` |
| `Gesture.Pan()` | `UIPanGestureRecognizer` | Compose `detectDragGestures` |
| `Gesture.Pinch()` | `UIPinchGestureRecognizer` | `ScaleGestureDetector` |
| `Gesture.LongPress()` | `UILongPressGestureRecognizer` | `onLongPress` / `detectTapGestures(onLongPress=)` |
| `GestureDetector` (뷰에 제스처 부착) | `view.addGestureRecognizer(_:)` | `Modifier.pointerInput` |
| `simultaneousWithExternalGesture` | `gestureRecognizer(_:shouldRecognizeSimultaneouslyWith:)` | 중첩 스크롤/`requestDisallowInterceptTouchEvent` 조정 |
| `Gesture.Race / Exclusive / Simultaneous` | recognizer 간 `require(toFail:)` 관계 설정 | 제스처 우선순위 수동 조정 |
| 제스처 상태 (BEGAN → ACTIVE → END) | `UIGestureRecognizer.State` (began/changed/ended) | `MotionEvent` ACTION_DOWN/MOVE/UP + 인식 로직 |

iOS 개발자라면 그대로 이해하면 된다: **RNGH는 사실상 `UIGestureRecognizer` 시스템을 크로스 플랫폼 API로 노출한 것**이다 (iOS에서는 실제로 그 위에 구현되어 있고, Android에서는 동등한 인식 상태 머신을 구현).

## 🧭 왜 이렇게 설계됐나

### RN 내장 터치 시스템의 구조적 한계

RN의 기본 터치 처리(`Pressable`, `PanResponder`)는 이렇게 동작한다:

1. 네이티브가 터치 이벤트를 받는다
2. 이벤트를 **JS 스레드로 전달**
3. JS의 responder 시스템이 "이 제스처를 누가 가져갈지" 판정
4. 판정 결과에 따라 뷰가 반응

문제는 2~3단계다. JS 스레드가 바쁘면(리스트 렌더, [[Re-render]] 폭풍) 이벤트 처리와 제스처 판정이 밀린다. 결과: 드래그가 뚝뚝 끊기고, 스크롤 중 터치 판정이 어긋나고, 터치 반응이 한 박자 늦는다. 네이티브에서는 있을 수 없는 일이다 — `UIGestureRecognizer`와 `MotionEvent` 파이프라인은 항상 메인/입력 파이프라인에서 돈다.

### RNGH의 해법: 인식을 네이티브로 되돌린다

RNGH는 제스처 **인식(recognition)** 을 네이티브 레이어에서 수행한다. "이건 탭이다 / 팬이 시작됐다 / 이 스크롤뷰가 제스처를 가져간다" 같은 판정이 JS 스레드 사정과 무관하게 이뤄지고, 네이티브 스크롤뷰 등 플랫폼 제스처와의 협상도 네이티브 규칙으로 처리된다.

그리고 인식 후의 **추적(tracking)** — 손가락을 따라 뷰를 움직이는 것 — 은 [[Reanimated]]가 맡는다. RNGH v2의 제스처 콜백(`onUpdate` 등)은 Reanimated가 설치되어 있으면 자동으로 [[Worklet]]으로 실행되어, 터치 이벤트 수신 → shared value 갱신 → 스타일 반영이 **전부 UI 스레드에서** 끝난다. 이 "RNGH + Reanimated" 결합이 RN 인터랙션의 사실상 표준 콤보다.

## ⚙️ 동작 원리

### v2 Gesture API 기본형

```typescript
const tap = Gesture.Tap().onEnd(() => { /* ... */ });

<GestureDetector gesture={tap}>
  <Animated.View />
</GestureDetector>
```

- `Gesture.Xxx()`로 제스처 객체를 만들고 빌더 체이닝으로 설정 (`numberOfTaps`, `minDistance` 등).
- `GestureDetector`로 뷰에 부착 — `addGestureRecognizer`에 해당.

콜백 생명주기는 `UIGestureRecognizer.State` 전이와 1:1로 대응된다:

| RNGH 콜백 | 시점 | UIGestureRecognizer |
|---|---|---|
| `onBegin` | 터치가 닿음 (아직 인식 전) | `.possible` 진입 |
| `onStart` | 인식 성공, 제스처 활성화 | `.began` |
| `onUpdate` | 활성 상태에서 매 이동 | `.changed` |
| `onEnd` | 정상 종료 (손 뗌) | `.ended` |
| `onFinalize` | 성공/실패/취소 무관 마무리 | `.ended`/`.cancelled`/`.failed` 공통 후처리 |

"탭인 줄 알았는데 움직여서 pan에게 넘어감" 같은 인식 실패 시 `onStart` 없이 `onFinalize`만 불린다 — cleanup 로직은 `onEnd`가 아니라 `onFinalize`에 두는 이유.

### 제스처 조합 — recognizer 관계 설정에 해당

| 조합자 | 의미 | iOS 대응 |
|---|---|---|
| `Gesture.Race(a, b)` | 먼저 인식한 쪽이 독점 (기본 동작) | 기본 배타 동작 |
| `Gesture.Simultaneous(a, b)` | 동시 인식 허용 (핀치+회전) | `shouldRecognizeSimultaneouslyWith` = true |
| `Gesture.Exclusive(a, b)` | a 우선, a가 실패해야 b (더블탭 vs 싱글탭) | `require(toFail:)` |
| `.simultaneousWithExternalGesture(ref)` | 다른 `GestureDetector`/스크롤뷰의 제스처와 동시 인식 | recognizer 간 delegate 협상 |
| `.blocksExternalGesture(ref)` | 이 제스처가 활성인 동안 외부 제스처 차단 | 외부 recognizer에 `require(toFail:)` 강제 |

ScrollView/[[FlatList]] 안에 스와이프 가능한 행을 넣는 것 같은 "중첩 제스처 협상"이 네이티브에서 그랬듯 여기서도 난이도의 핵심이고, 위 조합자들이 그 도구다.

## 💻 코드 예시

기준: RN 0.76+ / Expo SDK 57 / react-native-gesture-handler 2+ / react-native-reanimated 3+
(`npx expo install react-native-gesture-handler react-native-reanimated`)

드래그 가능한 박스 — pan + shared value 표준 콤보:

```typescript
// DraggableBox.tsx
import React from 'react';
import { StyleSheet } from 'react-native';
import {
  Gesture,
  GestureDetector,
  GestureHandlerRootView,
} from 'react-native-gesture-handler';
import Animated, {
  useAnimatedStyle,
  useSharedValue,
  withSpring,
} from 'react-native-reanimated';

export default function DraggableBox() {
  // 현재 위치 + 드래그 시작 시점의 위치(offset 패턴)
  const x = useSharedValue(0);
  const y = useSharedValue(0);
  const startX = useSharedValue(0);
  const startY = useSharedValue(0);

  // 콜백은 자동으로 worklet으로 실행됨 (Reanimated 설치 시)
  const pan = Gesture.Pan()
    .onStart(() => {
      startX.value = x.value;
      startY.value = y.value;
    })
    .onUpdate((e) => {
      // UI 스레드에서 매 프레임 실행 — JS 스레드가 바빠도 안 끊김
      x.value = startX.value + e.translationX;
      y.value = startY.value + e.translationY;
    })
    .onEnd(() => {
      // 놓으면 원점으로 스프링 복귀
      x.value = withSpring(0);
      y.value = withSpring(0);
    });

  const style = useAnimatedStyle(() => ({
    transform: [{ translateX: x.value }, { translateY: y.value }],
  }));

  return (
    <GestureHandlerRootView style={styles.root}>
      <GestureDetector gesture={pan}>
        <Animated.View style={[styles.box, style]} />
      </GestureDetector>
    </GestureHandlerRootView>
  );
}

const styles = StyleSheet.create({
  root: { flex: 1, alignItems: 'center', justifyContent: 'center' },
  box: { width: 100, height: 100, borderRadius: 20, backgroundColor: '#f76f4f' },
});
```

포인트:

- **offset 패턴**: `onUpdate`의 `translationX`는 "이번 제스처 시작점 기준 누적 이동량"이다 (`UIPanGestureRecognizer.translation`과 동일). 그래서 시작 시점 위치를 `startX/startY`에 저장해 두고 더한다.
- `onEnd`의 이벤트에는 `velocityX/velocityY`도 들어오므로, `withSpring(0, { velocity: e.velocityX })`처럼 넘기면 손을 뗀 속도가 스프링에 이어져 자연스럽다 — `UIPanGestureRecognizer.velocity(in:)`을 `UIViewPropertyAnimator`에 넘기던 그 패턴.

### 조합 예시 — 더블탭 리셋 추가

```typescript
// DraggableBox의 pan 아래에 이어서:
const doubleTap = Gesture.Tap()
  .numberOfTaps(2)
  .onEnd(() => {
    x.value = withSpring(0);
    y.value = withSpring(0);
  });

// 먼저 성립하는 쪽이 이긴다: 더블탭이 성립하면 리셋, 움직이면 pan
const composed = Gesture.Race(doubleTap, pan);

// <GestureDetector gesture={composed}> 로 교체
```

`GestureDetector`는 뷰당 하나만 두고, 여러 제스처는 조합자로 묶어 하나의 gesture로 만들어 넘기는 것이 관례다. recognizer를 여러 개 `addGestureRecognizer` 하고 delegate로 관계를 조율하던 것보다 선언적이다.

## ⚠️ 함정 (Pitfalls)

1. **`GestureHandlerRootView` 누락** — 최상위(보통 앱 루트)를 `GestureHandlerRootView`로 감싸지 않으면 제스처가 **조용히 무시**된다(특히 Android). 에러도 없이 "탭이 안 먹는" 증상이라 처음엔 반드시 이걸 의심할 것. [[Expo Router]]/[[React Navigation]] 최신 버전은 루트에서 처리해 주는 경우가 있으나, 직접 구성한 앱이라면 명시적으로 감싸는 게 안전하다 (공식 문서 확인).
2. **제스처 콜백 = worklet임을 잊음** — `onUpdate` 안에서 `setState`나 내비게이션을 직접 호출하면 에러. JS 사이드 이펙트는 `runOnJS(fn)(args)`로. 반대로 Reanimated 없이 RNGH만 쓰면 콜백은 JS 스레드에서 돌며, 그 경우 `.runOnJS(true)` 설정과 관련한 동작 차이는 공식 문서 확인.
3. **콜백에서 일반 지역 변수로 위치 누적** — `let start = 0`을 컴포넌트 스코프에 두고 worklet에서 갱신해도 복사 캡처라 동작하지 않는다. 스레드를 넘나드는 값은 전부 shared value로 (offset 패턴).
4. **RN 내장 `Pressable`/`PanResponder`와 혼용** — 두 터치 시스템이 서로를 모르기 때문에 판정이 충돌한다. 인터랙션이 조금이라도 복잡해지면 RNGH로 통일하는 편이 낫다. 스크롤뷰도 RNGH가 재수출하는 `ScrollView`를 쓰면 제스처 협상이 매끄럽다.
5. **스크롤 안의 pan 제스처** — 세로 스크롤뷰 안에서 가로 스와이프 행을 만들 때 `activeOffsetX`/`failOffsetY`로 활성화 방향을 제한하지 않으면 스크롤을 가로챈다. `UIScrollView`의 pan과 recognizer 협상을 하던 그 문제와 동일하며, 도구는 `activeOffsetX([-10, 10])` + `simultaneousWithExternalGesture`.
6. **화면 가장자리에서 시작하는 pan** — iOS 엣지 스와이프 백, Android 백 제스처와 영역이 겹친다. 시스템 제스처가 우선권을 가지므로 가장자리에서 시작하는 커스텀 드래그는 피하거나 여백을 두고 설계할 것.
7. **한쪽 플랫폼에서만 확인** — 제스처 협상 기본값(특히 네이티브 스크롤과의 관계)은 iOS/Android가 미묘하게 다르다. 드래그·스와이프 UI는 반드시 양쪽에서 확인.

한 줄 정리: **인식은 네이티브(RNGH), 추적은 UI 스레드 worklet([[Reanimated]]), JS 스레드는 결과 통보만 받는다.**
