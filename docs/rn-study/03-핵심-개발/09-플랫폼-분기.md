# 09. 플랫폼 분기

> [!abstract] 한 줄 요약
> "한 코드베이스"는 "동일한 코드"가 아니다. RN은 런타임 분기(`Platform`)와 빌드 타임 분기(`.ios.tsx`/`.android.tsx`)를 제공하며, 진짜 어려운 문제는 분기 문법이 아니라 **어디까지 플랫폼 관습을 존중할지** 결정하는 것이다.

## 🔁 iOS/AOS 대응 개념

| RN | iOS | Android |
|---|---|---|
| `Platform.OS === 'ios'` | (해당 없음 — 항상 iOS) | (해당 없음) — KMP의 `expect`/`actual`이 발상 면에서 유사 |
| `.ios.tsx` / `.android.tsx` | 타겟별 소스 분리 | flavor/sourceSet 분리 |
| `shadowColor/Offset/Opacity/Radius` | `CALayer.shadow*` (그대로 매핑) | (대응 없음 — 이 속성들 무시됨) |
| `elevation` | (대응 없음) | `View.elevation` (Material 그림자) |
| react-native-safe-area-context | `safeAreaLayoutGuide` / SwiftUI safe area | `WindowInsets` |
| `KeyboardAvoidingView` | 키보드 프레임 노티 기반 수동 회피 | `windowSoftInputMode="adjustResize"` 계열 |
| 헤더 백 제스처/백 버튼 | 스와이프 백 (edge pan) | 시스템 백 버튼/백 제스처 |

## 🧭 왜 이렇게 설계됐나

RN의 약속은 "learn once, write anywhere"다 — "write once, run anywhere"가 **아니다**. 두 플랫폼은 그림자 렌더링, 키보드 처리, 내비게이션 관습, 폰트 메트릭이 근본적으로 다르고, RN은 이를 억지로 통일하는 대신 **각 플랫폼의 네이티브 뷰를 그대로 쓰면서** 필요한 곳에 분기 도구를 제공하는 쪽을 택했다. Flutter(자체 렌더링으로 통일)와의 철학적 차이가 바로 이 지점이다.

분기 도구가 두 층인 이유:

- **런타임 분기(`Platform`)**: 값 몇 개, 스타일 몇 줄 차이 — 코드 한가운데서 가볍게.
- **빌드 타임 분기(파일 확장자)**: 구현 자체가 다를 때 — Metro 번들러가 타겟 플랫폼에 맞는 파일만 번들에 포함하므로, 반대 플랫폼 코드는 바이너리에 아예 실리지 않는다. iOS 타겟별 소스 분리, Android sourceSet과 같은 발상.

## ⚙️ 동작 원리

### 1. `Platform.OS` / `Platform.select` — 런타임 분기

```typescript
import { Platform } from 'react-native';

Platform.OS;        // 'ios' | 'android' (web이면 'web')
Platform.Version;   // iOS: 문자열(예: '18.1') / Android: API 레벨 숫자(예: 35)

const behavior = Platform.OS === 'ios' ? 'padding' : undefined;

const styles = {
  header: {
    ...Platform.select({
      ios: { fontFamily: 'SF Pro Display' },
      android: { fontFamily: 'sans-serif-medium' },
      default: {},
    }),
  },
};
```

`Platform.select`는 객체에서 현재 플랫폼 키를 골라주는 헬퍼일 뿐이지만, 스타일 시트 안에서 삼항 연산자 중첩을 피하게 해 줘서 관용구로 굳었다.

### 2. 파일 확장자 분기 — 빌드 타임 해석

```
components/
  DatePicker.tsx          # 공통 (또는 없음)
  DatePicker.ios.tsx      # iOS 빌드에서 import 'DatePicker' 하면 이것
  DatePicker.android.tsx  # Android 빌드에서는 이것
```

`import { DatePicker } from './DatePicker'` — import 경로에는 확장자를 쓰지 않으며, Metro가 번들링 시점에 플랫폼에 맞는 파일을 고른다. 인터페이스(props 타입)는 공통 `.d.ts`나 공용 타입 파일로 맞춰 두는 것이 관례. 컴포넌트뿐 아니라 유틸(`haptics.ios.ts`)에도 동일하게 적용된다. `.native.tsx`(iOS+Android 공통, web 제외)도 있다.

**선택 기준**: 분기가 서너 줄이면 `Platform.select`, 파일의 절반이 달라지면 확장자 분기. 조건문 범벅인 파일이 보이면 후자로 리팩터링할 신호다.

### 3. 그림자 — 가장 자주 마주치는 스타일 분기

- **iOS**: `shadowColor` / `shadowOffset` / `shadowOpacity` / `shadowRadius` — `CALayer` 그림자 프로퍼티에 그대로 매핑. Android에서는 전부 무시된다.
- **Android**: `elevation` 하나 — Material elevation으로 그림자+z순서. iOS에서는 그림자를 만들지 않는다(z-order에는 영향).

```typescript
const cardShadow = Platform.select({
  ios: {
    shadowColor: '#000',
    shadowOffset: { width: 0, height: 2 },
    shadowOpacity: 0.15,
    shadowRadius: 8,
  },
  android: { elevation: 4 },
});
```

RN 0.76부터 [[New Architecture]] 기반의 크로스 플랫폼 `boxShadow` 스타일(CSS 문법)이 추가되어 이 이원화를 통합하는 방향이다 — 지원 범위와 제약은 사용 중인 RN 버전의 공식 문서 확인. 당분간은 위의 이중 정의 패턴이 실무 기본값이다.

### 4. Safe Area — react-native-safe-area-context

노치/펀치홀/홈 인디케이터 회피. iOS `safeAreaLayoutGuide`, Android `WindowInsets`에 해당하는 값을 하나의 API로 노출한다.

- 앱 루트에 `SafeAreaProvider` (Expo Router는 기본 제공 — 공식 문서 확인).
- 화면에서 `useSafeAreaInsets()`로 `{ top, bottom, left, right }`를 받아 패딩에 반영하거나, `SafeAreaView`로 감싼다.
- RN 코어의 `SafeAreaView`는 iOS 전용 + 기능 제한이라, **라이브러리 쪽을 쓰는 것이 표준**이다.

Android는 전통적으로 시스템 바가 콘텐츠 밖에 있었지만 edge-to-edge가 기본이 되는 흐름이라, 이제 양 플랫폼 모두 inset 처리가 필수라고 생각하는 편이 안전하다.

### 5. KeyboardAvoidingView — 플랫폼 차이의 축소판

```tsx
<KeyboardAvoidingView
  behavior={Platform.OS === 'ios' ? 'padding' : undefined}
  style={{ flex: 1 }}
>
```

- **iOS**: 시스템이 레이아웃을 조정해 주지 않으므로 JS가 키보드 노티를 받아 `padding`/`position`/`height` 방식으로 직접 회피해야 한다.
- **Android**: `windowSoftInputMode`(Expo 기본 `adjustResize` 계열)가 창 크기를 줄여 주므로, `behavior`를 지정하지 않거나 다르게 줘야 이중 회피로 레이아웃이 튀는 걸 막는다.

키보드 UX가 복잡하다면(채팅 입력창 등) 커뮤니티에서는 keyboard-controller 계열 라이브러리를 쓰는 흐름도 있다 — 공식 문서/라이브러리 문서 확인.

### 6. 관습 차이 — 어디까지 존중할 것인가

기술 분기보다 어려운 판단. 기준선:

- **OS가 소유한 인터랙션은 무조건 존중**: Android 시스템 백 버튼/백 제스처는 반드시 동작해야 하고, iOS 엣지 스와이프 백도 임의로 막지 않는다. [[React Navigation]]/[[Expo Router]]가 기본 처리해 주는 것을 깨지 말 것.
- **플랫폼 기본 컴포넌트 관습은 가급적 존중**: 날짜 선택기, 알럿, 헤더의 백 버튼 모양(iOS: chevron+타이틀, Android: 화살표), 타이포그래피 기본값 등은 내비게이션 라이브러리와 플랫폼 기본값에 맡기면 각자 자연스럽다.
- **브랜드 UI는 통일해도 된다**: 버튼, 카드, 색상, 커스텀 컴포넌트는 디자인 시스템대로 양쪽 동일하게. 사용자는 "우리 앱답다"를 기대하지 "iOS답다"를 카드 모서리에서 기대하지 않는다.

경험칙: **네비게이션·시스템 인터랙션은 플랫폼답게, 콘텐츠·브랜드는 앱답게.**

## 💻 코드 예시

기준: RN 0.76+ / Expo SDK 57 / react-native-safe-area-context (`npx expo install react-native-safe-area-context`).

```typescript
// NoteComposer.tsx — safe area + 키보드 회피 + 플랫폼 그림자를 한 화면에
import React, { useState } from 'react';
import {
  KeyboardAvoidingView,
  Platform,
  StyleSheet,
  TextInput,
  View,
} from 'react-native';
import { useSafeAreaInsets } from 'react-native-safe-area-context';

export function NoteComposer() {
  const insets = useSafeAreaInsets();
  const [text, setText] = useState('');

  return (
    <KeyboardAvoidingView
      style={styles.flex}
      behavior={Platform.OS === 'ios' ? 'padding' : undefined}
    >
      <View
        style={[
          styles.flex,
          { paddingTop: insets.top, paddingBottom: insets.bottom },
        ]}
      >
        <View style={[styles.card, cardShadow]}>
          <TextInput
            value={text}
            onChangeText={setText}
            placeholder="오늘의 기록…"
            multiline
          />
        </View>
      </View>
    </KeyboardAvoidingView>
  );
}

const cardShadow = Platform.select({
  ios: {
    shadowColor: '#000',
    shadowOffset: { width: 0, height: 2 },
    shadowOpacity: 0.15,
    shadowRadius: 8,
  },
  android: { elevation: 4 },
});

const styles = StyleSheet.create({
  flex: { flex: 1 },
  card: {
    margin: 16,
    padding: 16,
    borderRadius: 12,
    backgroundColor: '#fff',
  },
});
```

(앱 루트에 `SafeAreaProvider`가 있어야 `useSafeAreaInsets`가 동작한다.)

## ⚠️ 함정 (Pitfalls)

1. **한쪽 플랫폼에서만 테스트하는 습관** — 이 장 전체의 결론이자 최대 함정. 시뮬레이터가 편한 쪽(대개 iOS)만 보다가 릴리즈 직전에 Android에서 그림자 실종, 키보드가 입력창 가림, 백 버튼에 앱 종료 같은 문제를 몰아서 발견한다. **기능 단위로 양쪽 확인**을 습관화할 것. 특히 그림자·키보드·safe area·백 처리 4종은 분기 자체가 다르므로 한쪽 통과가 다른 쪽을 전혀 보증하지 않는다.
2. **Android에서 `shadow*`만 쓰고 그림자가 없다고 당황** — `shadow*`는 iOS 전용. `elevation`을 같이 정의해야 한다. 반대로 `elevation`은 iOS에서 아무 그림자도 만들지 않는다.
3. **Android 그림자 실종의 다른 원인** — `elevation`이 있어도 `backgroundColor`가 없으면 그림자가 안 그려지고, 부모의 `overflow: 'hidden'`이 그림자를 잘라먹는다. iOS의 `masksToBounds`와 그림자가 공존 못 하던 그 문제의 데자뷔.
4. **`Platform.Version` 타입 혼동** — iOS는 문자열, Android는 숫자(API 레벨). 숫자 비교 전에 플랫폼 확인 필수.
5. **KeyboardAvoidingView에 양쪽 모두 `behavior="padding"`** — Android에서 `adjustResize`와 겹쳐 이중 회피 → 입력창이 키보드보다 한참 위로 튄다. 플랫폼별로 다르게 줄 것.
6. **RN 코어 `SafeAreaView`를 그대로 사용** — iOS 전용이고 유연성이 없다. react-native-safe-area-context로 통일.
7. **파일 확장자 분기의 타입 불일치** — `.ios.tsx`와 `.android.tsx`의 props 시그니처가 슬금슬금 어긋나면 한쪽 빌드만 깨진다. 공용 타입을 별도 파일로 빼서 양쪽이 import 하게 만들 것.
8. **Android 백 버튼 무시** — 모달·편집 중 상태에서 시스템 백을 처리하지 않으면 데이터 유실 또는 즉시 앱 종료. iOS 출신 개발자가 가장 자주 빠뜨리는 지점. [[React Navigation]]의 `beforeRemove` 이벤트나 `BackHandler` 처리 확인.
