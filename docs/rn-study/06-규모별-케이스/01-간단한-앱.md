# 간단한 앱 — 1인 개발 / 빠른 출시

> [!abstract] 한 줄 요약
> 혼자서 빠르게 스토어까지 가야 할 때, 기술 스택 결정을 최소화하고 커뮤니티 기본값(default)을 그대로 따르는 전략.

## 🔁 iOS/AOS 대응 개념

| RN (이 케이스의 선택) | iOS | Android |
|---|---|---|
| Expo (managed workflow) | Xcode 템플릿 + 수동 설정 없이 시작하는 것에 해당 (직접 대응물 없음) | Android Studio 템플릿 + Gradle 수동 설정 없이 시작 |
| [[Expo Router]] 또는 [[React Navigation]] | UIKit Navigation / SwiftUI NavigationStack | Jetpack Navigation / Compose Navigation |
| Zustand | 단순한 ObservableObject / @Observable | 단순한 ViewModel + StateFlow |
| [[EAS]] Build | Xcode Cloud / fastlane | Play Console + CI |

네이티브에서 "새 프로젝트 → 아키텍처 고민(TCA? MVVM? 모듈 분리?)"에 쓰던 에너지를, 이 케이스에서는 **의도적으로 쓰지 않는 것**이 핵심이다.

## 📌 핵심 내용

### 왜 "결정 최소화"가 전략인가

1인 개발에서 가장 비싼 자원은 시간이 아니라 **결정 피로**다. RN 생태계는 같은 문제에 대한 대안이 항상 3개 이상 있다(내비게이션, 상태관리, 스타일링, 폼, 네트워킹…). 각 선택지를 비교·검증하는 비용이 앱 한 개 만드는 비용보다 커지기 쉽다.

그래서 이 케이스의 전략은 단순하다: **커뮤니티/Expo가 검증한 기본값을 그대로 쓰고, 커스텀은 필요가 증명될 때까지 미룬다.**

권장 기본 스택 (Expo SDK 57 / RN 0.76+ 기준):

| 영역 | 선택 | 이유 |
|---|---|---|
| 프레임워크 | Expo | 네이티브 프로젝트를 직접 관리하지 않음([[Prebuild]]). 업그레이드·빌드·제출까지 한 팀이 관리하는 수직 통합 |
| 라우팅 | [[Expo Router]] | 파일 기반 라우팅. 딥링크가 공짜로 따라옴. 결정할 것이 "파일을 어디 둘까"뿐 |
| 상태관리 | Zustand (또는 [[State]] + Context) | 보일러플레이트가 거의 없음. Redux급 구조는 이 규모에서 과잉 |
| 서버 상태 | TanStack Query (서버 통신이 있다면) | 캐싱/재시도/로딩 상태를 직접 만들지 않음 |
| 빌드/제출 | [[EAS]] Build + Submit | 로컬에 Xcode/Gradle 환경 세팅 없이 CI 빌드 → 스토어 제출 |
| 스타일 | `StyleSheet.create` + 토큰 파일 하나 | 스타일 라이브러리 도입은 미룸 |

핵심은 각 칸의 선택지가 "최고"라서가 아니라, **문서·예제·트러블슈팅 자료가 가장 많은 길**이라서다. 혼자 막혔을 때 구글/GPT가 바로 답을 주는 스택이 1인 개발에서는 성능보다 중요하다.

### 폴더 구조 — 컨벤션을 따른다

[[Expo Router]]를 쓰면 `app/` 디렉토리가 곧 라우트다. 구조 설계라는 결정 자체가 사라진다:

```
my-app/
├── app/                  # 라우트 = 파일 (Expo Router)
│   ├── _layout.tsx       # 루트 레이아웃 (탭/스택 선언)
│   ├── index.tsx         # 홈 화면
│   ├── settings.tsx
│   └── record/
│       └── [id].tsx      # 동적 라우트
├── components/           # 재사용 UI 조각
├── stores/               # Zustand 스토어 (도메인당 1파일)
├── lib/                  # 유틸, API 클라이언트
├── theme/                # 색·간격·타이포 토큰
├── assets/
└── app.json              # Expo 설정 (네이티브 설정의 단일 소스)
```

**이 레포(Logit)가 정확히 이 케이스의 실례다.** Logit은 [[React Navigation]]을 직접 쓰는 변형이라 `src/screens` + `src/navigation`으로 화면과 라우팅을 나눴지만, 발상은 같다 — `src/components`(공용 UI), `src/store`(상태), `src/theme`(디자인 토큰), `src/data`(도메인 데이터), `src/lib`(유틸). 도메인별 레이어드 모듈 분리, 의존성 주입 컨테이너, 멀티 모듈 같은 것은 없다. 이 규모에서는 그게 맞다.

### "필요해질 때까지 안 만드는 것" 목록

시니어일수록 처음부터 잘 갖추고 싶은 유혹이 크다. 아래는 **도입 신호가 올 때까지 미루는 것**과 그 신호다.

| 미루는 것 | 도입 신호 |
|---|---|
| 무거운 상태관리 (Redux Toolkit, MobX) | Zustand 스토어끼리 의존이 얽히고, 상태 변화 추적이 안 되기 시작할 때. 그 전까지는 스토어 파일을 쪼개는 것으로 충분 |
| [[Monorepo]] | 앱이 2개 이상이 되고 코드를 실제로 공유할 때. "언젠가 공유하겠지"는 신호가 아님 |
| E2E 테스트 (Maestro) | 릴리즈마다 손으로 확인하는 핵심 플로우가 5개를 넘고, 회귀가 실제로 발생했을 때 |
| 단위 테스트 인프라 | 순수 로직(계산, 파싱, 정렬)이 생기면 그 파일부터. UI 스냅샷 테스트는 이 규모에서 유지비 > 효용 |
| 디자인 시스템 패키지 | 같은 컴포넌트를 세 번째 복붙할 때 `components/`로 승격. 별도 패키지는 팀이 생길 때 |
| 커스텀 [[Config Plugin]] | Expo 기본 제공 + 커뮤니티 플러그인으로 안 되는 네이티브 설정이 실제로 필요할 때 |
| [[Dev Client]] 커스텀 네이티브 모듈 | Expo SDK에 없는 네이티브 기능이 필요할 때 — 그 전까지는 SDK 모듈만으로 감 |
| 자체 백엔드 | Supabase/Firebase 같은 BaaS로 시작. 벤더 제약이 실제 기능을 막을 때 |
| [[OTA Update]] 운영 체계 | 출시 후 "심사 없이 급히 고쳐야 하는 상황"을 한 번 겪은 뒤에 |
| CI 파이프라인 정교화 | 처음엔 `eas build` 수동 실행으로 충분. 릴리즈 주기가 규칙적이 될 때 자동화 |

반대로 **처음부터 갖추는 것**은 짧다: TypeScript strict 모드, ESLint + Prettier, git. 이 셋은 나중에 넣는 비용이 처음 넣는 비용의 수십 배다.

### 개발 루프

- 로컬: `npx expo start` → [[Dev Client]] 또는 시뮬레이터에서 Fast Refresh로 개발.
- 네이티브 설정 변경(권한, 아이콘, 플러그인)은 `app.json`에서 → [[Prebuild]]가 iOS/Android 프로젝트에 반영. `ios/`, `android/` 디렉토리를 직접 수정하지 않는 것이 원칙(수정하는 순간 업그레이드 비용이 급증 — [[02-버전-업그레이드-전략]] 참고).
- 배포: `eas build --profile production` → `eas submit`. 서명 키/프로비저닝 프로파일은 EAS가 관리해 준다(네이티브에서 하던 인증서 지옥의 상당 부분이 사라짐).

## 💻 코드/설정 예시

Zustand 스토어 — 이 규모의 상태관리는 이게 전부다 (TypeScript, Zustand 5 기준):

```typescript
// stores/useRecordStore.ts
import { create } from 'zustand';

type Record = { id: string; title: string; createdAt: number };

type RecordStore = {
  records: Record[];
  add: (title: string) => void;
  remove: (id: string) => void;
};

export const useRecordStore = create<RecordStore>((set) => ({
  records: [],
  add: (title) =>
    set((s) => ({
      records: [...s.records, { id: crypto.randomUUID(), title, createdAt: Date.now() }],
    })),
  remove: (id) =>
    set((s) => ({ records: s.records.filter((r) => r.id !== id) })),
}));

// 컴포넌트에서 — 필요한 조각만 구독 (불필요한 Re-render 방지)
const records = useRecordStore((s) => s.records);
```

EAS 빌드 프로파일 — 결정할 것이 거의 없는 최소 구성:

```json
// eas.json
{
  "build": {
    "development": {
      "developmentClient": true,
      "distribution": "internal"
    },
    "production": {
      "autoIncrement": true
    }
  },
  "submit": {
    "production": {}
  }
}
```

### 출시 전 최소 체크리스트

이 규모에서도 생략하면 안 되는 것들:

- [ ] release 빌드를 실기기에서 최소 한 번 전체 플로우 확인 (dev와 release는 다른 물건 — [[01-릴리즈-파이프라인]])
- [ ] 앱 아이콘·스플래시·권한 문구(`app.json`의 permission description) 확인 — 심사 반려 단골
- [ ] 크래시 수집(Sentry) 연결 — 1인 개발일수록 사용자 제보 외에 수단이 없다
- [ ] 개인정보처리방침 URL — 양 스토어 공통 요구
- [ ] Android `versionCode` / iOS build number 자동 증가 설정 (`autoIncrement`)

## ⚠️ 함정 (Pitfalls)

- **네이티브 시니어의 과잉 설계.** iOS/AOS에서 하던 대로 첫 주에 모듈 분리, DI, 프로토콜 추상화부터 깔면 이 케이스의 이점(속도)이 사라진다. RN + Expo의 생산성은 "구조를 안 만들어도 되는 규모에서 안 만드는 것"에서 나온다.
- **`ios/`·`android/` 디렉토리를 직접 수정하고 커밋.** 그 순간 managed workflow의 이점이 깨지고, SDK 업그레이드마다 네이티브 diff를 손으로 관리해야 한다. 네이티브 설정은 `app.json` / [[Config Plugin]] 경유가 원칙.
- **Expo Go로 끝까지 가려는 시도.** Expo Go는 데모용이다. SDK에 포함되지 않은 네이티브 모듈이 하나라도 필요해지는 순간 [[Dev Client]] 빌드로 전환해야 하며, 처음부터 Dev Client 기준으로 개발 루프를 잡는 편이 낫다.
- **AsyncStorage에 민감 정보 저장.** 간단한 앱이라도 토큰은 `expo-secure-store`(Keychain/Keystore)로. [[04-보안과-접근성]] 참고.
- **"나중에 필요할 라이브러리" 선설치.** 네이티브 의존성 하나하나가 업그레이드 비용이다. 의존성 수는 이 케이스의 핵심 관리 지표.
- **상태관리 라이브러리부터 고르느라 일주일 소모.** Zustand로 시작해서 문제가 생기면 그때 바꿔도 마이그레이션 비용은 이 규모에서 하루 이하다.
