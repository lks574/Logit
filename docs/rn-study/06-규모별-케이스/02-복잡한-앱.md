# 복잡한 앱 — 대규모 단일 앱

> [!abstract] 한 줄 요약
> 화면 수십 개, 네이티브 모듈 다수, 실사용자 트래픽이 있는 단일 앱에서 성능·안정성·운영을 어떻게 감당하는가.

## 🔁 iOS/AOS 대응 개념

| RN | iOS | Android |
|---|---|---|
| [[Re-render]] 최적화 | 불필요한 `body` 재계산 방지 (SwiftUI) / 셀 재사용·diffable data source | Compose recomposition 최적화 / RecyclerView 튜닝 |
| [[Hermes]] 프로파일링 | Instruments Time Profiler | Android Studio Profiler / Perfetto |
| Sentry (JS + 네이티브) | Crashlytics + 심볼리케이션 | Crashlytics + ProGuard mapping |
| 딥링크 (Universal Links) | Universal Links / URL Scheme | App Links / Intent Filter |
| 백그라운드 작업 제약 | BGTaskScheduler, 30초 룰 | WorkManager, Doze/App Standby |
| [[OTA Update]] | 대응물 없음 (RN 고유) | 대응물 없음 (RN 고유) |

마지막 두 줄이 이 케이스의 요점이다: **백그라운드·푸시·딥링크는 네이티브 지식이 그대로 적용**되고, **OTA는 네이티브에 없던 새로운 운영 축**이다.

## 📌 핵심 내용

### 성능 — 문제의 90%는 JS 스레드의 [[Re-render]]

복잡한 앱에서 "느리다"의 원인은 대부분 네이티브가 아니라 JS 층이다. 진단 순서:

1. **React DevTools Profiler로 리렌더 추적.** React Native DevTools(RN 0.76+ 기본, `j` 키로 실행)에 포함된 Profiler에서 커밋별로 어떤 컴포넌트가 왜 렌더됐는지 본다. "Highlight updates" 계열 기능으로 화면에서 깜빡이는 영역을 눈으로 확인하는 것부터 시작.
2. **원인 패턴 식별.** 상위 컴포넌트의 [[State]] 변경이 트리 전체를 다시 그리는 경우, 렌더마다 새로 만들어지는 객체/함수가 자식의 props 비교를 무력화하는 경우가 압도적 다수.
3. **[[Memoization]] 적용.** `React.memo` / `useMemo` / `useCallback`을 **측정된 병목에만** 적용. (React Compiler가 이를 자동화하는 방향이지만, 활성화 여부·안정성은 사용 중인 버전의 공식 문서 확인.)

**리스트 튜닝.** 대규모 앱의 체감 성능은 리스트가 좌우한다.

- [[FlatList]] 기본기: `keyExtractor` 안정화, `renderItem`에 넘기는 아이템 컴포넌트를 `React.memo`로 감싸기, 아이템 높이가 고정이면 `getItemLayout` 제공.
- 아이템 수가 수백 개 이상이거나 스크롤 성능이 목표에 못 미치면 **FlashList(Shopify)** 로 교체하는 것이 사실상 표준. 셀 재활용 방식이 RecyclerView/UITableView에 가깝다.
- 리스트 아이템 안에서 인라인 익명 함수·객체 생성은 아이템 수만큼 곱해진다 — 리스트에서 가장 흔한 리렌더 폭풍의 진원지.

**시작 시간(TTI).** [[Hermes]]가 바이트코드 AOT로 파싱 비용을 이미 제거했으므로, 남는 병목은 (1) 초기 [[Bundle]]에서 실행되는 JS 양, (2) 첫 화면 렌더까지의 컴포넌트 트리 깊이. 대응: 초기 라우트에 불필요한 모듈 import 제거(lazy import), 첫 화면에서 무거운 컴포넌트 지연 마운트, [[Metro]] 번들 분석으로 뚱뚱한 의존성 식별. 네이티브 쪽 시작 시간(dyld, Application onCreate)은 기존 네이티브 지식 그대로.

**프로파일링 도구 정리.** JS 층은 React DevTools Profiler + Hermes CPU 프로파일(React Native DevTools의 Performance 패널), 네이티브 층은 Instruments/Android Studio Profiler를 그대로 쓴다. 어느 층인지 판단하는 방법은 [[03-디버깅과-프로파일링]] 참고. Flipper는 deprecated — 새로 배우지 말 것.

### 네이티브 모듈 다수 + 커스텀 [[Config Plugin]]

복잡한 앱은 네이티브 의존성이 수십 개다(카메라, 지도, 결제, 분석, 푸시…). Expo [[Prebuild]] 체제를 유지하려면 각 모듈의 네이티브 설정(Info.plist 키, Gradle 설정, entitlements)을 [[Config Plugin]]으로 선언한다.

- 대부분의 주요 라이브러리는 자체 플러그인을 제공한다 — `app.json`의 `plugins` 배열에 추가하면 끝.
- 제공하지 않는 설정(예: 특정 entitlement, 커스텀 Notification Service Extension)은 **직접 플러그인을 작성**한다. 플러그인은 "prebuild 시점에 네이티브 프로젝트 파일을 수정하는 JS 함수"일 뿐이다.
- 이 체제의 보상: `ios/`·`android/`를 git에서 지우고도(CNG) 재현 가능한 네이티브 프로젝트를 유지 → 업그레이드 비용 급감([[02-버전-업그레이드-전략]]).

### 에러 추적 — JS 에러와 네이티브 크래시는 다른 생물

Sentry(RN에서 사실상 표준, `@sentry/react-native`)는 두 층을 모두 수집하지만 성격이 다르다:

| | JS 에러 | 네이티브 크래시 |
|---|---|---|
| 증상 | 릴리즈 빌드에서는 기본적으로 에러 경계/글로벌 핸들러로 잡힘, 앱이 죽지 않을 수 있음 | 프로세스 즉사 (네이티브에서 알던 그 크래시) |
| 심볼화 재료 | **source map** (Metro가 생성, minify된 스택 → 원본 코드) | dSYM / ProGuard mapping (기존 지식 그대로) |
| 대응 | [[OTA Update]]로 즉시 수정 가능 | 스토어 재배포 필요 |

운영 포인트: 빌드 파이프라인에서 source map과 dSYM **둘 다** 업로드되는지 확인할 것. 하나라도 빠지면 절반의 크래시가 읽을 수 없는 스택으로 온다. Sentry의 EAS/Expo 통합 플러그인이 이를 자동화한다.

### 딥링크·푸시·백그라운드 — 네이티브 지식이 그대로 쓰이는 영역

- **딥링크**: OS 레벨 설정(Universal Links의 AASA 파일, Android App Links의 assetlinks.json + intent filter)은 네이티브 지식 그대로. RN 층에서는 [[Expo Router]]라면 파일 경로가 곧 딥링크 경로고, [[React Navigation]]이라면 linking config로 URL→화면 매핑을 선언한다.
- **푸시 알림(개요)**: 토큰 발급·APNs/FCM 구성·권한 요청은 네이티브와 동일한 구조. RN 층 선택지는 expo-notifications 또는 Notifee/react-native-firebase 조합. 리치 푸시(이미지, Notification Service Extension)는 결국 네이티브 확장 작업이며 Config Plugin으로 프로젝트에 주입한다.
- **백그라운드 작업**: RN이라고 OS 제약이 사라지지 않는다. iOS BGTaskScheduler의 기회주의적 스케줄링, Android Doze — 전부 그대로 적용된다. JS 코드는 앱 프로세스가 살아있을 때만 돈다는 점이 추가 제약. "백그라운드에서 JS 타이머 돌리기" 같은 발상은 성립하지 않는다.

### [[OTA Update]] 운영 정책

EAS Update로 JS [[Bundle]]과 에셋을 스토어 심사 없이 교체할 수 있다. 단:

- **JS-only 변경만 가능.** 네이티브 코드/의존성이 바뀌면 runtime version이 달라져 OTA 불가 — 스토어 배포로만.
- **스토어 규정과의 관계**: Apple/Google 모두 인터프리터로 실행되는 스크립트의 업데이트를 조건부 허용하지만, **앱의 목적·성격을 바꾸는 변경이나 심사 우회 목적의 기능 은닉은 금지**다. OTA는 "버그 수정과 마이너 개선" 채널로 운영하고, 큰 기능은 스토어 릴리즈에 태우는 것이 안전한 정책. (정확한 조항은 App Store Review Guidelines / Play 정책 공식 문서 확인.)
- 운영 규칙 예: 채널(production/staging) 분리, 점진 롤아웃, 문제 시 롤백 절차 문서화, "OTA로 나간 버전"도 릴리즈 노트와 Sentry release로 추적.

### 기능 플래그

OTA와 조합하면 강력하다: 코드는 미리 배포하고 플래그로 켠다. 스토어 심사 일정과 기능 공개 일정을 분리할 수 있다. LaunchDarkly/Statsig/Firebase Remote Config 등 선택지는 네이티브와 동일 — RN SDK가 있는지만 확인. 주의: 플래그 평가 전 첫 렌더가 어떤 상태로 보일지(깜빡임) 설계할 것.

### [[Monorepo]]

앱이 커지면 디자인 시스템·유틸·API 클라이언트를 패키지로 분리하고 pnpm workspaces + Turborepo(빌드 캐싱)로 묶는 구성이 흔하다. RN 특유의 주의점은 [[Metro]]:

- Metro가 workspace 루트의 패키지를 해석하도록 `watchFolders`와 resolver 설정 필요. Expo는 monorepo를 자동 감지해 상당 부분 처리해 주지만, 정확한 설정은 Expo 공식 monorepo 가이드 확인.
- **React/react-native가 단일 인스턴스여야 한다.** hoisting으로 버전이 두 벌 깔리면 "Invalid hook call" 같은 기괴한 에러가 난다. 해결은 버전 고정 + 중복 검사.
- 네이티브 의존성이 있는 패키지는 [[Autolinking]]이 워크스페이스 경로를 찾을 수 있어야 한다.

### 이 규모에서 팀이 유지해야 하는 운영 문서

코드 밖에서 필요한 것들 — 없으면 담당자 퇴사와 함께 증발하는 지식:

- **의존성 대장**: 네이티브 코드를 포함한 라이브러리 목록 + 각각의 대안·업그레이드 리스크. [[02-버전-업그레이드-전략]]의 입력 자료가 된다.
- **OTA 정책 문서**: 무엇을 OTA로 내보낼 수 있는가, 롤백 절차, 발행 권한자.
- **버전 3축 현황**: 현재 스토어에 살아있는 바이너리들의 runtime version 목록 ([[01-릴리즈-파이프라인]]).
- **증상→층 판정 가이드**: 신규 합류자가 "이거 JS 문제예요, 네이티브 문제예요?"를 스스로 판단할 수 있게 ([[03-디버깅과-프로파일링]]).

## 💻 코드/설정 예시

리스트 리렌더 억제의 전형 (RN 0.76+):

```tsx
// ❌ 매 렌더마다 새 함수·새 객체 → 모든 아이템 리렌더
<FlatList
  data={items}
  renderItem={({ item }) => <Row item={item} onPress={() => open(item.id)} />}
/>

// ✅ 아이템 memo + 콜백 안정화
const Row = React.memo(function Row({ item, onPress }: RowProps) {
  return <Pressable onPress={() => onPress(item.id)}>…</Pressable>;
});

const renderItem = useCallback(
  ({ item }: ListRenderItemInfo<Item>) => <Row item={item} onPress={open} />,
  [open], // open도 useCallback으로 안정화되어 있어야 함
);

<FlatList data={items} renderItem={renderItem} keyExtractor={(i) => i.id} />
```

커스텀 [[Config Plugin]] 뼈대 — Info.plist에 키 하나 추가:

```typescript
// plugins/withMyPlistKey.ts
import { ConfigPlugin, withInfoPlist } from 'expo/config-plugins';

const withMyPlistKey: ConfigPlugin = (config) =>
  withInfoPlist(config, (c) => {
    c.modResults.NSFaceIDUsageDescription = '잠금 해제에 Face ID를 사용합니다.';
    return c;
  });

export default withMyPlistKey;
```

```json
// app.json
{ "expo": { "plugins": ["./plugins/withMyPlistKey"] } }
```

## ⚠️ 함정 (Pitfalls)

- **측정 없는 최적화.** 모든 컴포넌트에 `React.memo`를 도포하는 것은 비교 비용만 추가한다. Profiler로 병목을 찾고 그곳만 고친다.
- **리렌더 문제를 네이티브 문제로 오진.** "스크롤이 버벅인다 → 네이티브 리스트가 느린가?" — 아니다. 십중팔구 JS 스레드가 리렌더로 포화된 것. 증상→원인 매핑은 [[03-디버깅과-프로파일링]].
- **OTA를 기능 배포 채널로 남용.** 심사 우회로 간주될 수 있는 변경을 OTA로 내보내면 계정 리스크. 정책을 문서화하고 지킬 것.
- **source map 업로드 누락.** JS 에러가 `index.android.bundle:1:2847391` 로만 찍히는 Sentry는 없는 것과 같다.
- **네이티브 크래시를 JS 층에서 찾기 (또는 그 반대).** 층 판단이 먼저다. 크래시 로그에 Hermes/JS 스택이 없으면 네이티브 문제 — 기존 크래시 분석 지식으로 간다.
- **monorepo에서 react-native 이중 설치.** 증상이 원인과 멀리 떨어져 나타나는 대표 케이스. 패키지 추가 후 이상하면 먼저 중복 설치부터 의심.
- **백그라운드에서 JS가 계속 돈다고 가정.** 업로드 재개, 위치 추적 같은 것은 네이티브 모듈(과 OS 제약)의 영역이다.

- **기능 플래그 평가 전 첫 렌더 방치.** 스플래시가 걷힌 뒤 플래그가 로드되며 UI가 바뀌는 깜빡임 — 캐시된 플래그로 첫 렌더, 갱신은 다음 실행에 반영하는 정책이 무난하다.
