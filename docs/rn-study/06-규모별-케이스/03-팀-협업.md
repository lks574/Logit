# 팀 협업 — 여러 개발자가 함께 만드는 RN 앱

> 팀 규모에서는 "잘 짜는 것"보다 "서로 다르게 짜지 못하게 하는 것"이 생산성을 결정한다 — 타입·린트·테스트·CI를 기계로 강제하는 방법.

## iOS/AOS 대응 개념

| RN | iOS | Android |
|---|---|---|
| TypeScript strict | Swift의 타입 시스템 (기본 제공되던 것을 RN에선 설정으로 켜야 함) | Kotlin의 타입 시스템 |
| ESLint + Prettier | SwiftLint + SwiftFormat | ktlint / detekt |
| Jest 단위 테스트 | XCTest (유닛) | JUnit |
| React Native Testing Library | ViewInspector류 / XCTest UI 일부 | Compose UI Test / Robolectric |
| Maestro (E2E) | XCUITest | Espresso / UI Automator |
| [[EAS]] Build + 브랜치 전략 | Xcode Cloud / fastlane + CI | GitHub Actions + Gradle |

네이티브 팀에서 하던 협업 장치가 거의 1:1로 존재한다. 다른 점은 **JS 생태계에서는 이 장치들이 기본으로 켜져 있지 않다**는 것 — 팀이 명시적으로 강제해야 한다.

## 핵심 내용

### TypeScript strict + 린트 — CI 게이트로 강제

JS는 컴파일러가 막아주던 실수를 런타임까지 통과시킨다. 팀 협업에서 이건 재앙이므로:

- **`tsconfig.json`에 `"strict": true`는 협상 불가.** `any` 남용은 lint 규칙(`@typescript-eslint/no-explicit-any` 등)으로 조인다. Swift/Kotlin에서 오는 개발자에게 strict TS는 "원래 있던 안전망을 되찾는 것"이다.
- **ESLint**: `eslint-config-expo` 또는 RN 공식 설정에서 시작 + `eslint-plugin-react-hooks`는 필수. `exhaustive-deps` 규칙이 의존성 배열 버그(팀에서 가장 흔한 RN 버그)를 기계적으로 잡는다.
- **Prettier**: 포맷 논쟁을 제거하는 도구. 설정을 최소화하고 기본값을 받아들일 것.
- **강제 방식**: CI에서 `tsc --noEmit` + `eslint` + `prettier --check`가 실패하면 머지 불가. 로컬 pre-commit hook(husky + lint-staged)은 보조 수단이지 대체재가 아니다 — hook은 우회 가능하므로 게이트는 CI에 둔다.

### 디자인 시스템 / 공용 컴포넌트 계층

여러 명이 화면을 만들면 같은 버튼이 네 가지로 구현된다. 방지책은 계층 규칙:

```
theme/       → 토큰 (색·간격·타이포·radius). 원시값 직접 사용 금지
components/  → 디자인 시스템 (Button, Card, Sheet…). 토큰만 소비
features/    → 화면·기능. 디자인 시스템만 조립, 스타일 하드코딩 금지
```

- 규칙은 문서가 아니라 **lint로 강제**할 수 있는 만큼 강제한다(예: 색상 리터럴 금지 규칙).
- 컴포넌트에는 스토리북 대신 가벼운 갤러리 화면(개발 빌드 전용 라우트)도 충분하다. 도구보다 "새 화면을 만들 때 여기부터 본다"는 합의가 본질.
- 규모가 더 커지면 [[Monorepo]]에서 디자인 시스템을 패키지로 분리 — 단, [[Metro]] 설정 주의([[02-복잡한-앱]] 참고).

### 테스트 피라미드 — XCTest/Espresso 경험의 이식

| 층 | 도구 | 네이티브 대응 | 무엇을 테스트 |
|---|---|---|---|
| 단위 | Jest | XCTest / JUnit | 순수 로직: 스토어, 유틸, 리듀서, 포맷터. 가장 많이, 가장 빠르게 |
| 컴포넌트 | React Native Testing Library (RNTL) | Compose UI Test에 가까움 | "사용자가 보는 것" 기준: 렌더 → 쿼리 → 상호작용 → 단언. 구현 세부(내부 state) 검증 금지 |
| E2E | Maestro | XCUITest / Espresso | 핵심 플로우 소수: 로그인, 결제, 온보딩. YAML로 선언적 작성 |

- RNTL의 철학은 Espresso와 같다: `getByText`, `getByRole`처럼 **사용자 관점 쿼리**를 쓴다. `testID`는 최후 수단.
- Jest에서 네이티브 모듈은 mock이 필요하다. Expo/RN이 제공하는 preset(`jest-expo` 등)이 대부분 처리하지만, 커스텀 네이티브 모듈은 팀이 mock을 유지해야 한다.
- E2E 도구로 Detox도 있으나, 신규 팀은 셋업 비용이 훨씬 낮은 Maestro로 시작하는 것이 일반적 권장. E2E는 flaky해지기 쉬우므로 개수를 엄격히 제한하고 "깨지면 즉시 고치거나 삭제" 규칙을 둔다.
- 피라미드 비율 감각: 단위 다수 > 컴포넌트 적당히 > E2E 한 자릿수~십수 개.

### CI/CD — 네이티브 변경과 JS-only 변경의 분리

RN 팀 CI의 핵심 통찰: **JS만 바뀐 커밋은 네이티브 재빌드가 필요 없다.**

- 앱 바이너리는 (네이티브 코드 + 네이티브 의존성 + 설정)의 함수다. 이 조합이 같으면 기존 빌드에 새 JS [[Bundle]]만 얹어 테스트할 수 있다 — [[OTA Update]]와 EAS의 development build 재사용이 이 원리 위에 있다.
- 실무 구성 예:
  - **모든 PR**: lint + typecheck + Jest + RNTL (수 분, 시뮬레이터 불필요)
  - **네이티브에 영향 주는 변경만**: [[EAS]] Build 트리거. `package.json`의 네이티브 의존성, `app.json`, config plugin, `ios/`·`android/` 변경 감지로 판단. EAS의 fingerprint 개념이 이 판단을 자동화한다(정확한 동작은 공식 문서 확인)
  - **머지 후**: internal distribution 빌드 배포(TestFlight/내부 트랙), E2E는 nightly
- 브랜치 전략은 팀 취향(trunk-based든 GitFlow든)이지만, RN 특유의 축은 **"이 브랜치의 JS가 어떤 runtime version의 바이너리 위에서 도는가"**를 항상 답할 수 있어야 한다는 것([[01-릴리즈-파이프라인]]의 버전 3축 참고).

### 코드 오너십 — 네이티브/JS 경계

특히 브라운필드(기존 네이티브 앱에 RN 통합) 팀에서 중요:

- **경계를 명시한다**: `ios/`·`android/`·config plugins·네이티브 모듈은 네이티브 담당, `src/`는 JS 담당 — CODEOWNERS 파일로 리뷰어 자동 지정.
- 네이티브 모듈의 **인터페이스(TypeScript 타입, [[New Architecture]]라면 Turbo Module spec)가 두 세계의 계약**이다. 계약 변경은 양쪽 리뷰 필수.
- 위험 신호: "네이티브 쪽은 아무도 안 봐서 그대로 뒀어요" — 업그레이드 때 폭탄이 된다. 최소 한 명은 네이티브 diff를 읽을 수 있어야 하고, 그게 네이티브 출신 시니어의 팀 내 핵심 가치다.

### PR 리뷰 — RN 특유의 체크포인트

일반 리뷰 항목 외에 RN에서 추가로 보는 것:

1. **리렌더 유발 패턴**: 렌더 중 새 객체/배열/함수 생성해서 자식 props로 전달, context value 미-memoization, 스토어 전체 구독(셀렉터 없이). → [[Re-render]] · [[Memoization]]
2. **`useEffect` 의존성 배열**: 빠진 의존성(스테일 클로저), 매 렌더 바뀌는 의존성(무한 루프), 그리고 근본 질문 — "이거 effect가 필요한 코드인가?" (파생 상태는 렌더 중 계산이 원칙)
3. **네이티브 diff**: prebuild 산출물이나 `Podfile.lock`·`gradle` 변경이 섞여 있으면 의도한 변경인지 확인. "왜 이 PR에 네이티브 diff가 있지?"는 항상 물어볼 가치가 있는 질문.
4. **의존성 추가**: 새 npm 패키지, 특히 네이티브 코드를 포함한 패키지는 업그레이드 비용·유지보수 상태(마지막 릴리즈, New Architecture 지원)를 리뷰에서 확인.
5. **리스트 렌더링**: [[FlatList]]에 인라인 renderItem, key 불안정, 아이템 미-memo.
6. **애니메이션이 JS 스레드를 타는지**: 제스처·애니메이션은 [[Reanimated]]로 UI 스레드에서 — JS 스레드 애니메이션은 부하 시 뚝뚝 끊긴다.

## 코드/설정 예시

CI 게이트 (GitHub Actions, JS-only 검증 파이프라인):

```yaml
# .github/workflows/pr.yml
name: PR Checks
on: pull_request
jobs:
  verify:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 22, cache: npm }
      - run: npm ci
      - run: npx tsc --noEmit
      - run: npx eslint .
      - run: npx prettier --check .
      - run: npx jest --ci
```

Maestro E2E — 선언적 YAML (XCUITest 코드와 비교하면 진입 장벽이 체감된다):

```yaml
# .maestro/login.yaml
appId: com.myteam.myapp
---
- launchApp
- tapOn: "이메일"
- inputText: "qa@example.com"
- tapOn: "비밀번호"
- inputText: "test1234"
- tapOn: "로그인"
- assertVisible: "홈"
```

RNTL 컴포넌트 테스트 — 사용자 관점 단언:

```typescript
import { render, screen, fireEvent } from '@testing-library/react-native';

test('기록 추가 시 목록에 나타난다', () => {
  render(<RecordScreen />);
  fireEvent.changeText(screen.getByPlaceholderText('제목'), '아침 러닝');
  fireEvent.press(screen.getByText('추가'));
  expect(screen.getByText('아침 러닝')).toBeOnTheScreen();
});
```

### 온보딩 — 네이티브 개발자를 RN 팀에 합류시킬 때

- 첫 주 목표는 "JS 문법"이 아니라 **[[Re-render]] 멘탈 모델**이다. 선언형 UI 경험(SwiftUI/Compose)이 있으면 절반은 끝나 있다.
- 짝 리뷰: 처음 한 달은 네이티브 출신의 PR을 JS 시니어가, JS 출신의 네이티브 diff를 네이티브 시니어가 교차 리뷰.
- vault의 [[01-간단한-앱]] → 동작 방식 노트 → 이 노트 순서가 온보딩 커리큘럼으로 그대로 쓰인다.

## 함정 (Pitfalls)

- **린트/타입 게이트를 "나중에" 켜기.** 위반이 수백 건 쌓인 뒤에는 아무도 못 켠다. 저장소 첫 주에 켜는 것이 유일한 타이밍.
- **E2E 과투자.** 네이티브에서 UI 테스트가 flaky했던 기억 그대로다. 적게, 핵심만, 깨지면 즉시 조치.
- **`exhaustive-deps`를 eslint-disable로 침묵시키는 문화.** 그 주석 하나하나가 잠재 스테일-클로저 버그다. disable에는 사유 주석을 강제.
- **JS 개발자만으로 팀 구성.** 업그레이드·크래시·빌드 문제는 결국 네이티브 층으로 내려간다. 경계 담당 없이 운영하면 "빌드가 안 되는데 아무도 이유를 모르는" 날이 온다.
- **네이티브 변경과 JS 변경을 한 PR에 섞기.** 리뷰 난이도와 롤백 난이도가 동시에 올라간다. 분리 가능한 것은 분리.
- **snapshot 테스트를 컴포넌트 테스트로 착각.** 거대 스냅샷은 아무도 안 읽고 `--u`로 갱신만 된다. 의미 있는 단언으로 대체.
- **PR마다 풀 네이티브 빌드.** CI 시간과 비용이 폭발한다. JS-only 경로를 먼저 설계할 것.
