# 버전 업그레이드 전략

> [!abstract] 한 줄 요약
> RN 생태계 최대 고통 포인트인 버전 업그레이드 — 왜 힘든지 구조를 이해하면, Expo SDK 단위 + CNG가 왜 정답에 가까운지 보인다.

## 🔁 iOS/AOS 대응 개념

| RN | iOS | Android |
|---|---|---|
| RN 코어 업그레이드 | Xcode + iOS SDK 메이저 업그레이드 | AGP + compileSdk 업그레이드 |
| 네이티브 의존성 호환성 | CocoaPods/SPM 의존성 충돌 | Gradle 의존성 지옥 |
| Expo SDK 업그레이드 | 대응물 없음 — "검증된 조합 세트"라는 개념 자체가 새로움 | 대응물 없음 |
| [[Prebuild]] 재생성 | 프로젝트 파일 마이그레이션을 자동화한 것에 해당 | 동일 |

네이티브에서 Xcode 메이저 업그레이드는 1년에 한 번, 대체로 하루 이틀 작업이다. RN에서는 그 규모의 이벤트가 **더 자주, 더 많은 축이 얽힌 채** 온다. 이 차이를 인정하는 것이 전략의 출발점이다.

## 📌 핵심 내용

### 왜 힘든가 — 세 층이 동시에 움직인다

RN 앱 하나의 버전 의존 구조:

```
RN 코어 (0.7x — 자주 릴리즈)
├── React 버전 (코어와 결합)
├── JS 툴체인: Metro, Babel, TypeScript
├── 네이티브 빌드 도구: Xcode 최소 버전, AGP, Kotlin, NDK, CocoaPods
└── 수십 개 네이티브 라이브러리
      └── 각각이 "지원하는 RN 버전 범위"를 따로 가짐
```

고통의 구조적 원인:

1. **RN 코어는 네이티브 API 표면과 결합되어 있다.** 코어를 올리면 네이티브 모듈들이 사용하는 내부 API가 바뀔 수 있고, 각 라이브러리가 새 버전을 지원할 때까지 기다리거나 패치해야 한다.
2. **N개의 네이티브 의존성 = N개의 호환 매트릭스.** 라이브러리 A는 RN 0.76부터, B는 0.74까지만 — 이런 교착이 실제로 발생한다. [[New Architecture]] 전환기(0.76에서 기본값 됨)에는 "New Arch 지원 여부"라는 축이 하나 더 있었다.
3. **빌드 도구 연쇄.** RN 버전이 요구하는 Xcode/AGP 최소 버전이 올라가면, 그게 다시 다른 네이티브 설정을 건드린다.
4. **템플릿 diff.** RN 업그레이드는 npm 패키지 교체가 아니라 **네이티브 프로젝트 템플릿의 변경을 내 프로젝트에 반영하는 일**이다. `ios/`·`android/`를 직접 관리하는 프로젝트(bare)는 이 diff를 손으로 머지해야 한다.

### Expo SDK 단위 업그레이드가 고통을 줄이는 이유

Expo SDK는 "RN 코어 + Expo 모듈 수십 개 + 빌드 도구 버전"의 **검증된 조합 스냅샷**이다.

- 위 호환 매트릭스 검증을 **Expo 팀이 대신 수행**한다. SDK 57이면 그 안의 RN 버전, 각 expo-* 모듈 버전, Xcode/AGP 요구사항이 함께 테스트된 상태.
- `npx expo install`은 현재 SDK와 호환되는 버전을 골라 설치하고, `npx expo install --check`(및 expo-doctor)는 어긋난 버전을 진단·교정한다. "이 라이브러리 버전이 내 RN과 호환되나"라는 조사 작업이 도구로 내려간다.
- 업그레이드가 "RN 코어 + 의존성 50개를 각각 조사"에서 "**SDK 한 단계 올리고 어긋난 것 교정**"으로 바뀐다.
- 물론 Expo 외부의 서드파티 네이티브 라이브러리는 여전히 직접 확인해야 한다 — 다만 그 수를 줄이는 것 자체가 전략이다([[01-간단한-앱]]의 "의존성 수 = 관리 지표").

### CNG/[[Prebuild]] 프로젝트가 업그레이드에 유리한 이유

[[CNG]](Continuous Native Generation): `ios/`·`android/`를 git으로 관리하지 않고, `app.json` + [[Config Plugin]]에서 **매번 생성**하는 방식.

- 업그레이드의 최대 난관인 "네이티브 프로젝트 템플릿 diff 수동 머지"가 **사라진다.** 새 SDK의 템플릿으로 prebuild를 다시 돌리면 네이티브 프로젝트가 새 기준으로 재생성되고, 내 커스텀은 Config Plugin으로 그 위에 재적용된다.
- 비유: 네이티브 프로젝트를 "소스"가 아니라 "빌드 산출물"로 취급하는 것. `.xcodeproj`를 손으로 고치는 대신 생성 규칙을 고친다.
- bare 프로젝트(네이티브 디렉토리 직접 관리)는 반대로 upgrade helper의 diff를 손으로 적용해야 한다. 브라운필드는 어쩔 수 없이 이 길이지만, 그린필드에서 bare를 선택할 이유는 계속 줄고 있다.
- 전제 조건: **네이티브 커스텀을 전부 Config Plugin으로 표현해 둘 것.** `ios/` 파일을 한 번이라도 직접 고쳐서 커밋하면 CNG 계약이 깨진다.

### 업그레이드 주기 전략

| 전략 | 내용 | 평가 |
|---|---|---|
| **매 SDK 추종** | 새 SDK 안정화(초기 패치 몇 주) 후 바로 업그레이드 | 회당 비용 최소·예측 가능. 여력 있는 팀의 최선 |
| **격 SDK** | 한 버전 건너뛰며 업그레이드 | 빈도는 줄지만 회당 diff 2배. 현실적 절충 |
| **필요할 때만** | OS 대응·라이브러리 요구 등 강제될 때만 | 최악. 아래 "함정" 참고 |

- 강제 마감이 존재한다: 새 iOS/Android 메이저 대응, 스토어의 targetSdk 요구, Expo SDK의 지원 종료, 그리고 EAS 빌드 환경의 Xcode 버전. "안 올리기"는 선택지가 아니라 유예일 뿐이다.
- 실무 리듬 권장: SDK 릴리즈 → 체인지로그와 breaking changes 훑기 → `.x` 패치 몇 개 나올 때까지 대기 → 전용 브랜치에서 업그레이드 → 전 플로우 회귀 테스트(E2E 있으면 여기서 빛남) → 릴리즈 하나를 업그레이드 전용으로.

### 도구

- **`npx expo install expo@^58 && npx expo install --fix`** 류의 SDK 업그레이드 커맨드 + 각 SDK의 공식 업그레이드 가이드(체인지로그)가 1차 자료.
- **expo-doctor**: 버전 불일치·설정 문제 진단.
- **React Native Upgrade Helper** (react-native-community): 두 RN 버전 간 템플릿 diff를 보여주는 웹 도구. bare 프로젝트의 필수품이고, CNG 프로젝트에서도 "내부에서 뭐가 바뀌었나" 파악용으로 유용.
- **EAS fingerprint**: 업그레이드 후 네이티브 지문이 바뀌므로 새 빌드·새 runtime version이 필요함을 기계적으로 알 수 있다([[01-릴리즈-파이프라인]]의 버전 3축).

## 💻 코드/설정 예시

SDK 업그레이드의 실제 절차 (Expo SDK 57 기준, 세부는 각 SDK 공식 가이드 확인):

```bash
# 0. 전용 브랜치
git switch -c upgrade/sdk-58

# 1. SDK 코어 업그레이드
npx expo install expo@^58.0.0

# 2. SDK 호환 버전으로 의존성 일괄 교정
npx expo install --fix

# 3. 진단
npx expo-doctor

# 4. (CNG) 네이티브 프로젝트 재생성 — 손으로 머지할 diff가 없다
npx expo prebuild --clean

# 5. 로컬 빌드로 검증 후, 회귀 테스트
npx expo run:ios
npx expo run:android
```

Config Plugin으로 커스텀을 유지하는 것이 왜 전제 조건인지 — 안티패턴 대비:

```
❌ ios/MyApp/Info.plist를 직접 수정 후 커밋
   → prebuild --clean 하는 순간 소실. 업그레이드 때마다 수동 재적용 + 잊으면 버그

✅ app.json의 plugins에 선언 (또는 커스텀 플러그인)
   → 어떤 SDK에서 재생성해도 자동 재적용
```

```json
// app.json — 네이티브 커스텀의 단일 소스
{
  "expo": {
    "plugins": [
      ["expo-image-picker", { "photosPermission": "기록에 첨부할 사진을 선택합니다." }],
      "./plugins/withMyCustomEntitlement"
    ]
  }
}
```

### 업그레이드 회귀 테스트 — 무엇을 확인하나

업그레이드 후 회귀는 기능 로직보다 **경계 지점**에서 난다. 우선순위:

1. **네이티브 모듈을 쓰는 모든 플로우**: 카메라, 사진, 파일, 푸시, 결제, 지도 — 라이브러리 버전이 함께 움직인 곳
2. **시작 시퀀스**: 스플래시 → 첫 화면, 딥링크 콜드 스타트
3. **내비게이션 전환·제스처**: [[React Navigation]]/[[Expo Router]]와 screens 라이브러리가 자주 함께 갱신됨
4. **양 플랫폼 모두**: 한쪽만 확인하고 출시하는 것이 전형적 사고. 특히 Android 저사양 실기기
5. **release 빌드로**: dev에서만 확인하면 [[Hermes]] 바이트코드·minify 경로가 미검증으로 남는다

E2E(Maestro)가 있다면 이 목록의 상당 부분이 자동화된다 — 업그레이드 전략과 테스트 투자([[03-팀-협업]])는 사실상 한 세트다.

## ⚠️ 함정 (Pitfalls)

- **미루다 한 번에 여러 버전 점프 — 최대의 함정.** SDK 3개를 건너뛰면 breaking change가 곱해질 뿐 아니라, **중간 버전을 경유해야 하는 마이그레이션**(예: New Architecture 전환 같은 큰 전환점)이 겹쳐서 난이도가 비선형으로 뛴다. 실패 사례의 전형: "2년 묵힌 앱, 업그레이드 견적 6주, 결국 재작성 논의". 작게 자주가 유일한 해법이다.
- **`ios/`·`android/` 직접 수정으로 CNG 계약 파기.** 급해서 Xcode에서 한 줄 고치고 커밋 → 다음 prebuild에서 증발하거나, CNG를 포기하고 bare로 전락. 급한 수정도 반드시 Config Plugin으로 회수할 것.
- **npm 시맨틱 버저닝 신뢰.** 네이티브 코드가 포함된 패키지는 patch 버전에서도 네이티브 빌드가 깨질 수 있다. lockfile을 커밋하고, 의존성 갱신은 업그레이드 이벤트로 취급.
- **업그레이드와 기능 개발을 같은 브랜치/릴리즈에.** 회귀가 나왔을 때 원인 분리가 안 된다. 업그레이드는 단독 릴리즈로.
- **서드파티 라이브러리의 유지보수 상태 미확인.** 마지막 커밋 2년 전인 네이티브 라이브러리는 다음 업그레이드의 지뢰다. 도입 시점에 대안·포크 가능성까지 평가([[03-팀-협업]]의 PR 리뷰 항목).
- **업그레이드 직후 OTA 발행.** runtime version이 바뀌었는지 확인 없이 내보내면 구 바이너리와의 호환성 사고. [[01-릴리즈-파이프라인]]의 3축 점검 먼저.
- **"Expo니까 자동이겠지."** SDK 업그레이드가 줄여주는 것은 조사·머지 비용이지 검증 비용이 아니다. 회귀 테스트는 온전히 남는다.
- **베타/RC SDK를 프로덕션 앱에.** 새 SDK 릴리즈 직후에는 생태계 라이브러리들이 아직 따라오지 못한 상태다. 안정 릴리즈 + 초기 패치 대기가 기본.
- **업그레이드 작업량 과소 견적.** "패치 버전인데요"가 이틀이 되는 생태계다. 계획에는 항상 검증·트러블슈팅 버퍼를 포함할 것.

- **캐시 오염을 코드 문제로 오진.** 업그레이드 직후의 기괴한 빌드 에러는 절반이 캐시다. Pods·Gradle·[[Metro]] 캐시·`node_modules`를 전부 지우고 클린 빌드부터 — 그 다음에 코드를 의심한다.
- **lockfile 미커밋 상태로 업그레이드.** 팀원마다 다른 버전이 설치되어 "내 머신에서는 되는데"가 양산된다. `package-lock.json`(또는 pnpm/yarn lockfile)은 항상 커밋.
