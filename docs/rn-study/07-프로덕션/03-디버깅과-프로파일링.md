# 디버깅과 프로파일링

> [!abstract] 한 줄 요약
> 네이티브 개발자가 RN에서 가장 낯설어하는 영역 — 핵심은 "어느 층의 문제인가"를 먼저 판정하고, JS 층은 React Native DevTools, 네이티브 층은 아는 도구 그대로 쓰는 것.

## 🔁 iOS/AOS 대응 개념

| RN | iOS | Android |
|---|---|---|
| React Native DevTools (콘솔·브레이크포인트) | Xcode 디버거 + lldb | Android Studio 디버거 |
| React DevTools Profiler ([[Re-render]] 추적) | SwiftUI의 `Self._printChanges()` / Instruments SwiftUI 템플릿 | Compose Layout Inspector의 recomposition counts |
| [[Hermes]] CPU 프로파일 | Instruments Time Profiler | Android Studio CPU Profiler |
| React 컴포넌트 트리 검사 | Xcode View Hierarchy Debugger | Layout Inspector |
| Network 패널 | Charles/Proxyman/Instruments | Network Profiler |
| `console.log` | `print`/`os_log` | `Log.d` |

**Flipper는 deprecated다.** 검색하면 아직 많이 나오지만 새로 배우지 말 것. RN 0.76+의 공식 도구는 **React Native DevTools**.

## 📌 핵심 내용

### 0단계 — 어느 층의 문제인지 먼저 판정한다

RN 디버깅이 낯선 이유는 도구가 아니라 **층이 두 개**라서다. 잘못된 층에서 몇 시간을 태우는 것이 최악의 시나리오이므로, 증상으로 층을 먼저 가른다:

| 증상 | 층 | 도구 |
|---|---|---|
| Red Box (dev) / JS 에러 화면, 앱 프로세스는 생존 | JS | React Native DevTools |
| 프로세스 즉사, OS 크래시 리포트 생성 | 네이티브 | Xcode Organizer / Play Console 크래시, lldb — **기존 지식 그대로** |
| 크래시 스택에 `hermes::` / `facebook::jsi` 프레임 | 경계 (JS 값이 네이티브를 통과하다 터짐) | 양쪽: 네이티브 스택으로 지점 특정 + JS 쪽 호출 재현 |
| UI가 굳음, 터치 무반응, 그러나 죽지 않음 | 대개 JS 스레드 블로킹 | 아래 매핑표 |
| 빌드가 안 됨 | 네이티브 빌드 시스템 | Xcode/Gradle 로그 — 기존 지식 그대로 |

크래시 로그 읽는 법(스레드 덤프, 시그널, 심볼리케이션)은 네이티브 지식이 100% 그대로다. RN이 추가하는 것은 "JS 스택은 source map으로 따로 심볼화한다"([[01-릴리즈-파이프라인]])는 점뿐.

### React Native DevTools — JS 층의 주력 도구

RN 0.76+에서 기본 제공. dev 빌드 실행 중 터미널에서 `j`를 누르면 열린다(Chrome DevTools 기반 UI). Hermes와 CDP(Chrome DevTools Protocol)로 직접 통신하므로 과거의 "Remote JS Debugging"(JS를 Chrome에서 실행하던 이상한 방식)과 달리 **실제 기기의 실제 엔진을 디버깅한다**.

구성 요소:

- **Console**: `console.log` 출력 확인, 객체 인스펙션.
- **Sources / 브레이크포인트**: TS 원본 소스에 브레이크포인트, step-in/out, watch — lldb에서 하던 그것. `debugger;` 문도 동작.
- **Network 패널**: `fetch`/XHR 요청 확인. (네이티브 층에서 나가는 요청은 안 보임 — 그건 Charles/Proxyman으로.)
- **React DevTools (Components / Profiler 탭)**: 컴포넌트 트리 검사(props/[[State]] 실시간 확인·수정)와 렌더 프로파일링.

한계도 알아둘 것: dev 빌드 전용이다. release 빌드에는 붙일 수 없으므로, 프로덕션 이슈는 Sentry 등 원격 수집 + 재현으로 접근한다.

### 네이티브 디버거와 병행하는 법

RN 프로젝트도 결국 Xcode/Android Studio 프로젝트다([[Prebuild]] 산출물을 열면 된다):

- **네이티브 브레이크포인트**: 커스텀 네이티브 모듈, 서드파티 라이브러리의 네이티브 코드에 Xcode/AS 디버거로 브레이크포인트 — 평소처럼.
- **동시 디버깅**: 시뮬레이터에서 앱을 Xcode로 실행해 둔 채 React Native DevTools를 병행하면 JS 브레이크포인트와 네이티브 브레이크포인트를 같이 걸 수 있다. JS→네이티브 호출을 추적할 때 유용.
- **JS 예외로 인한 크래시**: release에서 처리 안 된 JS 에러는 네이티브 예외로 전환되어 프로세스를 죽인다. 크래시 리포트에 `RCTFatal` 등 RN 런타임 프레임이 보이면 "네이티브 크래시처럼 보이는 JS 문제"다 — source map으로 JS 스택을 찾는 쪽이 본질.

### 성능 프로파일링

**JS/React 층:**

- **React DevTools Profiler**: 녹화 → 상호작용 → 정지. 커밋별 flame graph에서 어떤 컴포넌트가 렌더됐는지, "왜 렌더됐는지"(props 변경/부모 렌더/state 변경)를 본다. [[Re-render]] 폭풍 진단의 1번 도구.
- **Hermes CPU 프로파일**: React Native DevTools의 Performance 패널 또는 CLI로 샘플링 프로파일 수집 — "JS 스레드에서 어떤 함수가 시간을 먹는가". Instruments Time Profiler의 JS판.
- **Perf Monitor** (dev 메뉴): JS 스레드 / UI 스레드 FPS를 오버레이로 상시 표시. "어느 스레드가 떨어지는가"의 1차 신호. UI FPS만 하락 → 네이티브 렌더 부하, JS FPS 하락 → JS 층 문제.

**네이티브 층 — 기존 도구가 그대로 유효하다:**

- Instruments (Time Profiler, Allocations, Core Animation), Android Studio Profiler, systrace/Perfetto 전부 RN 앱에 그대로 쓴다. 시작 시간, 메모리, 네이티브 모듈 병목은 이쪽이 정답.
- RN 앱의 스레드 구조를 알고 읽으면 된다: 메인(UI) 스레드 / JS 스레드 (+ 렌더 관련 백그라운드 스레드들). 프로파일에서 JS 스레드가 100%로 고정되어 있으면 그 원인은 JS 층으로 넘어가서 Hermes 프로파일로 찾는다.

### 로그 — `console.log`는 어디로 가는가

- **dev**: Metro 터미널과 React Native DevTools Console 양쪽에 출력.
- **release**: 기본적으로 **어디에도 안 간다고 취급하라.** (Babel 플러그인으로 제거되기도 하고, 남더라도 수집 수단이 없다.) 프로덕션 관찰은 Sentry breadcrumb, 원격 로깅으로 설계한다.
- 네이티브 모듈 안의 `os_log`/`Log.d`는 평소처럼 Console.app/logcat으로 — JS 로그와 네이티브 로그가 **다른 곳에 쌓인다**는 것만 기억.
- 팁: 시뮬레이터/기기의 OS 레벨 로그(logcat 등)에도 JS 콘솔이 섞여 나오는 경우가 있으나 보장된 계약은 아니다. 의존하지 말 것.

### 흔한 증상 → 원인 매핑표

| 증상 | 유력 원인 | 확인 방법 |
|---|---|---|
| 버튼 탭 후 반응이 늦음, 전체적으로 굼뜸 | **JS 스레드 블로킹** — 무거운 동기 연산(파싱, 정렬, 큰 JSON)이 이벤트 루프 점유 | Perf Monitor에서 JS FPS 급락 → Hermes 프로파일로 함수 특정 |
| 스크롤은 부드러운데 내용 갱신이 늦음 | 위와 동일 (스크롤은 네이티브가 처리, 갱신은 JS 몫) | 동일 |
| 타이핑마다 화면 전체가 버벅 | **리렌더 폭풍** — 입력 state가 상위에 있어 트리 전체 렌더 | React DevTools Profiler에서 커밋 범위 확인 |
| 리스트 스크롤 시 빈 셀/프레임 드랍 | [[FlatList]] 아이템 렌더 비용 과다, 미-[[Memoization]] | Profiler + [[02-복잡한-앱]]의 리스트 튜닝 |
| 애니메이션이 부하 시 뚝뚝 끊김 | 애니메이션이 JS 스레드에서 구동됨 | [[Reanimated]]/native driver로 UI 스레드 이전 |
| 잦은 대량 상태 업데이트 시 전반적 지연 | 렌더/직렬화 부하 (구세대 브릿지라면 "브릿지 홍수") — 이벤트를 너무 잘게 자주 보냄 | 업데이트 배칭·스로틀. New Architecture([[New Architecture]])에서 완화되었으나 공짜는 아님 |
| 시작이 느림 | 초기 [[Bundle]] 실행량 과다 or 네이티브 초기화 | 네이티브 구간은 Instruments, JS 구간은 Hermes 프로파일 |
| 메모리 지속 증가 | JS 클로저/리스너 누수 or 네이티브 누수 | DevTools 메모리 계열 + Instruments Allocations 병행 |

## 💻 코드/설정 예시

디버깅 진입 (RN 0.76+ / Expo SDK 57):

```bash
npx expo start
# 터미널에서:
#   j → React Native DevTools 열기
#   기기에서 흔들기(or Cmd+D / Cmd+M) → Dev Menu → Perf Monitor 토글
```

JS 스레드 블로킹의 전형과 처방:

```typescript
// ❌ 탭 핸들러에서 동기로 무거운 일 → 이 동안 모든 터치·렌더 정지
const onPress = () => {
  const result = heavyParse(bigJsonString); // 수백 ms
  setResult(result);
};

// ✅ 1차: 상호작용 뒤로 미루기
import { InteractionManager } from 'react-native';
const onPress = () => {
  InteractionManager.runAfterInteractions(() => setResult(heavyParse(bigJsonString)));
};
// ✅ 근본: 연산 자체를 줄이거나(서버/사전 계산), 네이티브 모듈·워커로 이전
```

`debugger` 문 — lldb의 breakpoint와 같은 감각:

```typescript
function suspiciousFn(input: Item[]) {
  debugger; // React Native DevTools가 연결되어 있으면 여기서 정지
  return input.filter(...);
}
```

### 프로덕션 이슈 대응 절차 (요약)

dev 도구가 붙지 않는 release 이슈의 표준 루트:

1. **Sentry에서 층 확인**: JS 에러(소스맵 스택)인가, 네이티브 크래시(dSYM/mapping 스택)인가
2. **JS 에러** → 스택의 원본 코드 확인 → dev 빌드에서 재현 → React Native DevTools로 원인 → 수정 → JS-only면 [[OTA Update]]로 즉시 배포 가능
3. **네이티브 크래시** → 기존 크래시 분석 절차 그대로 (시그널, 스레드, 라이브러리 버전) → 수정은 스토어 재배포
4. **재현 안 되는 성능 제보** → 기기 모델·OS 분포 확인 → 저사양 실기기 + release 빌드로 프로파일링

## ⚠️ 함정 (Pitfalls)

- **Flipper 기준의 낡은 자료를 따라감.** 2023년 이전 블로그의 디버깅 세팅은 대부분 무효다. React Native DevTools 기준 공식 문서로.
- **층 판정 없이 도구부터 잡음.** 네이티브 크래시를 JS 콘솔에서 찾거나, 리렌더 문제를 Instruments에서 찾으며 시간 소모. 위의 0단계 표부터.
- **dev 빌드 성능으로 성능을 판단.** dev 모드는 각종 검증 코드로 몇 배 느리다. **성능 측정은 반드시 release 빌드에서.** 반대로 "release에서만 나는 버그"는 dev 도구가 안 붙으므로 Sentry+재현 전략 필요.
- **`console.log`가 프로덕션 관찰 수단이라고 가정.** release에서는 수집되지 않는다. 관찰 가능성은 별도로 설계.
- **`console.log(hugeObject)`를 루프에서 남발.** dev에서 직렬화 비용이 커서 그 자체가 성능 문제를 만든다 — 프로파일링 중이라면 결과를 오염시킨다.
- **JS 브레이크포인트에 걸린 동안 앱이 통째로 멈춘 것을 버그로 착각.** JS 스레드가 정지하면 RN UI 갱신도 정지한다 — 정상이다.
- **시뮬레이터에서만 프로파일링.** 시뮬레이터는 아키텍처·GPU·스로틀링이 다르다. 성능 수치는 실기기, 가급적 저사양 Android 기기 기준.
- **한 번 측정으로 결론.** JS 엔진 워밍업·GC·기기 온도로 편차가 크다. 최적화 전후 비교는 같은 조건에서 여러 번 측정한 값으로.

- **[[Metro]] 캐시를 잊음.** "고쳤는데 그대로예요"의 일부는 번들 캐시다. `npx expo start --clear`로 캐시 초기화가 디버깅 전 기본 위생.
- **에러 메시지의 층을 오독.** "Unable to resolve module"은 Metro(번들링) 문제, "Native module cannot be null"은 네이티브 링킹([[Autolinking]]/재빌드 필요) 문제 — 둘 다 코드 로직 문제가 아니다.
