# 보안과 접근성

> [!abstract] 한 줄 요약
> 보안: 저장 위치와 번들의 가독성만 이해하면 나머지는 네이티브 지식 그대로. 접근성: VoiceOver/TalkBack 경험이 거의 1:1로 이전되는, 네이티브 시니어가 팀에서 가장 잘할 수 있는 영역.

## 🔁 iOS/AOS 대응 개념

| RN | iOS | Android |
|---|---|---|
| expo-secure-store | Keychain | Keystore(EncryptedSharedPreferences 계열) |
| AsyncStorage | UserDefaults급 (평문, 비민감 데이터용) | SharedPreferences급 |
| JS [[Bundle]] 노출 | 바이너리 리버싱보다 쉬움 (아래 설명) | 동일 |
| 인증서 피닝 | URLSession 피닝 / ATS | Network Security Config |
| accessibilityLabel | accessibilityLabel (이름까지 같음) | contentDescription |
| accessibilityRole | accessibilityTraits | className / role 의미 |
| accessibilityState | accessibilityTraits(.selected 등) + value | stateDescription / isChecked 등 |
| 스크린리더 | VoiceOver | TalkBack |

## 📌 핵심 내용

## 보안

### 저장: 토큰은 Keychain/Keystore로 — RN이라고 달라지지 않는다

- **AsyncStorage는 평문이다.** iOS에서는 앱 샌드박스 내 파일, Android에서는 앱 데이터 디렉토리의 DB에 암호화 없이 저장된다. UserDefaults/SharedPreferences에 토큰을 안 넣던 그 감각 그대로 — **세션 토큰, 리프레시 토큰, 개인정보는 금지.**
- 올바른 곳: **expo-secure-store** — iOS Keychain, Android Keystore 기반 암호화 저장을 하나의 JS API로 감싼 것. (bare 프로젝트에서는 react-native-keychain 등 대안.)
- Keychain 접근성 클래스(잠금 해제 시에만 등), Keystore의 하드웨어 보호 같은 세부 지식은 그대로 유효하며, expo-secure-store 옵션으로 일부 노출된다 — 정확한 옵션은 공식 문서 확인.
- 크기 제한이 있다(SecureStore는 작은 값 용도, iOS 기준 수 KB 수준). 큰 데이터를 암호화해야 하면 "키는 SecureStore, 데이터는 그 키로 암호화해 파일/DB에" 패턴.

### JS [[Bundle]]은 읽힌다 — 비밀키를 넣지 말 것

네이티브 바이너리도 리버싱되지만, JS 번들은 난이도가 한 단계 낮다:

- 릴리즈 번들은 [[Hermes]] 바이트코드지만, **바이트코드도 디컴파일·역해석 도구가 존재**하고, minify는 난독화이지 암호화가 아니다. **앱에 담긴 문자열은 공개된 것으로 간주**하라 — 네이티브에서 `strings` 명령으로 하드코딩 키 찾던 것과 같은 이야기.
- 따라서: API secret, 서명 키, 관리자 토큰은 번들에 절대 넣지 않는다. 서버가 대신 호출하는 구조(BFF)로 옮긴다.
- **`EXPO_PUBLIC_` 접두사 환경 변수는 이름 그대로 public이다** — 번들에 문자열로 박제된다. "환경 변수니까 안전하다"는 착각이 RN에서 흔한 사고. 빌드타임 비밀은 EAS Secrets 등으로 빌드 서버까지만 전달하고, 런타임 비밀은 클라이언트에 아예 존재하지 않게 설계.
- 클라이언트에 넣을 수밖에 없는 값(예: 공개 API 키)은 서버측 제약(도메인/앱 제한, rate limit)으로 방어 — 네이티브와 동일한 원칙.

### 네트워크: 인증서 피닝 개요

- 기본 전송 보안(ATS, cleartext 금지)은 OS 레벨 이야기라 그대로 적용된다.
- 피닝은 RN 층 표준 API가 없어서 **네이티브 층에서 구현**하는 것이 정석이다: iOS URLSession 피닝 / Android Network Security Config — 즉 기존 지식으로 하는 작업이며, Expo에서는 관련 설정을 [[Config Plugin]]으로 주입한다. 커뮤니티 라이브러리(react-native-ssl-pinning 등)도 있으나 유지보수 상태 확인 필수.
- 네이티브에서 알던 트레이드오프도 그대로: 핀 로테이션 실패 = 앱 전체 네트워크 마비. 백업 핀과 만료 전략 없이 도입하지 말 것.
- JS 층의 `fetch`는 결국 OS 네트워크 스택을 타므로, 네이티브 층 피닝이 JS 요청에도 적용된다.

### [[OTA Update]] 서명 — 코드 배포 채널의 무결성

OTA는 "스토어 서명 없이 코드가 앱에 들어오는 경로"이므로 그 자체가 공격 표면이다:

- 기본 방어는 HTTPS지만, **expo-updates는 코드 서명(code signing)을 지원**한다 — 업데이트 매니페스트를 개인키로 서명하고, 앱에 심은 인증서로 검증. 서버가 탈취되어도 서명 안 된 업데이트는 거부된다. 설정 방법은 공식 문서 확인.
- 운영 관점: 서명 키 관리가 새로 생기는 책임이다(앱 서명 키 관리하던 감각 그대로). 그리고 "누가 OTA를 발행할 수 있는가"의 권한 관리(EAS 계정/토큰)가 릴리즈 서명 권한만큼 중요해진다.

### 그 외 짧게

- 스크린샷 방지, 루팅/탈옥 탐지, 백그라운드 스냅샷 가리기 — 전부 네이티브 층 작업이며 기존 지식 그대로. RN 라이브러리는 그 래퍼일 뿐.
- 딥링크 파라미터 검증, WebView 하드닝도 네이티브와 동일한 원칙.

## 접근성

### 네이티브 지식이 거의 그대로 이전된다

RN의 접근성 API는 **iOS/Android 접근성 트리 위의 얇은 추상화**다. VoiceOver/TalkBack으로 검수하던 경험, "이 요소의 이름·역할·상태·값은 무엇인가"라는 프레임이 그대로 통한다. RN에서 props 몇 개만 매핑하면 된다:

| RN prop | 의미 | iOS 대응 | Android 대응 |
|---|---|---|---|
| `accessible` | 하위 뷰를 하나의 접근성 요소로 묶음 | isAccessibilityElement | focusable + 병합 |
| `accessibilityLabel` | 읽어줄 이름 | accessibilityLabel | contentDescription |
| `accessibilityRole` | `"button"`, `"header"`, `"image"`… | accessibilityTraits | 위젯 클래스/role 의미 |
| `accessibilityState` | `{ disabled, selected, checked, expanded, busy }` | traits + 상태 | stateDescription 등 |
| `accessibilityValue` | 슬라이더 등의 현재 값 | accessibilityValue | (범위 정보) |
| `accessibilityHint` | 동작 결과 힌트 | accessibilityHint | (유사 개념) |
| `accessibilityLiveRegion` (Android) / `AccessibilityInfo.announceForAccessibility` | 동적 변경 알림 | UIAccessibility notification | liveRegion |

- `Pressable`/`Button` 등 RN 기본 컴포넌트는 상당 부분 기본값을 채워주지만, **커스텀 터치 영역(제스처 기반, 아이콘 버튼)은 role과 label을 손으로 줘야 한다** — 네이티브에서 커스텀 UIControl 만들 때와 같다.
- 검수 도구도 그대로: VoiceOver/TalkBack 직접 구동, Accessibility Inspector, Android의 접근성 검사기. RN 전용 도구를 새로 배울 필요가 거의 없다.
- 시니어 네이티브 개발자의 팀 기여 포인트: JS 출신 동료들은 접근성 검수 경험이 없는 경우가 많다. **접근성 기준을 PR 리뷰 항목으로 만드는 것**([[03-팀-협업]])이 큰 레버리지다.

### RN에서 특히 신경 쓸 지점

- **터치 타깃 크기**: 44pt/48dp 가이드라인은 그대로. `hitSlop` prop으로 시각 크기를 키우지 않고 터치 영역만 확장 가능.
- **동적 글자 크기**: `Text`는 기본적으로 OS 글꼴 크기 설정을 따른다(`allowFontScaling` 기본 true). 끄지 말고, 커지는 텍스트를 수용하는 레이아웃으로 설계.
- **화면 전환 포커스**: 커스텀 내비게이션 UI에서는 새 화면 진입 시 스크린리더 포커스 이동을 확인. [[React Navigation]]/[[Expo Router]]는 기본 처리를 제공하는 편이나 커스텀 헤더·모달은 검수 필요.
- **reduce motion**: `AccessibilityInfo.isReduceMotionEnabled()`로 조회해 [[Reanimated]] 애니메이션을 절제 — 네이티브의 UIAccessibility.isReduceMotionEnabled 대응.

### 최소 검수 체크리스트

**보안 (릴리즈 전):**
- [ ] 번들에서 비밀 문자열 검색 — 산출물에 secret이 박혀 있지 않은지
- [ ] 토큰류가 전부 SecureStore 경유인지 (AsyncStorage 사용처 전수 확인)
- [ ] OTA 코드 서명 활성화 + 발행 권한자 목록 최신화
- [ ] cleartext 통신 없음 (ATS/Network Security Config 기본값 유지)

**접근성 (화면 추가 시):**
- [ ] VoiceOver/TalkBack으로 해당 화면 전체 탐색 1회
- [ ] 모든 인터랙티브 요소에 이름·역할·상태
- [ ] OS 글자 크기 최대로 올려서 레이아웃 확인
- [ ] 터치 타깃 44pt/48dp

## 💻 코드/설정 예시

토큰 저장 (Expo SDK 57):

```typescript
import * as SecureStore from 'expo-secure-store';

// ✅ 토큰은 Keychain/Keystore로
await SecureStore.setItemAsync('refresh_token', token);
const token = await SecureStore.getItemAsync('refresh_token');

// ❌ 이렇게 하면 평문 저장
// await AsyncStorage.setItem('refresh_token', token);
```

접근성 있는 커스텀 아이콘 버튼 — 네이티브 감각의 직역:

```tsx
<Pressable
  onPress={toggleFavorite}
  accessibilityRole="button"
  accessibilityLabel="즐겨찾기"
  accessibilityState={{ selected: isFavorite }}
  accessibilityHint="이 기록을 즐겨찾기에 추가하거나 제거합니다"
  hitSlop={8}
>
  <StarIcon filled={isFavorite} />
</Pressable>
// VoiceOver: "즐겨찾기, 선택됨, 버튼" — UIButton + traits와 같은 결과
```

여러 텍스트를 하나의 요소로 묶기 (셀 단위 낭독):

```tsx
<View accessible accessibilityLabel={`${title}, ${date}, ${duration}분`}>
  <Text>{title}</Text>
  <Text>{date}</Text>
  <Text>{duration}분</Text>
</View>
```

## ⚠️ 함정 (Pitfalls)

- **"환경 변수에 넣었으니 안전"** — `EXPO_PUBLIC_*`와 앱 코드에 들어간 모든 값은 번들에서 추출 가능하다. 클라이언트 비밀이라는 개념 자체가 성립하지 않는다.
- **AsyncStorage에 토큰 저장.** RN 튜토리얼 예제에 흔해서 그대로 복사되는 사고 유형 1위. 코드 리뷰에서 기계적으로 걸러낼 것.
- **Hermes 바이트코드를 암호화로 착각.** 읽기 어려워졌을 뿐 읽힌다. 위협 모델을 "번들은 공개"로 잡아라.
- **SecureStore에 대용량 저장 시도.** 크기 제한으로 실패하거나 플랫폼별 이상 동작. 키만 저장하고 데이터는 따로 암호화.
- **OTA 인프라 권한 방치.** EAS 토큰이 유출되면 서명 없는 체제에서는 전체 사용자에게 임의 JS를 배포당하는 것과 같다. 코드 서명 + 발행 권한 최소화.
- **접근성을 "라벨 달기"로 축소.** 이름만 있고 role/state가 없으면 "즐겨찾기"라고만 읽히고 버튼인지, 눌려 있는지 알 수 없다 — 네이티브에서 알던 그대로, 이름·역할·상태 3종 세트.
- **`accessibilityLabel`을 테스트 셀렉터로 오용.** E2E 편의로 라벨에 ID 문자열을 넣으면 스크린리더 사용자가 그 ID를 듣게 된다. 테스트 훅은 `testID`로.
- **`allowFontScaling={false}` 남발.** "디자인 깨져서 껐어요"는 접근성 버그를 디자인 버그로 바꾼 것뿐이다.
