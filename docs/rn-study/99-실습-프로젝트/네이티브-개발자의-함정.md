# 네이티브 개발자의 함정

> **살아있는 노트.** iOS/Android 감각을 그대로 RN에 가져올 때 미끄러지는 지점을 모은다.
> 형식: 증상 → 원인 → 올바른 멘탈모델. 실습([[README|실습 프로젝트]]) 중 걸린 것은 바로 여기에 추가.

---

## 1. useEffect를 viewDidLoad / onCreate처럼 쓴다

- **증상**: "화면 뜰 때 한 번"을 의도했는데 effect가 여러 번 돌거나, 의존성 배열 경고를 끄고 넘어갔다가 데이터가 갱신되지 않는다.
- **원인**: `useEffect`는 lifecycle 콜백이 아니라 **"렌더 결과를 외부 시스템과 동기화"**하는 도구다. 의존성이 바뀔 때마다 다시 실행되고, dev 모드(StrictMode)에선 마운트 시 두 번 돌 수도 있다.
- **멘탈모델**: "언제 실행되나"가 아니라 "**무엇과 동기화하나**"를 먼저 묻는다. 렌더로 표현 가능한 것(파생 값)은 effect가 아니라 본문 계산으로. 진짜 1회성 초기화는 앱 레벨(모듈 스코프·Provider)로 올린다. [[Hook]]

## 2. [[Re-render]]를 draw 호출로 오해한다

- **증상**: "렌더가 이렇게 자주 돌면 성능 다 죽는 것 아닌가?" 하고 조기 최적화하거나, 반대로 `setNeedsDisplay`/`invalidate()` 같은 갱신 API를 찾아 헤맨다.
- **원인**: [[Re-render]] = **컴포넌트 함수 재실행 → 새 JSX 트리 반환**일 뿐이다. React가 이전 트리와 diff(Reconciliation)해서 **바뀐 네이티브 뷰만** [[Fabric]]을 통해 갱신한다.
- **멘탈모델**: 렌더 함수는 "현재 [[State]]의 UI 서술"이고, 화면 갱신 명령이 아니다. 갱신 API는 없다 — state를 바꾸면 UI가 따라온다. 렌더 자체는 싸고, 비싼 계산이 본문에 있을 때만 [[Memoization]]을 고려한다.

## 3. 명목적 타이핑 감각으로 TS 구조적 타이핑에 놀란다

- **증상**: 선언한 적 없는 타입인데 함수에 그냥 들어간다. "이거 왜 컴파일되지?" 반대로 `protocol` 채택처럼 명시적 선언을 찾다가 헤맨다.
- **원인**: Swift/Kotlin/Java는 명목적(이름 기준), TypeScript는 **구조적(모양 기준)** 타이핑이다. 프로퍼티 구조가 맞으면 같은 타입으로 취급한다.
- **멘탈모델**: TS의 타입은 "혈통 증명"이 아니라 "**모양 검사**"다. `interface` 구현 선언 없이도 모양만 맞으면 통과. 덕분에 `Omit<StoredRecord,'id'|'sync'>` 같은 조합 타입이 자연스럽다(레포 `src/store/StoreContext.tsx` 액션 시그니처 참고).

## 4. "메인 스레드에서 다 된다"는 가정 → JS 스레드 블로킹

- **증상**: 버튼이 늦게 반응하고 화면 전환이 버벅이는데 CPU 프로파일러엔 메인(UI) 스레드가 한가하다.
- **원인**: RN의 앱 로직은 단일 **JS 스레드**에서 돈다. 무거운 동기 루프(JSON 파싱, 큰 배열 연산)가 돌면 터치 응답·[[React Navigation]] 전환이 전부 밀린다. UI 스레드는 멀쩡해도 앱은 멈춘 것처럼 보인다.
- **멘탈모델**: "메인 스레드 = UI 스레드 = 내 코드 스레드"라는 네이티브 등식을 버린다. RN은 **UI 스레드 / JS 스레드 / 백그라운드**가 분리. JS 스레드는 절대 오래 점유하지 않는다 — 애니메이션은 [[Reanimated]]의 [[Worklet]]처럼 UI 스레드에서 돌리고, 무거운 계산은 쪼개거나 네이티브([[Turbo Module]])로 내린다.

## 5. [[Stale Closure]] — 캡처 리스트 감각으로는 예상 못 하는 버그

- **증상**: `setInterval`·이벤트 리스너·비동기 콜백 안에서 읽은 state가 항상 **옛날 값**이다. Swift라면 `[weak self]`로 참조를 잡아 최신 프로퍼티를 읽었을 텐데.
- **원인**: 컴포넌트 함수는 렌더마다 다시 실행되고, 그 안에서 만든 클로저는 **그 렌더 시점의 변수 스냅샷**을 캡처한다. 객체 참조가 아니라 값 자체가 렌더 세대에 묶인다.
- **멘탈모델**: "self 참조를 통해 항상 최신을 읽는다"가 아니라 "**클로저는 자신이 태어난 렌더의 세계에 갇힌다**"로 바꾼다. 해법: 의존성 배열에 정직하게 넣기, 함수형 업데이트 `setX(prev => ...)`, 최신 값이 꼭 필요하면 `useRef`.

## 6. setState가 동기라고 가정한다

- **증상**: `setCount(count + 1)` 직후 `console.log(count)`를 찍으면 옛 값. 연속 두 번 호출했는데 1만 증가.
- **원인**: state 갱신은 **비동기 배치**다. React는 여러 갱신을 모아 한 번에 [[Re-render]]하고, 현재 실행 중인 함수 안의 `count` 변수는 이미 캡처된 옛 값이다(§5와 같은 뿌리).
- **멘탈모델**: `setX`는 "지금 바꿔라"가 아니라 "**다음 렌더를 이 값으로 예약**"이다. 직전 값 기반 갱신은 반드시 `setX(prev => prev + 1)`. 갱신 후의 값이 필요한 로직은 다음 렌더(본문/effect)에서 처리한다.

## 7. 리스트 key에 index를 쓴다

- **증상**: 항목 삭제/재정렬 후 엉뚱한 행에 입력값·체크 상태가 남아 있다.
- **원인**: `key`는 diff 때 "이 항목이 이전 트리의 누구인가"를 잇는 **identity**다. index를 쓰면 삭제 시 뒷 항목들이 앞 항목의 identity(와 state)를 물려받는다.
- **멘탈모델**: `key` = UITableView의 셀 재사용 식별자가 아니라 **데이터의 고유 id**. 이 레포는 `uid('r-…')`로 생성한 `record.id`를 쓴다(`src/store/StoreContext.tsx`). 안정적 id가 없으면 만들어서라도 부여한다. [[FlatList]]의 `keyExtractor`도 같은 원리.

## 8. 한쪽 플랫폼에서만 테스트한다

- **증상**: iOS 시뮬레이터에선 완벽한데 Android에서 그림자가 없고, 텍스트가 잘리고, 키보드가 입력창을 가린다. (이 레포도 웹+iOS 위주 검증이라 Android 잔여 확인 항목이 남아 있다 — `docs/PROGRESS.md`.)
- **원인**: RN은 "한 번 작성"이지 "한 번 테스트"가 아니다. 그림자(`shadowColor` vs `elevation`), 폰트 메트릭, 키보드 동작, 뒤로가기 버튼 등이 플랫폼별로 다르다.
- **멘탈모델**: 네이티브에서 기기 매트릭스를 돌리던 습관을 그대로 유지하되, 축이 하나 늘었다고 생각한다: **iOS × Android (× web)**. 최소한 머지 전에 양쪽에서 화면을 한 번씩 띄운다. 웹 프리뷰는 빠르지만 `Alert` 등 네이티브 UI가 생략됨을 기억.

## 9. node_modules를 직접 수정한다

- **증상**: 라이브러리 버그를 `node_modules/` 안 파일을 고쳐 해결 → 다음 `npm install`에서 조용히 증발 → "고쳤던 버그가 다시 생겼다".
- **원인**: `node_modules`는 빌드 산출물 캐시에 가깝다. lockfile 기준으로 언제든 통째로 재생성된다(CI에선 매번).
- **멘탈모델**: 벤더 라이브러리 패치의 정석은 **patch-package**(또는 pnpm의 `pnpm patch`) — diff를 `patches/`에 커밋하고 install 후 자동 재적용. Pods를 직접 고치지 않고 fork/패치를 쓰는 감각과 같다. 물론 최선은 업스트림 PR.

## 10. 네이티브 의존성 추가 후 pod install / [[Prebuild]]를 잊는다

- **증상**: 라이브러리 설치 후 `Cannot find native module ...`, `... is not a function`, 또는 빌드는 되는데 런타임 크래시. JS 코드는 분명 맞는데.
- **원인**: 네이티브 코드가 포함된 패키지는 JS 설치(`npm install`)만으로 끝나지 않는다. [[Autolinking]]이 네이티브 프로젝트에 반영되려면 `pod install`(iOS) 내지 [[Prebuild]] 후 **앱 재빌드**가 필요하다. [[Fast Refresh]]는 JS만 갱신한다.
- **멘탈모델**: 경계를 항상 의식한다 — "**JS 변경 = 리프레시로 끝 / 네이티브 변경 = 재빌드**". Expo에선 `npx expo install <pkg>` → `npx expo run:ios`(내부적으로 prebuild+pod install). [[Expo Go]]는 미리 포함된 네이티브 모듈만 갖고 있어 임의의 네이티브 의존성을 아예 못 싣는다 → 그래서 이 레포도 [[Dev Client]] 빌드를 쓴다(`docs/DEVICE_TESTING.md`).

---

> **새 함정을 발견하면 여기에 추가.** 형식은 위와 같이 "증상 → 원인 → 올바른 멘탈모델" 3~6줄.
